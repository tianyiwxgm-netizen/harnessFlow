## 2. L2 清单（7 个）

| L2 ID | 名称 | 一句话职责 | 聚合自 BF / scope 条目 | 核心问题 |
|---|---|---|---|---|
| **L2-01** | 11 主 Tab 主框架 + 路由守则 | 维护 11 主 tab（项目总览 / Gate / 产出物 / 进度 / WBS / 决策流 / 质量 / KB / Retro / 事件 / Admin）的路由骨架 + tab 间导航 + 单项目单 session 边界守则 + tab 级权限限制 | scope §3.3.1 现有 11 tab + 现有 UI mock | 11 视图怎么组织 |
| **L2-02** | Gate 决策卡片（推送 + Go/No-Go/Request-change） | 接 IC-16 Gate 卡片推送 → 渲染 Gate 展示面 → 收集用户 Go/No-Go/Request-change → 通过 IC-17 回 L1-01 → 期间硬阻断主 loop 推进 | BF-S1-04 / BF-S2-09 / BF-S3-05 / BF-S7-05 + scope §3.4 P1 Gate 待办中心 | 4 次 Stage Gate 怎么交互 |
| **L2-03** | 进度实时流（event stream 消费与渲染） | 订阅 L1-09 事件总线（IC-L2-02）→ 按 type 前缀过滤 + 时间轴渲染 + 渐进加载 + 跨 tab 共享数据 + 保证消费延迟 ≤ 2s → 驱动所有 tab 的实时视图 | BF-X-01 决策心跳 + scope §3.3.1 执行时间轴 + scope §3.4 P0 决策轨迹 / Loop 历史 | 怎么实时 |
| **L2-04** | 用户干预入口（panic/pause/resume/change_request） | 全局暴露干预动作（紧急停 / 暂停 / 恢复 / 变更请求 / 澄清答案 / 授权）→ 通过 IC-17 user_intervene 推 L1-01 → 全程审计留痕 + 本地状态同步 | BF-L3-09 请示用户流 + scope §5.10.6 panic 义务 + BF-E-12 硬红线授权 | 用户干预怎么落 |
| **L2-05** | KB 浏览器 + 候选晋升 UI | 浏览 3 层 KB 条目（Session / Project / Global）+ 按 kind / scope / applicable_context 过滤 + 查看 observed_count 累积 + 支持用户显式发起"晋升"动作（旁路阈值）+ 同步 KB 变更展示 | scope §3.3.1 项目资料库 tab + scope §3.3.2 知识库后台 + BF-S7-04 晋升仪式 | KB 怎么给人看 + 怎么晋升 |
| **L2-06** | 裁剪档配置 UI（完整 / 精简 / 自定义） | 在 S1/S2 前让用户显式选择合规裁剪档（PM-13 完整 / 精简 / 自定义）+ 自定义档时提供 checklist 勾选各产出物 + 校验不得违反 L1 硬约束 + 通过 IC-17 推 L1-02 | scope §5.2 L1-02 裁剪机制 + PM-13 合规可裁剪 | 怎么让用户选裁剪档 |
| **L2-07** | Admin 子管理模块（9 后台 + 红线告警角 + 审计查询） | 承载后台 9 模块（执行引擎 / 执行实例 / KB / Supervisor / Verifier / Subagents / Skills / 统计 / 诊断）+ 红线事件告警角（scope §3.4 P0）+ 审计追溯查询面板（scope §3.4 P1）+ 多模态内容展示（P1）| scope §3.3.2 后台 9 模块 + §3.4 新增缺口（红线告警 / 审计面板 / 多模态） | 管理视图 + 告警 + 审计 |

**为什么切成 7 个 L2（brainstorming 结论）**：

- **L2-01 是骨架底座**：所有 tab 的承载者，任何 tab 级交互都落在它的路由 + 权限规则上；独立出来避免与具体业务 tab 纠缠。
- **L2-02 单独存在原因**：Gate 是**强阻断式交互**（scope §5.10.6 义务 4 "在 Stage Gate 待审时阻断 L1-01 推进"），和普通 tab 展示的"只读消费"语义差别极大；切出后 Gate 阻断逻辑集中。
- **L2-03 是实时流底座**：所有 tab 的实时更新都依赖同一个事件订阅 + 渲染基础设施；切出后避免每个 tab 自己实现事件消费（重复代码 + 一致性漂移）。
- **L2-04 单独存在原因**：干预动作是 L1-10 的**唯一业务写能力**（其他全是只读），需要独立的审计 / 幂等 / panic 处理路径。
- **L2-05 与 L2-02 不合并**：晋升是"用户授权旁路阈值"路径（BF-E-11 / scope §5.6 PM-06），语义与 Gate 不同（Gate 是项目生命周期强节点 / 晋升是 KB 软节点），且晋升不阻断主 loop。
- **L2-06 单独存在原因**：裁剪档选择**只在 S1/S2 前发生一次**（scope §5.2.6 义务 3），频率低且流程特殊，切出后不混入 Gate 卡片的通用阻断逻辑。
- **L2-07 单独存在原因**：Admin 是"管理员级"功能集（配置 / 实例 / 诊断 / 告警 / 审计），访问者 / 频率 / 目的和"项目详情 11 tab"都不同；独立出来也便于未来权限隔离。

**关键规则**：

- **L2-03 是数据面唯一入口**：所有 tab 的实时数据都通过它消费事件总线；其他 L2 不直接订阅事件总线（除非特殊场景，经 §6 契约授权）。
- **L2-01 是渲染面骨架**：11 tab 路由 + 导航 + 页面生命周期；每 tab 内部业务交给对应 L2（如 Gate tab → L2-02 / KB tab → L2-05）。
- **L2-04 是用户意图唯一出口**：所有用户"写"动作（授权 / 决定 / 干预）经 L2-04 封装 + 审计 + 推 L1-01；其他 L2 不直接调 IC-17（除 L2-02 的 Gate 决定是专用通道也经 L2-04 的 wrapper）。
- **L2-06 是 one-shot 特殊路径**：只在 S1/S2 前出现一次，完成后即隐藏；不驻留 tab。
- **L2-07 与 11 tab 隔离**：Admin 是独立入口（不混在 11 tab 里），访问权限与项目详情 tab 不同。

## 5. L2 间业务流程（10 条）

### 流 A · 用户首次进入 UI（tab 初始化）

```
[用户浏览器访问 localhost:8765]
```

### 流 B · Stage Gate 推送与决策（最关键路径）

```
[L1-02 某阶段产出齐备]
     ↓ IC-16 push_stage_gate_card(gate_id, stage_from, stage_to,
                                   artifacts_bundle, required_decisions)
L2-02 Gate 决策卡片入口
     ↓
L2-02 写入本地 Gate 队列 + 触发 UI 提示：
   · Gate tab 未读徽标 +1
   · 顶部红色横幅 "S2 Gate 待审"
   · 声音提示（非首次）
     ↓
用户打开 Gate tab → 查看产出物 bundle 预览
   · 4 件套文档 / 9 计划 / TOGAF A-D / WBS / ADR 按类目列
   · 点任一产出物 → 内嵌查看器 / 外链跳转（只读）
     ↓
用户填写决定：
   · 选 Go / No-Go / Request-change
   · 附评论（必填，最少 10 字）

     ↓

     ↓

   · Go → 推进下阶段（event: stage_advanced）
   · No-Go → 标项目 CLOSED(dropped) 或回 S1（event: stage_rejected）
   · Request-change → 回当前阶段回修指定产出物（event: stage_change_requested）
     ↓
L2-03 收到 stage_* 事件 → 驱动 Gate tab 关闭该卡片 + 归档到 "Gate 历史"
     ↓
L2-01 更新顶部状态栏（当前 state / 心跳重启）
```

### 流 C · 事件实时流消费（L2-03 底座所有 tab 共用）

```
[L1-09 事件总线有新事件]
     ↓ IC-L2-02 异步 push
L2-03 事件接收器
     ↓

   · L1-03:wp_* → "WBS" store（驱动 tab ⑤）

   · L1-07:supervisor_* / hard_halt → "监督告警" store（驱动 Admin 红线告警角）
```

### 流 D · 用户干预（panic / pause / resume / change_request）

```
[用户点干预按钮，例如 "暂停"]
     ↓
L2-04 前置校验：
   · 当前是否允许该动作（panic 永远允许；pause 需主 loop 在运行态；resume 需 pause 过）
   · 幂等性检查（同一 action 10s 内重复 → 忽略，展示"已提交过"）
     ↓
L2-04 展示弹窗询问（非 panic）："确定 [动作] 吗？可附备注"
     ↓ 用户确认
L2-04 本地立刻进入"等响应"状态（按钮 loading / 禁其他干预）
     ↓
L2-04 IC-17 user_intervene(type=pause, payload={reason, ts})
     ↓
L1-01 ack 接收 → 事件 user_intervene_received → 总线
     ↓
L2-03 收到 ack 事件 → L2-04 退出"等响应" + 展示结果
     ↓
若 action=panic：
   · L2-04 UI 进入全锁模式（除 resume 按钮外全 disable）
   · 等 L1-09 flush 完成事件 → 展示"已安全停止"
若 action=pause / resume / change_request：
   · 正常更新状态栏
   · 审计留痕（L2-04 本地 log + 服务端事件）
```

## 8. L2-01 · 11 主 Tab 主框架 + 路由守则 详细定义

### 8.1 职责 + 锚定

**一句话职责**：承载项目详情的 **11 主 tab** 路由骨架（项目总览 / Gate / 产出物 / 进度 / WBS / 决策流 / 质量 / KB / Retro / 事件 / Admin 入口），保证 tab 间导航一致 + 页面生命周期管理 + 单项目单 session 边界守则 + 全局顶部状态栏（项目名 / 当前 state / 心跳 / panic 按钮）+ tab 级未读徽标 / 强提示条分发——是 L1-10 所有其他 L2 的**渲染容器底座**。

**上游锚定**：
- Goal §4 过程（人机互动）—— UI 是人类介入面
- scope §3.3.1 现有任务详情 11 tab 清单
- scope §5.10.3 In-scope "11 tab + admin + 新增 UI 缺口"
- scope §5.10.4 硬约束 5 "一个 UI session 只展一个项目"
- scope §5.10.5 禁止 5 "禁止跨项目数据混展"
- BF-X-01 主 Agent 决策心跳（心跳状态展示）

**下游服务**：
- L2-02 Gate 卡片（在 Gate tab 内渲染 / 触发未读徽标）
- L2-03 进度实时流（L2-01 注册 tab 所需 store 切片）
- L2-04 用户干预入口（顶部 panic / pause / resume 按钮栏）
- L2-05 KB 浏览器（在 KB tab 内渲染）
- L2-07 Admin 子管理（Admin 入口 + 红线告警角强提示条）

---

### 8.2 输入 / 输出

**输入（文字描述，非 schema）**：
- **用户路由操作**：进入某 tab / 切换 tab / 返回 / 刷新
- **tab 内部渲染需求**：各 L2（如 L2-02 / L2-05）向 L2-01 声明需要哪类 store 切片 + 订阅规则
- **顶部状态数据**：从 L2-03 订阅主 loop 心跳 / 当前 state / 降级状态
- **强提示事件**：从 L2-07（红线告警）/ L2-02（Gate 未决）来的"顶部横幅需求"
- **session 边界信号**：从 L1-09 接 session_resumed / session_closed 等事件，触发切项目 / 展示弹窗

**输出（文字描述）**：
- **渲染的 11 tab 骨架 + 顶部状态栏 + 全局强提示条**（给浏览器展示）
- **tab 切换事件**：记录用户在哪个 tab 停留多久（用于 retro）
- **未读徽标计数更新**：Gate / 告警类 tab 的 badge 数字
- **路由守则拒绝**：用户试图跨项目访问时拦截并提示
- 所有 tab 切换 / 进入 / 离开 → 事件（type=`L1-10:tab_navigated`）走 IC-09 落 L1-09

---

### 8.3 边界

**In-scope（本 L2 做什么）**：

1. 11 主 tab 路由 + 导航结构（即 scope §3.3.1 清单的 11 项）
2. 每 tab 的**页面生命周期管理**（加载 / 展示 / 缓存 / 销毁）
3. **顶部全局状态栏**（项目名 / 当前 state / 心跳倒计时 / 降级颜色 / panic 按钮）
4. **顶部强提示条**（硬红线告警 / Gate 未决 / session 恢复等）
5. **tab 级未读徽标**（Gate / 告警类）
6. **单项目单 session 守则**（一次只能展一个 project，跨项目操作被拦截）
7. 路由守则拒绝 + 用户友好提示
8. tab 间数据共享通过 L2-03 store（本 L2 只是容器，不持有业务数据）
9. 页面级埋点（tab 切换 / 停留时长 → 事件总线）

**Out-of-scope（本 L2 不做，谁做）**：

- ❌ 不做各 tab 内部业务渲染 → 各对应 L2（Gate → L2-02 / KB → L2-05 / Admin → L2-07 等）
- ❌ 不做事件消费 → L2-03
- ❌ 不做干预动作封装 → L2-04
- ❌ 不做裁剪档选择 → L2-06
- ❌ 不做 Admin 9 模块内部视图 → L2-07
- ❌ 不做登录 / 权限系统（V1 单用户本地）→ 未来
- ❌ 不做跨项目切换 UI（V1-V2 单项目）→ Goal §6.11

**边界规则**：

- 本 L2 是**渲染容器 + 路由守则**，不持有任何业务状态
- 任何"写"操作必须走 L2-04；本 L2 绝不自己发 IC-17
- tab 骨架的 11 项是**产品契约**，不得私自增减（scope §3.3.1）

---

### 8.4 约束

**业务模式引用**：

- **scope §5.10.3 In-scope 11 tab + admin**
- **scope §5.10.4 硬约束 5 单项目单 session**
- **scope §5.10.5 禁止 5 跨项目混展**

**硬约束清单**：

1. **11 主 tab 清单不得增减**（产品契约固定）
2. **单项目单 session**（同一浏览器 session 只渲染一个 project 的 11 tab）
3. **顶部状态栏永远可见**（除裁剪档模态 / panic 锁定状态外）
4. **panic 按钮永远可点击**（不受任何其他 UI 状态影响）
5. **心跳超过 30s 无 decision → 状态栏标"静默告警"**（scope §5.1.4 健康心跳）
6. **硬红线顶部横幅不可关闭**（直到 hard_halt_resolved 事件）
7. **Gate 未决期间"前进"类按钮必 disable**（scope §5.10.6 义务 4）

**性能约束（文字描述）**：

- tab 切换响应时间：P95 ≤ 100ms（已预加载数据）
- 首次加载 11 tab 骨架：P95 ≤ 500ms（不含业务数据）
- 顶部状态栏心跳更新：秒级（从事件到 UI 显示 ≤ 1s）
- 强提示条响应：事件到展示 ≤ 500ms

---

### 8.5 🚫 禁止行为（明确清单）

- 🚫 **禁止自己发 IC-17 用户意图**（全部通过 L2-04；本 L2 只渲染）
- 🚫 **禁止持有业务状态**（如 Gate 卡片内容 / KB 条目 / WP 数据；本 L2 只是容器）
- 🚫 **禁止跨项目切换 tab 数据**（一个 session 只展一个 project；V1-V2 硬性）
- 🚫 **禁止私自加 / 减 11 tab 之一**（产品契约固定；新需求走 scope §3.4 缺口）
- 🚫 **禁止 panic 按钮在任何 UI 状态下失效**（包括 Gate 阻塞 / 裁剪档模态 / 红线告警）
- 🚫 **禁止关闭硬红线顶部横幅**（唯一关闭路径是 hard_halt_resolved 事件）
- 🚫 **禁止在未订阅事件流的情况下展示"实时数据"**（本 L2 须声明订阅依赖给 L2-03）

---

### 8.6 ✅ 必须职责（明确清单）

- ✅ **必须**渲染完整 11 主 tab + 顶部状态栏 + 全局强提示条
- ✅ **必须**通过 L2-03 订阅所需事件流切片（不自订总线）
- ✅ **必须**维护 tab 级未读徽标（Gate / 告警 / Admin 等）并响应事件
- ✅ **必须**保证心跳 30s 超时后状态栏高亮告警（不沉默）
- ✅ **必须**在 Gate 未决时 disable "前进"按钮
- ✅ **必须**在跨项目访问尝试时明确拒绝并提示
- ✅ **必须**每次 tab 切换走 IC-09 append_event（审计用户浏览轨迹）
- ✅ **必须**在 session_resumed 事件到达时展示恢复弹窗并 disable 所有"前进"类按钮直到用户确认

### 8.9 🎯 交付验证大纲（Given-When-Then）

**正向场景 1 · 首次加载 11 tab 骨架**
- Given：浏览器访问 localhost:8765 + 有当前 project（状态 = S4）
- When：加载页面
- Then：
  - 骨架展示 ≤ 500ms 完成
  - 11 tab 全部可见（项目总览 / Gate / 产出物 / 进度 / WBS / 决策流 / 质量 / KB / Retro / 事件 / Admin 入口）
  - 顶部状态栏显示项目名 + 当前 state=S4 + 心跳倒计时
  - panic 按钮可见可点
  - 默认停留在"项目总览" tab

**正向场景 2 · tab 切换响应**
- Given：已加载，心跳正常
- When：用户点"WBS" tab
- Then：
  - 切换 ≤ 100ms（预加载数据）
  - URL 更新（便于分享锚点）
  - 写 `L1-10:tab_navigated` 事件（含 from / to / timestamp）

**正向场景 3 · 心跳静默告警**
- Given：L1-01 main loop 30s 无 decision 事件
- When：心跳倒计时到 0
- Then：
  - 顶部状态栏心跳区变黄色 + 文字"心跳静默"
  - 不阻塞用户操作 / 不自动干预
  - 有后续 decision 事件到达 → 恢复绿色

**负向场景 4 · 跨项目访问拦截**
- Given：当前 session 绑 project A
- When：用户手工改 URL 指向 project B 的 tab
- Then：
  - L2-01 拦截 + 展示友好提示 "一个 UI session 只能查看一个项目，请关闭重开"
  - 不展示 project B 任何数据
  - 记一条 `L1-10:cross_project_access_denied` 事件

**负向场景 5 · Gate 未决时"前进"按钮 disable**
- Given：Gate tab 有未决 Gate 卡片
- When：用户在其他 tab 点"推进到下阶段"类按钮
- Then：
  - 所有"前进"类按钮 disable + hover 提示"请先在 Gate tab 完成决定"
  - 不触发任何 IC-17

**集成场景 6 · 硬红线顶部横幅**
- Given：L1-07 产 hard_halt 事件
- When：L2-03 分发给 L2-07 红线告警角 + 顶部横幅
- Then：
  - 所有 tab 顶部红色横幅显示 "系统已硬暂停 · 等待授权"
  - 横幅不可关闭 / 不可忽略
  - 接 hard_halt_resolved 事件 → 自动移除横幅

**集成场景 7 · session 恢复弹窗阻塞推进**
- Given：Claude Code 重启，L1-09 发 system_resumed 事件
- When：L2-01 接到事件
- Then：
  - 展示恢复弹窗 "已恢复 project X, state=Y, 继续？"
  - 所有"前进"类按钮 disable
  - 用户选"继续" → L2-04 发 IC-17 resume
  - 弹窗关闭 + 按钮恢复

**性能场景 8 · 并发事件不卡顿**
- Given：L2-03 每秒推 100 条事件
- When：持续 10s
- Then：
  - tab 渲染主线程不卡死
  - 顶部状态栏更新仍秒级
  - tab 切换仍 ≤ 100ms

## 9. L2-02 · Gate 决策卡片（推送 + Go/No-Go/Request-change） 详细定义

### 9.1 职责 + 锚定

**一句话职责**：接 L1-02 经 IC-16 推送的 4 次 Stage Gate 卡片（S1-04 / S2-09 / S3-05 / S7-05）+ 通用请示卡片（BF-L3-09）→ 在 Gate tab 渲染决策面（产出物 bundle 预览 + 决策项输入）→ 期间**硬阻断主 loop 推进** → 收集用户 Go/No-Go/Request-change 决定（含评论）→ 经 L2-04 回 IC-17 → L1-01 → L1-02 或请求方——是 HarnessFlow 所有**用户决策性介入**的统一集散地。

**上游锚定**：
- Goal §2.7 人机协作 / Stage Gate
- Goal §3.3 methodology-paced autonomy "交付验收阶段强 Gate"
- Goal §3.5 硬约束 2 "必 4 次 Gate"
- scope §5.2 L1-02 "每次 Stage Gate 阻塞 L1-01 直到用户 Go/No-Go"
- scope §5.10.5 禁止 2 "禁止替用户做 Stage Gate 决定"
- scope §5.10.6 义务 4 "Stage Gate 待审时阻断 L1-01 推进"
- BF-S1-04 / BF-S2-09 / BF-S3-05 / BF-S7-05 · 4 次 Stage Gate
- BF-L3-09 请示用户流（Gate 属其中一种卡片类型）

**下游服务**：
- L2-04 用户干预入口（Gate 决定经其封装为 user_intervene）
- L2-01 11 tab 主框架（Gate tab 渲染容器 + 未读徽标触发）
- L2-03 进度实时流（读 Gate 相关历史事件构建 Gate 历史视图）

---

### 9.2 输入 / 输出

**输入**：
- **IC-16 Gate 卡片推送**（from L1-02）：含 gate_id / stage_from / stage_to / artifacts_bundle（产出物 ID 清单 + 路径 + 类型）/ required_decisions（必填决策项清单）
- **IC-16 变体 · 通用澄清卡片**（from L1-01）：含 question_id / question_text / option_type (text/radio/multi-select/free-text) / options / context_refs
- **用户交互输入**：选 Go/No-Go/Request-change / 评论（文字）/ Request-change 的勾选项与修改建议 / 澄清作答
- **事件流**：从 L2-03 订阅 `L1-02:stage_*` 事件（接 Gate 生命周期）

**输出**：
- **Gate tab 渲染视图**：活跃 Gate 卡片置顶 / Gate 历史归档区
- **用户决定封装**：推 L2-04（类型 gate_decision / clarify）
- **未读徽标与顶部横幅触发**：经 IC-L2-10 给 L2-01
- **Gate 决策落审计事件**：`L1-10:gate_decided`（含 gate_id / decision / comment / user_ts）经 IC-09 落 L1-09

---

### 9.3 边界

**In-scope**：

1. Gate 卡片接收 + 队列管理（同时至多 1 个活跃 Gate，多于 1 → 按 FIFO 排队）
2. 产出物 bundle 预览（md / json / 架构图 链接嵌入渲染）
3. 决策项收集（Go / No-Go / Request-change 单选 + 必填评论最少 10 字）
4. Request-change 时：产出物 checklist 勾选 + 修改建议 free-text
5. 通用澄清卡片渲染（text / radio / multi-select / free-text 4 种）
6. Gate tab 未读徽标与顶部横幅触发
7. Gate 历史归档（已决定的卡片存入历史区，可查可搜）
8. 主 loop 阻塞状态同步（通过接收 L1-02 的 `stage_waiting_gate` 事件来展示阻塞）

**Out-of-scope**：

- ❌ 不做产出物本身的编辑 → 只读预览（编辑走主 loop 的回修流程）
- ❌ 不做 Gate 阻塞的判定 → L1-02 L1-01 的逻辑（本 L2 只展示阻塞状态）
- ❌ 不做 IC-17 发送 → L2-04（本 L2 经 IC-L2-03 委托）
- ❌ 不做产出物自动审查（如代码 diff、产出物质量评分）→ 未来

## 10. L2-03 · 进度实时流（event stream 消费与渲染） 详细定义

### 10.1 职责 + 锚定

**一句话职责**：作为 L1-10 UI 消费 L1-09 事件总线的**唯一入口**——通过 IC-L2-01 向 L1-09 注册订阅（按 L1-XX: type 前缀过滤）→ 异步接收事件 → 去重 + 分类 + 路由到各 tab 的专属 store 切片 → 响应式驱动 L2-01 渲染层 → 实时维护跨 tab 共享的"事件时间轴"全局 store → 保证消费延迟 ≤ 2s（scope §5.10.4 硬约束 4）——是 L1-10 的**数据面唯一底座**。

**上游锚定**：
- scope §5.10.1 职责 "消费 L1-09 事件总线 + 推用户输入回 L1-01"
- scope §5.10.4 硬约束 4 "UI 消费事件总线延迟 ≤ 2 秒"
- scope §5.10.5 禁止 1 "禁止直接改 task-board / 事件总线"（本 L2 只读）
- scope §5.10.6 义务 1 "必须实时消费事件总线（≤ 2 秒延迟）"
- BF-X-01 主 Agent 决策心跳（驱动 tab ⑥ 决策流的实时流）
- BF-X-03 事件总线落盘流（本 L2 在落盘端的"下游消费者"）
- scope §3.4 P0 "决策轨迹走廊 / Loop 历史"（实时流是它们的数据源）

**下游服务**：
- L2-01 11 tab 主框架（订阅 + store 切片分发）
- L2-02 Gate 卡片（订阅 stage_* 事件驱动 Gate 历史）
- L2-05 KB 浏览器（订阅 kb_* 事件实时同步变更）
- L2-07 Admin 模块（订阅 supervisor_* / hard_halt 驱动告警角）

---

### 10.2 输入 / 输出

**输入**：
- **IC-L2-01 异步 push 事件**（from L1-09）：按订阅过滤规则推送的事件流
- **订阅注册请求**：来自 L2-01 / L2-02 / L2-05 / L2-07 的 tab 加载时触发（"我要这类 type 前缀的事件流"）
- **退订请求**：tab 卸载 / UI 关闭时
- **拉取历史批请求**：tab 首次加载需要"最近 N 条事件"初始化（pull + 订阅混合模式）

**输出**：
- **分发事件到各 L2 的 store 切片**（响应式更新；UI 框架自动渲染）
- **跨 tab 共享时间轴**（用户切 tab 不丢数据）
- **消费延迟指标**（event.ts → UI 渲染完成；用于 UI 降级判定）
- **断联降级信号**（心跳丢失 > 10s 通知 L2-01 顶部横幅降级）

---

### 10.3 边界

**In-scope**：

1. 通过 IC-L2-01 向 L1-09 注册订阅（按 type 前缀过滤）
2. 接收异步 push 事件 + 去重（按 event_id）
3. 按 type 前缀分类到内部 store 切片（如 `L1-01:decision` → 决策流切片）
4. 跨 tab 共享的"全局事件时间轴" store（驱动 tab ⑩ 事件 tab）
5. 每切片支持响应式订阅（各 L2 声明"我要这个切片"，事件到达自动通知）
6. 首次加载 pull（调 IC-L2-01 的 replay 变体或复用 L1-09 查询 API）
7. 心跳监控（> 10s 无预期事件 → 断联降级）
8. 重连与补齐（重连后对齐最后消费 event_id → 拉取增量）
9. 消费延迟计算（打点 event.ts 与 UI render 时间）
10. 与 L1-09 约定的事件 type 前缀映射表（附录 A）

**Out-of-scope**：

- ❌ 不做事件写入 → 本 L2 **只读** L1-09（任何写都经 L2-04 的 IC-17）
- ❌ 不做事件内容校验 → L1-09 自己管 schema
- ❌ 不做业务展示 → 各 L2 承担（Gate → L2-02 / KB → L2-05 等）
- ❌ 不做持久化 → UI 内存态；刷新即重拉（L1-09 是事实源）
- ❌ 不做跨项目事件混流 → 始终按当前 project 过滤（单 session 单项目约束）
- ❌ 不做反向推主 loop → 发事件给总线（那是 L2-04 职责）

**边界规则**：

- 本 L2 是**只读消费者**，绝不写事件总线
- 单 session 只订阅当前 project 的事件（跨项目混流违反 scope §5.10.4 硬约束 5）
- 事件 type 前缀映射表是 L1-10 与 L1-09 的约定契约（附录 A），不得私改

### 10.4 约束

**业务模式引用**：

- **PM-10 事件总线单一事实源**（本 L2 是消费端）
- **scope §5.10.4 硬约束 4 消费延迟 ≤ 2 秒**
- **scope §5.10.5 禁止 1 UI 只读**

**硬约束清单**：

1. **消费延迟 ≤ 2s**（P95；从事件落盘时间到 UI 渲染完成）
2. **单 session 单项目**（订阅过滤必带 project_id；不允许跨项目合流）

- 单事件 push → UI 渲染 P95 ≤ 1s（余量留给 L1-09 → 网络）

- When：L2-03 重连
- Then：
  - 拉取 last_seen_event_id 之后所有事件
  - 去重后补齐 20 条到 store 切片
  - 不重复、不丢失

**负向场景 5 · 重复 event_id 去重**
- Given：网络重试导致同一 event_id 被 push 2 次
- When：L2-03 收到两条
- Then：
  - 第一条正常处理
  - 第二条被去重丢弃
  - 不重复渲染

**负向场景 6 · 跨项目混流被拒**
- Given：当前 session 绑 project A
- When：某异常情况 L1-09 误推 project B 的事件
- Then：
  - L2-03 过滤时识别 project 不匹配 → 丢弃 + 告警
  - 不展示 project B 数据
  - 记 `L1-10:cross_project_event_rejected` 事件

**集成场景 7 · 高频吞吐不卡顿**
- Given：L1-09 每秒推 100 条事件
- When：持续 10s
- Then：
  - UI 渲染主线程不卡死
  - 各切片更新实时
  - 消费延迟 P99 ≤ 2s
  - 内存占用 ≤ 10MB（超出按规则丢弃最旧）

**性能场景 8 · 首次 pull 500 条**
- Given：UI 首次启动
- When：L2-03 调 IC-L2-01 pull 最近 500 条
- Then：
  - ≤ 2s 完成
  - 按 type 前缀分到各切片
  - 各 tab 可立即展示初始化数据

## 11. L2-04 · 用户干预入口（panic/pause/resume/change_request） 详细定义

### 11.1 职责 + 锚定

**一句话职责**：作为 L1-10 UI **用户意图的唯一写出口**——把全部 UI "写"动作（panic 紧急停 / pause 暂停 / resume 恢复 / change_request 变更 / clarify 澄清作答 / authorize 红线授权 / gate_decision Gate 决定 / kb_promote KB 晋升 / set_scale_profile 裁剪档选择 等）统一封装为 user_intervene 消息，经 scope §8 IC-17 推 L1-01 主 loop + 全程审计留痕——是 L1-10 对内**唯一被允许调 IC-17 的 L2**。

**上游锚定**：
- scope §5.10.2 "输出：用户输入回传（授权 / 澄清答案 / Go 决定 / 变更请求）推给 L1-01"
- scope §5.10.6 义务 2 "保留用户介入的完整审计日志"
- scope §5.10.6 义务 6 "panic 时立即 flush + 暂停整个系统"
- scope §8 IC-17 · `user_intervene({type, payload}) → {accepted}`
- BF-L3-09 请示用户流（作答通道）
- BF-E-12 硬红线上报流（授权通道）

**下游服务**：
- L1-01 主 loop（scope §8 IC-17 承接方）
- L2-02 Gate 卡片（经 IC-L2-03 委托）
- L2-05 KB 浏览器（经 IC-L2-05 委托）
- L2-06 裁剪档 UI（经 IC-L2-06 委托）
- L2-07 Admin 模块（经 IC-L2-07 委托）

---

### 11.2 输入 / 输出

**输入**：
- **panic 触发**：用户点全局永驻 panic 按钮（scope §5.10.6 义务 6）
- **pause / resume 触发**：用户点暂停 / 恢复按钮（需当前主 loop 状态匹配）
- **change_request 触发**：用户在任意 tab 发起变更请求（附修改意图）
- **clarify 作答**：来自 L2-02 的澄清卡片
- **authorize 红线授权**：来自 L2-07 红线告警角
- **gate_decision**：来自 L2-02 Gate 卡片
- **kb_promote**：来自 L2-05 KB 浏览器
- **set_scale_profile**：来自 L2-06 裁剪档 UI
- **其他 Admin 配置变更**：来自 L2-07

**输出**：
- **user_intervene 消息经 IC-17 推 L1-01**：含 type / payload / ts / user_id（单用户 V1 = default）
- **本地 UI 状态同步**：按钮 loading / 等响应 / 完成态
- **审计事件**（type=`L1-10:user_intervene_submitted`）经 IC-09 落 L1-09
- **panic 本地锁 UI 状态**（除 resume 外全 disable）
- **幂等拒绝提示**（10s 内同 action 重复 → 展示"已提交过"）

---

### 11.3 边界

**In-scope**：

1. 封装所有 UI 写动作为 user_intervene 格式
2. 前置校验（当前是否允许该动作 / 幂等性 / 参数完整性）
3. 非 panic 动作的二次确认弹窗
4. panic 动作**无需二次确认**（立即提交）
5. 经 IC-17 推 L1-01
6. 本地 UI 状态同步（loading / ack / error 展示）
7. 审计留痕（每次 user_intervene 双端落：UI 本地 log + L1-09 事件总线）
8. panic 时锁 UI（除 resume 按钮外全 disable 直到"已停"事件）
9. 幂等管理（同一 action 10s 内重复忽略）
10. 错误处理（IC-17 返回 rejected 时展示原因）

**Out-of-scope**：

- ❌ 不做业务逻辑 → 主 loop 做
- ❌ 不做决策正确性判断（用户说什么就推什么）→ 主 loop 做
- ❌ 不做授权范围审查（红线 whitelist 范围）→ L1-07 做
- ❌ 不发任何 read API → 本 L2 只写
- ❌ 不持久化用户凭证 → runtime 内存（scope §5.10.5 禁止 4）
- ❌ 不做多用户协同（V1 单用户）
- ❌ 不在 UI 侧猜用户意图（不做自动推断 / 不做预填默认值填写决定类字段）

**边界规则**：

- **本 L2 是 L1-10 唯一写出口**：所有其他 L2 经委托契约（IC-L2-03/05/06/07）调本 L2，不直接调 IC-17
- **panic 特权**：任何 UI 状态下 panic 按钮必可点（scope §5.10.6 义务 6）
- **幂等性**：同 action 10s 窗口内重复 → 忽略（防双击）

---

### 11.4 约束

**业务模式引用**：

- **scope §5.10.5 禁止 4 "禁止存储用户凭证"**
- **scope §5.10.6 义务 2 "保留完整审计日志"**
- **scope §5.10.6 义务 6 "panic 时立即 flush + 暂停整个系统"**
- **scope §8 IC-17 契约规范**

**硬约束清单**：

1. **panic 按钮全局永驻 + 永不 disable**（任何 UI 状态）
2. **每次 user_intervene 必写本地 log + 总线事件**（双端审计）
3. **凭证不落盘**（GitHub PAT 等仅 runtime 内存；scope §5.10.5 禁止 4）
4. **非 panic 动作必经二次确认**（防误操作；panic 例外）
5. **幂等窗口 10s**（同 action + 同 payload hash 10s 内重复 → 拒绝）
6. **IC-17 调用必携带完整 audit 上下文**（who / what / when / why）
7. **禁止绕过本 L2 直接发 IC-17**（其他 L2 必走委托契约）

**性能约束**：

- user_intervene 提交响应时间（到 IC-17 发出）：P95 ≤ 200ms
- IC-17 ack 延迟展示：P95 ≤ 1s
- panic 按钮点击到主 loop 开始停机：P95 ≤ 500ms
- 本地 log 写延迟：异步，不阻塞主路径

---

### 11.5 🚫 禁止行为（明确清单）

- 🚫 **禁止 panic 按钮在任何状态下 disable**（硬性永驻）
- 🚫 **禁止其他 L2 直接调 IC-17**（必经本 L2 的委托契约 IC-L2-03/05/06/07）
- 🚫 **禁止 user_intervene 不经审计落盘**（审计是用户介入合法性的证据）
- 🚫 **禁止存储用户凭证到本地**（scope §5.10.5 禁止 4）
- 🚫 **禁止在 UI 侧猜用户意图**（不做自动推断 / 不预填决定类字段）
- 🚫 **禁止双击误提交**（幂等窗口必执行）
- 🚫 **禁止 panic 需要二次确认**（panic 是紧急动作，确认会失救）
- 🚫 **禁止 IC-17 失败后用户无感知**（必须 UI 提示错误 + 事件留痕）

---

### 11.6 ✅ 必须职责（明确清单）

- ✅ **必须**封装所有 UI 写动作为 user_intervene 格式
- ✅ **必须**通过 IC-17 推 L1-01（唯一路径）
- ✅ **必须**每次介入双端审计（本地 log + L1-09 事件）
- ✅ **必须**panic 动作立即执行 + 锁 UI + flush 本地
- ✅ **必须**非 panic 动作二次确认
- ✅ **必须**幂等拒绝 10s 内重复（展示"已提交过"）
- ✅ **必须**IC-17 失败时 UI 展示错误原因 + 记失败事件
- ✅ **必须**对接所有其他 L2 的委托契约（IC-L2-03/05/06/07）

---

### 11.7 🔧 可选功能职责

- 🔧 **干预历史**（显示本 session 所有用户介入记录，便于用户回顾）
- 🔧 **撤销窗口**（非破坏性动作 3s 内可撤销，撤销走新 user_intervene）
- 🔧 **快捷键绑定**（如 Ctrl+P panic / Ctrl+Shift+P pause）
- 🔧 **介入模板**（常用变更请求文案的模板）
- 🔧 **离线队列**（网络断时本地排队，恢复后发送；panic 除外，panic 永远要求在线）

### 11.8 与其他 L2 / L1 交互（IC 契约清单）

| IC | 被调方 | 何时调 | 一句话 |
|---|---|---|---|

| **IC-09**（经 L2-03 间接 audit）| L1-09 | 每次介入 | 双端审计写 `L1-10:user_intervene_submitted` 事件 |

**作为被调方**：

### 11.9 🎯 交付验证大纲（Given-When-Then）

**正向场景 1 · panic 立即生效**
- Given：UI 正常运行 + 任何状态（Gate 活跃 / 裁剪档模态 / KB 浏览）
- When：用户点 panic 按钮
- Then：
  - 无二次确认
  - ≤ 200ms 发 IC-17(type=panic)
  - ≤ 500ms L1-01 开始停机
  - UI 立即锁（除 resume 外全 disable）
  - 本地 log + 事件总线双端审计
  - 等"已停"事件到达 → 解锁 UI + 显示 "已安全停止"

**正向场景 2 · pause 二次确认后执行**
- Given：主 loop 运行中
- When：用户点 pause
- Then：
  - 弹窗 "确定暂停吗？可附备注"
  - 用户确认 + 附备注 "检查进度"
  - 经 IC-17(type=pause) 推 L1-01
  - ack 返回 → UI 状态栏变 "已暂停"
  - 审计双端留痕

**正向场景 3 · 幂等拒绝重复提交**
- Given：用户刚提交 pause
- When：3s 内再次点 pause
- Then：
  - 本 L2 识别 action+payload 相同 + 窗口内
  - 不发 IC-17
  - UI 展示 "已提交过，请稍候"
  - 记 `L1-10:user_intervene_deduplicated` 事件

**正向场景 4 · Gate 决定委托封装**
- Given：L2-02 有活跃 Gate
- When：用户提交 Go + 评论
- Then：
  - L2-02 经 IC-L2-03 调 L2-04
  - L2-04 封装为 user_intervene(type=gate_decision, payload={gate_id, decision=Go, comment})
  - 经 IC-17 推 L1-01
  - 审计留痕

**负向场景 5 · IC-17 返回 rejected**
- Given：用户点 resume
- When：主 loop 当前不在 paused 态（state 不匹配）
- Then：
  - IC-17 ack(accepted=false, reason=invalid_state)
  - UI 展示红色错误 "当前不是暂停状态，无法恢复"
  - 记失败事件 `L1-10:user_intervene_rejected`

**负向场景 6 · 凭证不落盘**
- Given：红线授权需用户输入 GitHub PAT
- When：用户输入 + 提交
- Then：
  - PAT 仅 runtime 内存传输
  - 不写 localStorage / cookie / 文件
  - 使用后立即清零内存变量
  - 审计事件仅记 "pat_used=true"，不含 PAT 本身

**集成场景 7 · 全委托契约覆盖**
- Given：L2-02 / L2-05 / L2-06 / L2-07 各发起一个介入
- When：并发 4 个委托调用
- Then：
  - 4 个都经本 L2 封装
  - 4 次独立 IC-17 调用
  - 4 次独立审计事件
  - 互不影响（FIFO 或并发按 L1-01 决定）

**性能场景 8 · panic 到停机速度**
- Given：繁忙 tick 中
- When：panic 点击
- Then：
  - P95 ≤ 500ms 从点击到 L1-01 收 IC-17
  - L1-01 内部处理 panic 的延迟不属本 L2 责任
  - 本 L2 在 UI 侧 ≤ 200ms 完成封装 + 发送
