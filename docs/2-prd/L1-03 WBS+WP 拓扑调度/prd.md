**本 PRD 的职责**：把 L1-03 内部拆成 **5 个 L2** + 画清楚它们之间的 **架构 / 业务流 / 契约 / 验收大纲**。

**锚点速查**：
- **Goal §2.3** · WP 拓扑 "超大型项目不是一次跑完的。主 Agent 按业务模块 / 架构边界把项目拆成若干 Work Package（WP）…… 主 Agent loop 每次只调度 1-2 个 WP 并行推进，避免认知爆炸…… WP 级失败不污染整个项目。"
- **Goal §7.1** · 风险应对 "用 WP 拓扑拆解 wrap 长程不确定性"
- **BF-S2-07** · WBS 拆解流（S2 规划+架构阶段内子流，4 件套 + 9 计划 + 架构 → `docs/planning/wbs.md`）
- **BF-S4-01** · WP 取任务流（拓扑调度，上一 WP 完成 / 进 loop → 锁定下一 WP）
- **BF-S4-02** · WP 内 mini-PMP 流（WP 锁定 → WP 推进记录，本 L1 只负责触发，不负责内部推进）
- **BF-S4-05** · WP commit 流（自检 PASS → git commit WP 粒度，本 L1 接收完成事件）
- **BF-E-08** · WP 失败回退流（WP ≥ 3 次 WP-DoD 自检失败 → 生成回退建议）
- **PM-04** · WP 拓扑并行推进（同时 ≤ 1-2 个 WP、WP 粒度 ≤ 5 天）

## 2. L2 清单（5 个）

| L2 ID | 名称 | 一句话职责 | 聚合自 BF | 核心问题 |
|---|---|---|---|---|
| **L2-01** | WBS 拆解器 | 在 S2 阶段消费 4 件套 + TOGAF A-D 架构，按"业务模块 / 架构边界"做 WBS 层级拆解（项目 → 模块 → WP），为每个 WP 装配 4 要素（独立 Goal / DoD 表达式入口 / 依赖清单 / 工时估算）+ 推荐 skill，产出 `docs/planning/wbs.md` | BF-S2-07 | 怎么拆 |
| **L2-02** | 拓扑图管理器（DAG + 关键路径） | 维护 WP 之间的依赖拓扑结构（DAG 强校验无环 + 关键路径识别 + 并行度上限 ≤ 2）、节点状态快照（ready / running / done / failed / blocked）、全局拓扑一致性守护（任何新增 / 修改 / 完成事件先经本层校验），作为调度器与追踪器的"真值源" | BF-S2-07 产出后、BF-S4 全程 | 图怎么管 |
| **L2-03** | WP 调度器 | 响应 L1-01 主 loop 的"取下一可执行 WP"请求（IC-02），按拓扑依赖 satisfied + 关键路径优先 + 并行度 ≤ 2 的规则从 L2-02 挑选 WP 并锁定运行位，传递 WP 定义给 L1-04 Quality Loop；对"无可取 WP"场景做明确语义区分（全部完成 / 全部阻塞 / 待依赖） | BF-S4-01 | 怎么取下一个 |
| **L2-04** | WP 完成度追踪器 | 订阅 L1-04 的 WP 完成 / 失败事件（含 WP-DoD 自检通过、WP commit、verifier verdict），在拓扑图上打点并维护"项目级完成率 + 剩余工时 + 已完成 WP 清单 + 当前运行 WP 清单"四项指标，向 L1-07 监督（"进度节奏"维度）和 L1-10 UI（🔧 WBS tab）暴露只读数据 | BF-S4-02 + BF-S4-05 | 怎么跟进度 |
| **L2-05** | 失败回退协调器 | 追踪每个 WP 的连续失败次数（按同一 WP id 计数），到达 ≥ 3 次阈值时触发 BF-E-08 → 生成结构化回退建议（拆 WP / 改 WBS / 改 AC）并推给 L1-01 / L1-07，不执行回退动作（由 L1-07 决定走哪条回退路径），本 L2 只负责"识别 + 建议 + 死锁释放"三件事 | BF-E-08 | 失败怎么办 |

**为什么切成 5 个 L2（brainstorming 结论）**：
- **L2-01 拆解器 vs L2-02 拓扑图管理器 必须分开**：前者是"一次性产出动作"（在 S2 规划阶段跑一次 + 变更请求时再跑），后者是"常驻运行期服务"（整个 S4 阶段被调度器 / 追踪器反复访问），两者生命周期、触发节奏、状态性完全不同。
- **L2-03 调度器 vs L2-04 追踪器 必须分开**：前者响应主 loop 的"pull 请求"（主动出去挑 WP 锁定），后者订阅"push 事件"（被动接收完成 / 失败通知），职责方向完全相反；且调度涉及"选下一个"的决策，追踪只做"记账"。
- **L2-05 失败回退协调器 独立**：失败判定有独立的计数器语义（按 WP id 计、连续性判定、超限触发）和独立的下游（L1-07），不适合塞进 L2-04（会和"进度记账"混淆）或 L2-03（调度器不应含"善后"逻辑）。
- **L2-02 作为真值源必须被 L2-03/04/05 共同依赖**：任何节点状态变更必须经 L2-02 一致性校验，否则 DAG 可能被打穿（环 / 并行度越界 / 状态非法跃迁）。

**关键规则**：
- **L2-02 是 L1-03 的真值源**：其他 4 个 L2 的任何读 / 写 / 判定都必须经本层一致性守护；禁止绕过直接改拓扑状态。
- **L2-01 与其他 L2 时序分离**：L2-01 只在 S2 阶段（或变更请求触发）跑一次一次性动作；L2-02/03/04/05 在 S4 阶段常驻。
- **L2-03 和 L2-04 是方向相反的一对**：调度是 "pull 出去挑 WP"，追踪是 "push 事件进来打点"，两者通过 L2-02 同步节点状态。
- **L2-05 不做决策，只做识别 + 建议**：失败回退的"走哪条路"决策是 L1-07 监督的职责（4 级回退路由），本 L1 只提供建议。

## 5. L2 间业务流程（6 条）

### 流 A · S2 阶段 WBS 一次性拆解（BF-S2-07）

```
[L1-02 S2 阶段 4 件套 + 架构就绪]
     ↓ IC-19 request_wbs_decomposition
L2-01 入口
     ↓ 读 4 件套 + TOGAF A-D 架构
L2-01 分层拆解：项目级 → 模块级 → WP 级
     ↓ 为每个 WP 装配 4 要素：
       · 独立 Goal（可追溯回 4 件套某条）
       · DoD 表达式入口（待 L1-04 S3 填充）
       · 依赖清单（指向其他 WP id）
       · 工时估算（≤ 5 天硬约束）
     ↓
L2-01 自检：
  · 每 WP 粒度 ≤ 5 天？超 → 再拆一层
  · 每 WP 4 要素齐全？缺 → 修补
     ↓ 满足 → 输出 WBS 拓扑数据
L2-01 写 docs/planning/wbs.md（层级树 + 依赖图 + 工时）
     ↓ 回传拓扑数据
L2-02 装图（DAG 校验 + 关键路径识别 + 并行度上限标记）
     ↓ 任一硬约束违反（环 / 并行 > 2 / 粒度 > 5 天）→ 整体拒绝 + 返回错误
     ↓ 通过 → 状态固化
IC-19 返回 {wbs_topology} 给 L1-02
     ↓ S2 Stage Gate 等用户 Go/No-Go
全流程事件 → IC-09 append_event → L1-09
```

### 流 B · S4 阶段 WP 取任务（BF-S4-01，最高频路径）

```
[L1-01 主 loop 决策进 Quality Loop 前]
     ↓ IC-02 get_next_wp
L2-03 入口
     ↓ 读 L2-02 当前节点状态快照
L2-03 筛选候选：
  · 状态 == ready
  · 所有依赖 WP 状态 == done
  · 当前并行运行数 < 2
     ↓ 候选集为空？
       ├ 所有 done → 返回 {wp_id: null, reason: "all_done"}
       ├ 全阻塞/失败 → 返回 {wp_id: null, reason: "deadlock"} + 通知 L2-05
       └ 等依赖 → 返回 {wp_id: null, reason: "awaiting_deps"}
     ↓ 候选集非空 → 按关键路径优先排序（L2-02 提供关键路径标记）
L2-03 选定 top1 → 向 L2-02 请求锁定（state: ready → running）
     ↓ 锁定成功（L2-02 做一致性校验：再次确认依赖 satisfied + 并行度位空）
IC-02 返回 {wp_id, wp_def, deps_met: true} 给 L1-01
     ↓ L1-01 继续 IC-03 enter_quality_loop(wp_def)
全流程事件 → IC-09 append_event → L1-09
```

### 流 C · WP 完成事件回流（BF-S4-02 + BF-S4-05）

```
[L1-04 Quality Loop S5 裁定 PASS → 触发 BF-S4-05 WP commit]
     ↓ L1-04 广播 wp_done 事件（携带 wp_id + verifier_report 摘要 + commit_sha）
L2-04 订阅
     ↓ 向 L2-02 申请状态跃迁：running → done
L2-02 一致性校验：
  · wp_id 存在且当前状态 == running？否 → 拒绝 + 告警
  · 跃迁合法？合法 → 写入
     ↓
L2-04 更新项目级指标：
  · 完成率 += 本 WP 工时占比
  · 剩余工时 -= 本 WP 工时
  · 已完成 WP 清单 .append(wp_id)
  · 当前运行清单 .remove(wp_id)
     ↓ 释放并行度位
L2-04 推送 UI 刷新事件 → L1-10
L2-04 推送进度节奏数据 → L1-07
     ↓ 事件 → IC-09 append_event
     ↓
[返回 L1-01，L1-01 再次 IC-02 取下一个 WP]
```

### 流 D · WP 失败事件回流（BF-S4-04 自检 FAIL 或 BF-S5 verifier FAIL）

```
[L1-04 Quality Loop 某 WP 失败（WP-DoD 自检失败 或 verifier 判 FAIL）]
     ↓ L1-04 广播 wp_failed 事件（携带 wp_id + fail_level + reason）
L2-04 订阅
     ↓ 向 L2-02 申请状态跃迁：running → failed
L2-02 一致性校验 → 写入
     ↓
L2-04 更新指标：运行清单移除 + 失败清单追加 + 释放并行度位
L2-04 同步告知 L2-05：wp_id 失败计数 +1
     ↓
L2-05 检查：该 wp_id 累计失败次数 ≥ 3？
  ├ 否 → 仅记账，不触发回退；L2-02 把状态从 failed 放回 ready（允许 L1-04 重试）
  └ 是 → 触发 BF-E-08（见流 E）
     ↓ 事件 → IC-09 append_event
```

### 流 E · 失败回退建议（BF-E-08）

```
[L2-05 检测到某 wp_id 连续失败 ≥ 3 次]
     ↓
L2-05 读 L2-02 当前拓扑 + 读该 WP 历次失败原因摘要（来自事件总线）
     ↓
L2-05 生成结构化回退建议（3 选项之一或组合）：
  · 建议 A · 拆 WP：把当前 WP 再拆成更小 WP（返回 L2-01 做差量拆解）
  · 建议 B · 改 WBS 拓扑：调整依赖关系（可能前置某依赖）
  · 建议 C · 改 AC：AC 不合理 / 太严 → 回 S2 修 4 件套
     ↓
L2-05 推 L1-01：回退建议卡 {wp_id, suggestions, evidence_refs}
L2-05 把该 WP 标记 stuck（从调度候选池移除，L2-03 不再取它）
     ↓
L1-01 转发 L1-07 监督 → L1-07 基于 4 级回退路由规则决策最终走哪条
L1-07 回传决策（IC-14 push_rollback_route）→ L1-01 执行
     ↓ 事件 → IC-09 append_event
```

### 流 F · 跨 session 恢复（BF-E-02 → 重建拓扑）

```
[Claude Code 重启 + 存在未 CLOSED 项目]
     ↓ L1-09 触发 system_resumed 事件
L2-02 订阅 → 启动拓扑重建：
  · 优先从 checkpoint 读（若存在）
  · fallback 从事件总线回放（从项目 S2 Gate 通过点开始）
     ↓ 重建所有节点 + 状态
L2-04 订阅 system_resumed → 从事件总线补齐完成率 / 已完成清单
L2-05 订阅 → 从事件总线补齐失败计数
     ↓ 全部就绪 → 广播 "WBS 就绪" 事件
L1-01 下一 IC-02 请求即可正常调度
     ↓ 事件 → IC-09 append_event
```

## 8. L2-01 · WBS 拆解器 详细定义

### 8.1 职责 + 锚定

**一句话职责**：HarnessFlow 超大项目的"规划期一次性动作" —— 在 S2 阶段消费 L1-02 产出的 4 件套（需求 / 目标 / AC / 质量标准）+ TOGAF A-D 架构产出（含 ADR），按"业务模块 / 架构边界"做 WBS 层级拆解（项目级 → 模块级 → WP 级），为每个 WP 装配 4 要素（独立 Goal + DoD 表达式入口 + 依赖清单 + 工时估算）与推荐 skill，最终产出 `docs/planning/wbs.md`，并把拓扑数据交给 L2-02 拓扑图管理器装图入库。

**上游锚定**：
- Goal §2.3 WP 拓扑 "超大型项目不是一次跑完的。主 Agent 按业务模块 / 架构边界把项目拆成若干 Work Package（WP）"
- Goal §7.1 风险应对 "用 WP 拓扑拆解 wrap 长程不确定性"
- scope §5.3.1 职责（"把超大项目拆成若干 WP"）
- scope §5.3.3 In-scope 第 1/2 条（WBS 层级拆解 / WP 定义结构）
- scope §5.3.4 硬约束 "WP 粒度 ≤ 5 天工时 / WP 依赖必 DAG / WP 4 要素齐全"
- BF-S2-07 WBS 拆解流
- PM-04 WP 拓扑并行推进

**下游服务**：
- L2-02（装图前接收 WBS 拓扑数据 + 4 要素）
- L1-02（接收 wbs.md 作为 S2 Gate 硬性产出）
- L1-07 / L1-10（通过 L2-02 间接读到拓扑可视化）
- L2-05（变更请求 / 失败回退"拆 WP"时反向调用）

- WBS 拓扑数据结构化回传（交给 L2-02 装图）
- 拆解失败时返回结构化错误（含违反的硬约束 id + 违规 WP id）

### 8.3 边界

**In-scope**：
1. 按 4 件套 + TOGAF 架构边界做 WBS 层级拆解（项目 → 模块 → WP）
2. 每 WP 的 4 要素装配（Goal / DoD 表达式入口 / 依赖 / 工时）
3. WP 粒度自检（≤ 5 天硬约束，超限自动再拆一层）
4. 推荐 skill 建议（基于 WP 特征给出首选 skill 名称，供 L1-05 调度参考）
5. 差量拆解（变更请求 / 失败回退触发时保留已 done 部分）
6. `docs/planning/wbs.md` 的格式化产出

**Out-of-scope**：
- 不做 DAG 无环校验 → L2-02（L2-01 只组织层级，环校验归拓扑图管理器）
- 不做关键路径计算 → L2-02
- 不做运行期状态跃迁 → L2-02
- 不做 WP 调度 → L2-03
- 不做进度打点 → L2-04
- 不做失败回退决策 → L2-05 / L1-07
- 不做 DoD 表达式内容填充 → L1-04 S3（本 L2 只留入口占位）
- 不做 skill 实际调用 → L1-05（只给推荐名）
- 不做事件总线落盘 → L1-09（经 IC-L2-08）

**边界规则**：本 L2 是"规划期一次性动作"—— 只在 S2 阶段 + 变更请求 + 差量拆解三种时机运行，其余时间不应活跃；运行期所有状态性工作都归 L2-02/03/04/05。

### 8.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-04** WP 拓扑并行推进（本 L2 是拆解侧实现）
- **PM-07** 产出物模板驱动（wbs.md 按标准模板填充）
- **PM-08** 可审计全链追溯（每 WP 可追溯回 4 件套某条 + 架构某节点）
- **PM-10** 事件总线单一事实源（拆解过程全落 L1-09）

**硬约束清单（文字描述）**：
1. **WP 粒度 ≤ 5 天工时**：超过 5 天必须再拆；拆到"再拆没意义"的粒度下限由领域判断（不小于 0.5 天作为保底）。
2. **WP 4 要素齐全**：每个 WP 必须有独立 Goal + DoD 表达式入口 + 依赖清单 + 工时估算，缺一不可；缺则拒绝出站。
3. **WP Goal 必须可追溯回 4 件套**：每 WP Goal 引用 4 件套中某条需求 / 目标 / AC / 质量条款，失去追溯 = 无效 WP。
4. **依赖只能指向同 WBS 已存在 WP**：禁止悬空依赖（指向未存在 id）；禁止跨项目依赖。
5. **wbs.md 必须按 PM-07 模板落盘**：缺项 / 非模板 → S2 Gate 拒通过。
6. **差量拆解必须保留已 done 节点**：变更 / 回退触发时不得丢失历史完成 WP 的身份。
7. **推荐 skill 是建议不是硬绑定**：L1-05 仍保留"能力抽象层 fallback"能力，不因本 L2 建议被锁死。

**性能约束（文字描述）**：
- 拆解动作在 S2 阶段属低频但耗时可接受的"规划动作"，以可预期秒级～分钟级的用户等待为上限，超出时应分阶段预览供用户介入。
- 差量拆解应显著快于全量拆解（仅处理未 done 部分）。
- 产出 wbs.md 的可读性优先于形式严格性，确保用户评审阶段可肉眼 diff。

---

### 8.5 🚫 禁止行为（明确清单）

- 🚫 **禁止产出粒度 > 5 天的 WP**（硬约束 1 违反即拒绝出站）
- 🚫 **禁止产出缺 4 要素的 WP**（硬约束 2）
- 🚫 **禁止产出无追溯回 4 件套的 WP Goal**（硬约束 3）

## 9. L2-02 · 拓扑图管理器（DAG + 关键路径） 详细定义

### 9.1 职责 + 锚定

**一句话职责**：L1-03 的"运行期真值源" —— 维护 WP 之间的依赖拓扑结构（DAG 强校验无环）+ 关键路径识别 + 并行度上限（≤ 2）守护 + 节点状态机（ready / running / done / failed / blocked）+ 全局一致性守护（任何新增 / 修改 / 完成事件先经本层校验），作为 L2-03 调度器、L2-04 追踪器、L2-05 回退协调器的共同真值查询 / 写入对象。

**上游锚定**：
- scope §5.3.1 职责（"维护 WP 拓扑图（依赖关系 + 关键路径 + 并行度）"）
- scope §5.3.3 In-scope 第 3/5 条（DAG 校验 / 并行度控制）
- scope §5.3.4 硬约束（DAG 无环 / 并行 ≤ 2 / 未 satisfied 依赖不得取 WP）
- scope §5.3.5 禁止 "WBS 拓扑存在环 / 并行 > 2 个 WP / 取依赖未 satisfied 的 WP"
- PM-04 WP 拓扑并行推进
- PM-10 事件总线单一事实源（所有状态变更走事件总线）

**下游服务**：
- L2-03（读节点状态快照 + 申请状态锁定）
- L2-04（申请状态跃迁 running → done / failed）
- L2-05（读失败节点 + 标 stuck）
- L1-07（被观察："进度节奏"维度读拓扑指标）
- L1-10（UI 🔧 WBS tab 可视化只读）

---

### 9.2 输入 / 输出（文字级）

**输入**：
- 来自 L2-01 的"WBS 拓扑数据装图请求"（全量 / 差量两种）
- 来自 L2-03 的"节点状态快照读请求"
- 来自 L2-03 的"状态锁定请求"（ready → running）
- 来自 L2-04 的"状态跃迁申请"（running → done 或 running → failed）
- 来自 L2-05 的"stuck 标记申请"
- 来自 L1-09 的 system_resumed 事件（触发拓扑重建）

**输出**：
- 节点状态快照（供 L2-03 候选筛选）
- 一致性校验结果（通过 / 拒绝 + 拒绝原因）
- 关键路径清单（供 L2-03 优先级排序）
- 拓扑可视化只读数据（供 L1-10）
- 拓扑指标（完成率 / 剩余工时 / 当前运行数 / 已 done 数）基础数据（供 L2-04 做二次聚合）
- 全量状态变更事件（经 IC-L2-08 → IC-09）

---

### 9.3 边界

**In-scope**：
1. DAG 结构装入 + 强校验（无环）
2. 关键路径识别（标注关键路径节点 ID）
3. 并行度上限守护（当前 running 数 ≤ 2）
4. 节点状态机维护（ready / running / done / failed / blocked / stuck 六种状态与合法跃迁规则）
5. 一致性守护层（任何状态写入必经本层校验）
6. 跨 session 拓扑重建（从 checkpoint / 事件总线回放）
7. 差量合并（L2-01 差量拆解产物的合并入库）
8. 拓扑可视化只读视图（供 L1-10）

**Out-of-scope**：
- 不做 WBS 层级拆解 → L2-01
- 不做 WP 调度决策 → L2-03
- 不做完成率 / 剩余工时聚合 → L2-04（本 L2 只提供基础数据）
- 不做失败计数 → L2-05
- 不做 DoD 判定 → L1-04
- 不做 WP 内部执行 → L1-04
- 不做事件总线落盘 → L1-09
- 不做 UI 渲染 → L1-10（只暴露只读数据）

**边界规则**：本 L2 是"守门人 + 真值源"—— 只管拓扑结构与状态一致性，不做业务决策；任何绕过本层的状态修改都是违规。

---

### 9.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-04** WP 拓扑并行推进（本 L2 是核心实施层）
- **PM-10** 事件总线单一事实源（所有状态变更走 IC-09，重启可回放重建）
- **PM-08** 可审计全链追溯（每次状态跃迁有 before / after + 触发来源）

**硬约束清单（文字描述）**：
1. **DAG 无环强校验**：装图时必须校验无环；任何差量合并后二次校验；违反即整体拒绝。
2. **并行度上限 ≤ 2**：同时处于 running 状态的 WP 数不得超过 2；状态锁定请求若越界即拒绝。
3. **状态跃迁合法性封闭**：六状态间的合法跃迁路径封闭（例：ready → running、running → done、running → failed、failed → ready 重试、stuck 不可主动出、blocked 随依赖解除回 ready），任何非法跃迁请求拒绝。
4. **一致性守护不可绕过**：任何状态写入路径必经本层；禁止外部直接改拓扑数据。
5. **跨 session 可重建**：拓扑数据完全可从事件总线回放重建；允许 checkpoint 加速，但 checkpoint 不是唯一真值源。
6. **依赖只读入，不允许运行时修改依赖关系**：一旦装图完成，运行期禁止动态改依赖（变更必须走 L2-01 差量重拆路径）。
7. **关键路径标记随拓扑变化自动重算**：差量合并后必须刷新关键路径。

**性能约束（文字描述）**：
- 节点状态快照读：需成为 L2-03 调度的热路径查询，响应应快于 L2-03 调度整体延迟预算；
- 状态写入：亚秒级响应，不应成为 S4 阶段瓶颈；
- DAG 校验 + 关键路径计算：装图时一次性耗时可接受秒级；差量合并应显著快于全量；
- 并行度上限检查：O(1) 量级的当前 running 计数查询。

- 差量合并：显著快于全量装图；
- 并行度位检查：O(1)。

## 10. L2-03 · WP 调度器 详细定义

### 10.1 职责 + 锚定

**一句话职责**：响应 L1-01 主 loop 的"取下一可执行 WP"请求（对外 IC-02 get_next_wp） —— 从 L2-02 拓扑图管理器读当前节点状态快照，按"依赖 satisfied + 状态 ready + 并行度未越界 + 关键路径优先"四条规则筛选候选并选定 top1，申请状态锁定（ready → running），返回 WP 定义给 L1-01；对"无可取 WP"的语义（全部完成 / 死锁 / 等待依赖）明确区分，避免 L1-01 误判。

**上游锚定**：
- scope §5.3.1 职责（"在 S4 执行阶段调度'取下一可执行 WP'"）
- scope §5.3.3 In-scope 第 4/5 条（WP 调度算法 / 并行度控制）
- scope §5.3.4 硬约束（未 satisfied 依赖不得取 / 并行 ≤ 2）
- scope §5.3.5 禁止 "取依赖未 satisfied 的 WP / 并行 > 2"
- scope §8.2 IC-02 get_next_wp 契约
- BF-S4-01 WP 取任务流（拓扑调度）
- PM-04 WP 拓扑并行推进

**下游服务**：
- L1-01 主 loop（IC-02 调用方）
- L1-04 Quality Loop（接收本 L2 锁定的 WP 进 Quality Loop）
- L2-05（当调度返回 deadlock 语义时同步告知）

---

### 10.2 输入 / 输出（文字级）

**输入**：
- 来自 L1-01 主 loop 的 IC-02 get_next_wp 调用（常规情况下不携带参数；可选调用上下文如"偏好关键路径"hint 但不硬性）
- 来自 L2-02 的节点状态快照（本 L2 主动读取）
- 来自 L2-02 的关键路径清单

**输出**：
- `{wp_id, wp_def, deps_met: true}` 成功返回（选定并锁定 WP）
- `{wp_id: null, reason: "all_done" | "deadlock" | "awaiting_deps"}` 无可取 WP 的三态语义返回
- 每次调度决策的审计事件（经 IC-L2-08）
- 当 reason = deadlock 时向 L2-05 同步"死锁"信号

---

### 10.3 边界

**In-scope**：
1. 响应 IC-02 get_next_wp 调用
2. 候选筛选（依赖 satisfied + 状态 ready）
3. 并行度位检查（当前 running < 2）
4. 关键路径优先排序
5. 选定 top1 并申请状态锁定（IC-L2-03）
6. "无可取 WP" 的三态语义区分（all_done / deadlock / awaiting_deps）
7. 调度决策审计落盘

**Out-of-scope**：
- 不做 WBS 拆解 → L2-01
- 不维护拓扑结构 / 状态 → L2-02（本 L2 只读 + 申请写）
- 不做完成率聚合 → L2-04
- 不做失败计数 → L2-05
- 不做 WP 内部执行 → L1-04
- 不做 skill 选型 → L1-05
- 不做事件总线落盘 → L1-09

**边界规则**：本 L2 是"响应式 pull"—— 只在 L1-01 主动调用时出手；禁止后台主动调度 / 推送 WP。

---

### 10.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-04** WP 拓扑并行推进（本 L2 是调度侧实施）
- **PM-02** 主-副 Agent 协作（本 L2 只响应主 loop，不主动干预）
- **PM-10** 事件总线（调度决策走 IC-09 审计）

**硬约束清单（文字描述）**：
1. **依赖 satisfied 是选定的前提**：任何未满足全部前置依赖的 WP 不得被取；由 L2-02 二次校验兜底。
2. **并行度 ≤ 2 硬上限**：当前 running 数已 ≥ 2 时必须返回 null + awaiting_deps / all_done 语义（按实际情况），不得违规锁定。
3. **关键路径优先**：候选集非空且包含关键路径节点时，关键路径节点必须优先于非关键路径节点。
4. **三态语义不可合并**：all_done（所有 WP done）/ deadlock（全部 blocked or failed 无法推进）/ awaiting_deps（仍有 ready 但依赖未满足）必须分别清晰返回，不得混淆。
5. **选定即锁定**：返回 wp_def 前必须先在 L2-02 完成状态锁定；否则不得返回（防止并发重复取走同一 WP）。
6. **无状态调度**：本 L2 自身无持久化状态；任何"上次选了谁"等信息都从 L2-02 真值源重新读取。

**性能约束（文字描述）**：
- IC-02 响应时长：需快于 L1-01 主 loop 的决策 tick 间隔，亚秒级上限；
- 候选筛选 + 关键路径排序复杂度应线性于当前可取候选规模（非整图规模）；
- 并发安全：多次并发 IC-02 调用（边界情况）不得重复锁定同一 WP（由 L2-02 的锁定原语保证）。

---

### 10.5 🚫 禁止行为（明确清单）

- 🚫 **禁止取依赖未 satisfied 的 WP**（硬约束 1）
- 🚫 **禁止在 running 已 2 时锁定第 3 个**（硬约束 2）
- 🚫 **禁止非关键路径 WP 先于关键路径 WP 被取**（硬约束 3，候选含关键路径节点时）
- 🚫 **禁止返回 wp_def 前未完成 L2-02 锁定**（硬约束 5）
- 🚫 **禁止主动后台调度 / 推 WP**（违反"响应式 pull"边界）
- 🚫 **禁止用合并状态返回替代三态语义**（"没有 WP 可取"必须指明具体原因）
- 🚫 **禁止自己持久化 "上次谁"**（必须从 L2-02 读）
- 🚫 **禁止调度决策不走审计**

### 10.6 ✅ 必须职责（明确清单）

- ✅ **必须**响应每次 IC-02 get_next_wp 调用
- ✅ **必须**从 L2-02 读最新节点状态快照做筛选

- ✅ **必须**选定 top1 后走 L2-02 状态锁定

### 10.7 🔧 可选功能职责（可做但非硬性）

- 🔧 **候选解释**：返回 wp_def 同时附带"为什么选它"（关键路径 / 最小依赖深度 / 最早就绪）

## 11. L2-04 · WP 完成度追踪器 详细定义

### 11.1 职责 + 锚定

**一句话职责**：订阅 L1-04 Quality Loop 的 WP 完成 / 失败事件（含 WP-DoD 自检通过、WP commit、verifier verdict），在 L2-02 拓扑图上打点（running → done 或 running → failed），维护"项目级完成率 + 剩余工时 + 已完成 WP 清单 + 当前运行 WP 清单"四项聚合指标，向 L1-07 监督（"进度节奏"维度）和 L1-10 UI（🔧 WBS tab）暴露只读数据；同时在失败事件到达时把信号同步给 L2-05 做失败计数。

**上游锚定**：
- scope §5.3.1 职责（"每 WP 具备…… 推进…… 完成事件"）
- scope §5.3.3 In-scope 第 6 条（WP 完成度跟踪）
- scope §5.3.6 必须义务 "必须维护已完成 WP 清单 + 未完成 WP 剩余工时估算"
- BF-S4-02 WP 内 mini-PMP 流（完成事件来源）
- BF-S4-05 WP commit 流（完成事件来源）
- PM-10 事件总线单一事实源

**下游服务**：
- L2-02（申请状态跃迁）
- L2-05（失败事件同步）
- L1-07（"进度节奏"维度读数据）
- L1-10（UI 🔧 WBS tab 读数据）

---

### 11.2 输入 / 输出（文字级）

**输入**：
- 来自 L1-04 的 wp_done 事件（经事件总线订阅，携带：wp_id + verifier verdict 摘要 + commit_sha + 完成时间）
- 来自 L1-04 的 wp_failed 事件（携带：wp_id + fail_level + 失败原因摘要）

**输出**：
- 向 L2-02 发起状态跃迁申请（IC-L2-04）

### 11.3 边界

**In-scope**：
1. 订阅并消费 wp_done / wp_failed 事件
2. 向 L2-02 申请状态跃迁（合法性校验由 L2-02 负责）
3. 维护四项聚合指标（完成率 / 剩余工时 / 已完成清单 / 当前运行清单）
4. 并行度位释放（done / failed 都释放）
5. 向 L1-07 推送"进度节奏"维度数据
6. 向 L1-10 暴露只读 UI 数据
7. 失败事件同步给 L2-05
8. 跨 session 恢复时从事件总线补齐指标

**Out-of-scope**：
- 不做 WP 状态真值维护 → L2-02（本 L2 只申请跃迁）
- 不做调度决策 → L2-03
- 不做失败计数 + 回退建议 → L2-05
- 不做 WP 内部执行 → L1-04
- 不做 DoD 判定 → L1-04 S5
- 不做 UI 渲染 → L1-10（只给只读数据）

**边界规则**：本 L2 是"只做记账 + 转发"—— 不做判定、不做决策，只在事件到达时打点 + 聚合 + 广播。

### 11.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-10** 事件总线单一事实源（订阅驱动、重启可重放补齐）
- **PM-08** 可审计全链追溯（每次打点都有事件锚）
- **PM-04** WP 拓扑并行推进（并行度位释放由本 L2 执行）

**硬约束清单（文字描述）**：
1. **done / failed 必须都释放并行度位**：不得因状态不同而区别对待（否则会导致卡死并行度）。

4. **状态跃迁必经 L2-02**：本 L2 禁止直接改拓扑数据。
5. **聚合指标可从事件总线重建**：跨 session 恢复时本 L2 的所有指标必须可从事件总线完整重建。
6. **UI 数据是只读**：L1-10 不得通过本 L2 的通道反写（UI 反写通道是 L1-01 转发）。

**性能约束（文字描述）**：
- wp_done / wp_failed 事件到达后打点延迟：亚秒级；
- UI 拉取只读数据：亚秒级响应；
- 四项聚合指标重算：每次事件触发增量更新，避免整图扫描；
- 跨 session 重建：秒级可接受。

### 11.5 🚫 禁止行为（明确清单）

## 12. L2-05 · 失败回退协调器 详细定义

### 12.1 职责 + 锚定

**一句话职责**：追踪每个 WP 的连续失败次数（按同一 WP id 计数），到达 ≥ 3 次阈值时触发 BF-E-08 → 基于 L2-02 拓扑上下文 + 该 WP 历次失败原因摘要（读自事件总线）生成结构化回退建议（拆 WP / 改 WBS / 改 AC），把建议推给 L1-01 主 loop（由其转发 L1-07 监督做最终路径决策），同时把该 WP 在 L2-02 标 stuck（从调度候选池移除）；另外负责死锁场景（L2-03 返回 deadlock 时）的升级上报。

**上游锚定**：
- scope §5.3.4 硬约束 "WP 失败 ≥ 3 次 → 触发 BF-E-08 回退建议"
- scope §5.3.6 必须义务 "在 WP 失败 ≥ 3 次时触发 BF-E-08 回退建议"
- BF-E-08 WP 失败回退流
- BF-E-10 死循环保护流（与 BF-E-08 协同）
- PM-12 红线分级自治（失败回退属软红线 "WP 失败 < 3 次自动 replan" + 硬红线 "≥ 3 次升级"的边界实现）

**下游服务**：
- L2-02（标 stuck）
- L2-01（反向触发差量拆解 · 当用户 / L1-07 选"拆 WP"路径时）
- L1-01 主 loop（接收回退建议卡）
- L1-07 监督（最终决策走哪条 4 级回退）

---

### 12.2 输入 / 输出（文字级）

**输入**：
- 来自 L2-04 的失败同步信号（IC-L2-05，携带 wp_id + fail_level）
- 来自 L2-03 的 deadlock 语义通知（调度器返回 null + reason=deadlock 时同步）
- 来自事件总线的历次失败原因摘要（本 L2 主动读取）
- 来自 L1-07 的回退路径决策（IC-14 push_rollback_route）—— 用于判定某 WP 已进入回退流程

**输出**：
- 回退建议卡（推 L1-01，携带：wp_id + 3 选项建议 + 历次失败原因 + evidence 引用）
- IC-L2-06 标 stuck 请求（推 L2-02）
- IC-L2-07 差量拆解触发（推 L2-01，当路径选"拆 WP"时）
- 死锁升级上报（推 L1-01 → L1-07，含 deadlock 语义的整体分析）
- 全量追踪事件（经 IC-L2-08）

---

### 12.3 边界

**In-scope**：
1. 失败计数器维护（按 wp_id 累加）
2. 计数器重置规则（wp_done 时清零）
3. ≥ 3 次阈值识别
4. BF-E-08 回退建议生成（拆 WP / 改 WBS / 改 AC 三选一或组合）
5. 标 stuck（通过 L2-02 IC-L2-06）
6. 死锁场景识别与升级
7. 失败原因摘要汇总（从事件总线读）
8. 差量拆解触发（当路径决策选"拆 WP"）

**Out-of-scope**：
- 不做回退路径最终决策 → L1-07（本 L2 只给建议）
- 不做失败根因诊断 → L1-07 + L1-04（本 L2 只汇总历次原因）
- 不做 WP 状态真值维护 → L2-02
- 不做调度决策 → L2-03
- 不做完成率 / UI 指标 → L2-04
- 不做 WBS 拆解 → L2-01（只反向触发）
- 不做事件总线落盘 → L1-09
- 不做用户交互 → L1-10（只推卡片给 L1-01，L1-01 转发 UI）

**边界规则**：本 L2 是"识别 + 建议 + 释放"三件事 —— 识别失败阈值 / 建议回退方向 / 标 stuck 释放调度器，不参与最终决策。

---

### 12.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-04** WP 拓扑并行推进（本 L2 是失败兜底层）
- **PM-12** 红线分级自治（本 L2 划清软 / 硬边界）
- **PM-08** 可审计全链追溯（每次建议有历次失败证据链）
- **PM-10** 事件总线单一事实源（计数器可从事件回放重建）

**硬约束清单（文字描述）**：
1. **计数按"连续"识别**：只统计"连续失败"；一旦 wp_done 出现即清零。
2. **阈值 = 3 次**：不得擅自调高 / 调低；如需调整走 scope 变更流程。
3. **建议不等于决策**：本 L2 只产建议，不擅自执行；执行由 L1-07 决定 + L1-01 转发。
4. **标 stuck 必须经 L2-02**：禁止直接改拓扑数据。
5. **死锁识别必须及时**：L2-03 deadlock 通知必须立即触发升级上报，不得延迟到下一 tick。
6. **失败原因摘要必须可追溯**：每条建议附带事件总线的引用（event_id / commit_sha 等锚点）。
7. **计数器可跨 session 重建**：从事件总线回放 wp_failed / wp_done 事件即可恢复。

**性能约束（文字描述）**：
- 失败事件 → 计数更新 → 阈值判定：亚秒级；
- 回退建议生成：秒级（涉及事件总线读取 + 摘要汇总，可接受）；
- 死锁上报：立即触发，不延迟；
- 跨 session 重建：秒级。

7. **质量指标汇总**：单测覆盖率 / DAG 校验的 fuzz 测试（含环 / 悬空 / 自环 / 跨项目依赖等）/ 并行度越界的压力测试 / 跨 session 恢复的正确性测试 / IC-02 / IC-19 性能达标情况

## 附录 A · 术语（L1-03 本地）

| 术语 | 含义 |
|---|---|
| **WBS** | Work Breakdown Structure · 工作分解结构，把项目分层拆成可执行单元的树结构 |
| **WP（Work Package）** | 工作包 · WBS 最小可执行单元，粒度 ≤ 5 天工时 |
| **4 要素** | WP 必含的四项：独立 Goal / DoD 表达式入口 / 依赖清单 / 工时估算 |
| **DAG** | Directed Acyclic Graph · 有向无环图，WP 依赖关系必须满足 |
| **关键路径** | 从起点到终点工时之和最大的一条路径，影响项目总工期 |
| **并行度** | 同时处于 running 状态的 WP 数量，上限 ≤ 2 |
| **节点状态** | WP 在拓扑图上的状态，六种：ready / running / done / failed / blocked / stuck |
| **ready** | 可被调度但尚未选中，或依赖尚未全部 satisfied |
| **running** | 已被 L2-03 选中锁定，正在 L1-04 Quality Loop 执行 |
| **done** | WP-DoD 自检 + verifier 判定 PASS 后的终态 |
| **failed** | 单次失败态；若连续次数未达阈值会被 L2-02 放回 ready 允许重试 |
| **blocked** | 依赖链中某 WP 失败 / stuck 导致本 WP 无法推进 |
| **stuck** | 连续失败 ≥ 3 次 后被 L2-05 标记，从调度候选池移除，等待回退决策 |
| **拓扑图** | L2-02 维护的运行期真值源，含节点 + 依赖边 + 状态 + 关键路径标记 |
| **差量拆解** | 变更请求 / 失败回退触发时 L2-01 只处理未 done 部分，保留已完成 WP 身份 |
| **BF-E-08** | WP 失败回退流：某 WP 连续失败 ≥ 3 次触发，生成拆 WP / 改 WBS / 改 AC 建议 |
| **死锁（deadlock）** | 所有剩余 WP 均为 failed / blocked / stuck，无法推进 |
| **awaiting_deps** | 有 ready WP 但依赖未 done，需等待其他 WP 完成才能取 |
| **all_done** | 所有 WP 都已 done，可触发进 S7 收尾 |
| **回退建议卡** | L2-05 产出的结构化建议，含 wp_id + 3 选项（拆 WP / 改 WBS / 改 AC）+ evidence_refs |
| **推荐 skill** | L2-01 为每个 WP 给出的首选 skill 名，供 L1-05 调度参考（非硬绑定） |
| **wp_done 事件** | L1-04 Quality Loop S5 判 PASS 后广播的事件，含 wp_id + verdict 摘要 + commit_sha |
| **wp_failed 事件** | L1-04 某 WP 失败（WP-DoD 自检 / verifier 判 FAIL）时广播的事件，含 wp_id + fail_level |
| **一致性守护** | L2-02 对所有状态写入的强制校验角色（DAG / 并行度 / 跃迁合法性）|
| **状态跃迁** | 六状态之间的合法路径，例 ready → running / running → done / running → failed / failed → ready（重试）|
| **真值源** | L2-02 是 L1-03 内拓扑 + 节点状态的唯一真值来源；其他 L2 禁止自行维护副本真值 |
| **响应式 pull** | L2-03 只在 L1-01 主动 IC-02 调用时出手，不主动后台调度 |
| **docs/planning/wbs.md** | L2-01 产出的 WBS 标准产出物，按 PM-07 模板填充 |

**BF-S2-07 WBS 拆解流在 L2 的落位**：
1. 4 件套 + 9 计划 + 架构就绪 → L1-02 调 IC-19 → L2-01
2. L2-01 分层拆解 + 4 要素装配 + 粒度自检 + 推荐 skill → 产出 wbs.md
3. L2-01 → IC-L2-01 → L2-02 装图 + DAG 校验 + 关键路径识别 + 并行度守护初始化
4. 任一硬约束违反 → 整体拒绝 + 返回错误
5. 通过 → 状态固化 + 返回 L1-02 供 S2 Gate 审核

**BF-S4-01 WP 取任务流在 L2 的落位**：
1. L1-01 主 loop 决策"进 Quality Loop 前取 WP" → IC-02 get_next_wp → L2-03
2. L2-03 → IC-L2-02 → L2-02 读节点状态快照
3. L2-03 筛选（依赖 satisfied + ready + 并行度未越界）+ 关键路径优先排序
4. L2-03 → IC-L2-03 → L2-02 申请状态锁定 ready → running
5. L2-03 返回 {wp_id, wp_def, deps_met} 给 L1-01 → L1-01 续 IC-03 enter_quality_loop
6. 若无候选 → 三态语义返回（all_done / deadlock / awaiting_deps）

**BF-S4-02 WP 内 mini-PMP 流在 L2 的落位**：
1. L1-04 收到 wp_def → 在 WP 粒度跑 IMPL → 调 skill → 测试 → WP-DoD 自检
2. WP-DoD 自检 PASS → 触发 BF-S4-05 commit 流
3. WP-DoD 自检 FAIL → L1-04 广播 wp_failed 事件
4. （WP 内部推进不归 L1-03，L1-03 只接收完成 / 失败事件）

**BF-S4-05 WP commit 流在 L2 的落位**：
1. L1-04 完成 commit → 广播 wp_done 事件（含 commit_sha）
2. L2-04 订阅 → IC-L2-04 申请 running → done
3. L2-02 校验合法跃迁 + 写入 + 释放并行度位
4. L2-04 更新四项聚合指标 → 推 L1-07 / L1-10
5. 审计事件走 IC-L2-08

**BF-E-08 WP 失败回退流在 L2 的落位**：
1. L1-04 多次 wp_failed → L2-04 打点 → IC-L2-05 同步 L2-05
2. L2-05 累计连续失败计数；wp_done 清零
3. 计数 ≥ 3 → L2-05 触发 BF-E-08
4. L2-05 读事件总线失败摘要 + 生成 3 选项回退建议卡
5. L2-05 → IC-L2-06 → L2-02 标 WP stuck
6. L2-05 推建议卡 → L1-01 → 转发 L1-07 监督
7. L1-07 决策回退路径 → IC-14 push_rollback_route 回传 L1-01
8. L1-01 触发后续动作：
   - 选"拆 WP"→ L2-05 IC-L2-07 → L2-01 差量拆解 → L2-02 合并
   - 选"改 WBS / 改 AC"→ L1-02 走变更管理 → IC-19 二次调用 L2-01

**BF-E-10 死循环保护流在 L2 的落位**：
- L1-04 同级 FAIL ≥ 3 次 → 自动升级（属 L1-04 S5 职责）
- L1-03 L2-05 只对"同 WP 连续失败 ≥ 3 次"场景负责（与 BF-E-10 正交但协同）
- 死锁场景（整图 deadlock）由 L2-05 升级上报，由 L1-07 决策是否升级到重度 FAIL 回 S2

**BF-E-02 跨 session 恢复在 L2 的落位**：
- system_resumed 事件 → L2-02 从 checkpoint / 事件总线重建拓扑与节点状态
- L2-04 从事件总线回放 wp_done / wp_failed 历史重建四项指标
- L2-05 从事件总线回放重建连续失败计数器
- 全部就绪 → 广播 "WBS 就绪" → L1-01 下一 IC-02 正常调度
