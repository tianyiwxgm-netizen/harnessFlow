## 2. L2 清单（5 个）

| L2 ID | 名称 | 一句话职责 | 聚合自 BF | 核心问题 |
|---|---|---|---|---|
| **L2-01** | 3 层分层管理器 | 维护 Session / Project / Global 三层物理隔离 + scope 优先级规则 + Session 7 天过期 + 跨项目只走 Global 的隔离守则 | 全 BF（分层底座） | 谁管分层边界 |
| **L2-02** | KB 读（分层查询） | 响应 `kb_read` 请求，按 scope 优先级合并 + kind 过滤 + `applicable_context` 匹配 + 返回条目集合 | BF-L3-05 read 路径 | 怎么读 |
| **L2-03** | 观察累积器（Session 层） | 接受 L1-01/L1-04/L1-07 的观察信号，写入 Session KB 并自动累计 `observed_count` / `first_observed_at` / `last_observed_at`，命中同类条目时合并而非新建 | BF-L3-05 write 路径 + BF-X-05 反向收集 | 怎么写 + 怎么攒够证据 |
| **L2-04** | KB 晋升仪式执行器 | Session → Project 自动晋升（observed_count ≥ 2）或用户批准晋升；Project → Global 必须用户显式批准 或 observed_count ≥ 3；S7 收尾时批量走仪式 | BF-S7-04 晋升仪式流 | 怎么从候选升到正式 |
| **L2-05** | 检索 + Rerank 上下文相关性排序器 | 在 L2-02 召回候选条目后按「当前阶段 + 任务类型 + 技术栈 + observed_count + last_observed_at」多信号 rerank，截断 top_k 返回；阶段切换时按注入策略表主动推送 | BF-X-05 注入策略 + BF-L3-05 排序步骤 | 怎么排序选最相关 |

**为什么切成 5 个 L2（brainstorming 结论）**：
- **L2-01 是底座**：所有读 / 写 / 晋升都落在 3 层物理存储 + 层间规则上，分层正确性是系统可信的前提。
- **L2-02 和 L2-05 分开**：召回（L2-02）和 rerank（L2-05）是 KB 检索的两个阶段，一个管「找对了吗」，一个管「排好了吗」，两者变更节奏不同（L2-02 贴 scope，L2-05 贴阶段）。
- **L2-03 和 L2-04 分开**：观察累积（L2-03）是"写 Session"的日常热路径，晋升仪式（L2-04）是 S7 的低频仪式路径，两者触发机制、运行节奏、用户干预方式完全不同。

**关键规则**：
- **L2-01 是底座**：其他 4 个 L2 都依赖它的分层规则；任何跨层操作必须经 L2-01 的守则检查。
- **L2-02 / L2-05 协作形成检索链路**：先召回后 rerank，不可跳过 rerank 直接返回原始顺序。
- **L2-03 / L2-04 形成写入 → 晋升链路**：Session 候选必须通过 L2-04 仪式才能升到 Project / Global。
- **所有 L2 的读写事件都必须走 L1-09**（IC-09），这是 L1-06 自身可审计的唯一手段。

## 5. L2 间业务流程（6 条）

### 流 A · 决策前 KB 读（L1-01 主 loop 最高频路径）

```
[L1-01 主 loop 进入某个决策点]
     ↓ IC-06 kb_read(kind?, scope?, context_filter, top_k?)
L2-02 KB 读入口
     ↓
L2-01 校验分层规则（允许读哪些 scope）
     ↓
L2-02 按 scope 优先级 S > P > G 召回候选
     ↓
L2-02 按 kind 过滤（pattern / trap / recipe / tool_combo / anti_pattern / ...）
     ↓
L2-02 按 applicable_context 匹配（route / task_type / 技术栈）
     ↓ 候选集
L2-02 调用 L2-05 rerank
     ↓
L2-05 多信号打分（相关性 + observed_count + last_observed_at 近因）
     ↓
L2-05 截断 top_k
     ↓
L2-02 返回 entries 给 L1-01
     ↓
所有步骤事件 → IC-09 append_event → L1-09
```

### 流 B · L1-04 写 Session KB 候选（发现新 trap / pattern）

```
[L1-04 Quality Loop 运行中 S4 / S5]
     ↓ 识别到新经验（FAIL 原因 / 有效组合 / 防假完成 trap）
     ↓ IC-07 kb_write_session(entry)
L2-03 观察累积器入口
     ↓
L2-03 schema 校验 → 失败则拒绝 + 告警
     ↓ 通过
L2-03 查重：title + kind 命中 Session 层已有条目？
     ├── 命中 → 合并：observed_count +1 + last_observed_at 更新
     └── 未命中 → 新建 Session 条目（first/last_observed_at = 现在, count=1）
     ↓
L2-01 分层规则写入 Session 存储
     ↓
L2-03 返回 {id, observed_count_after}
     ↓
事件 → IC-09 append_event
```

### 流 C · 阶段切换主动注入（BF-X-05）

```
[L1-02 state_transition 完成]
     ↓ stage_changed 事件广播 → L1-09
L2-05 订阅事件 → 触发
     ↓
L2-05 查阶段注入策略表（S1→trap+pattern / S2→recipe+tool_combo / ...）
```

## 8. L2-01 · 3 层分层管理器 详细定义

### 8.1 职责 + 锚定

**一句话职责**：HarnessFlow 知识库的"分层底座" —— 维护 Session / Project / Global 三层物理隔离与 scope 优先级规则（S > P > G），保障跨项目不共享 Project、Session 7 天过期自治、kind 分类（pattern / trap / recipe / tool_combo / anti_pattern / project_context / external_ref / effective_combo）的字段定义统一，并作为所有 L2 读写、晋升、检索的"必经关卡"保证 3 层规则不可绕过。

**上游锚定**：
- Goal 附录 A 术语："KB 三层 = Global / Project / Session 知识库"
- scope §5.6.1 职责（"维护 3 层 KB"）
- scope §5.6.3 In-scope 第 1 条 "3 层物理存储（jsonl / md 文件系统）"
- scope §5.6.4 硬约束 1-4 全部（Session 7 天过期 / Global 需 ≥3 或用户批准 / schema validate / 注入策略不可绕）
- PM-06 KB 三层 + 阶段注入

**下游服务**：
- L2-02（读前校验层）
- L2-03（写前申请位）
- L2-04（晋升前校验跨层规则）
- L2-05（读阶段策略时查询 3 层物理布局）

---

### 8.2 输入 / 输出（文字级）

**输入**：
- 来自 L2-02 的"读 scope 校验请求"（问：当前项目允许读哪些层 / 哪些项目隔离边界）
- 来自 L2-03 的"写 Session 申请"（问：给当前会话分配 Session 存储位置）
- 来自 L2-04 的"跨层晋升规则校验请求"（问：某条目从 A 层升到 B 层是否合规）
- 来自后台定时器的"Session 7 天过期扫描触发"
- 来自 L1-02 的"项目激活事件"（新建项目 / 跨 session 恢复）

**输出**：
- 返回 L2-02 的"允许读的 scope 列表 + 项目隔离上下文"
- 返回 L2-03 的"Session 写入位分配"
- 返回 L2-04 的"晋升规则判定结果（允许 / 拒绝 + 拒绝原因）"
- 后台任务产出的"本轮过期条目清单"
- 项目激活时广播的"KB 分层就绪"事件

---

### 8.3 边界

**In-scope**：
1. Session / Project / Global 三层物理隔离维护
2. scope 优先级规则定义（读合并时 S > P > G）
3. 跨项目隔离（Project 层按 project_id 锁定）
4. Session 7 天过期标记（不物理删除，保留审计）
5. kind 分类统一定义（8 类 kind 的白名单维护）
6. schema 定义一致性（所有层共用同一个条目定义基座）
7. 项目激活 / 切换时的分层上下文准备

**Out-of-scope**：
- 不做具体的读操作 → L2-02
- 不做具体的写操作 → L2-03
- 不做相关性排序 → L2-05
- 不做晋升判定逻辑 → L2-04
- 不做 schema 字段设计（设计走技术方案阶段）→ 3-1/L1-06
- 不做向量索引 / embedding → scope §5.6.3 明禁

**边界规则**：本 L2 是"守门人"，只管"分层规则是否被遵守"，不执行具体读写业务。

---

### 8.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-06** KB 三层 + 阶段注入（本 L2 是 PM-06 的物理实现载体）
- **PM-10** 事件总线单一事实源（所有分层变更走 IC-09）

**硬约束清单（文字描述）**：
1. **三层物理隔离硬性不可合并**：Session / Project / Global 必须独立存储位，禁止共用。
2. **跨项目只走 Global**：两个 project_id 之间的知识共享只能通过 Global 层，Project 层按 project_id 严格隔离。
3. **Session 条目 7 天生命期**：会话结束后 7 天内未晋升自动过期标记。
4. **kind 白名单封闭**：8 类 kind 固定，禁止运行时扩展；未知 kind 的写入请求直接拒绝。
5. **schema 校验不可旁路**：任何写入 / 晋升必须先通过 schema validate。
6. **晋升仅单向且不跨级跳跃**：只能 Session → Project、Project → Global；禁止 Session 直接 → Global。
7. **过期条目仍可审计追溯**：物理上保留，逻辑上从 L2-02 召回结果剔除。

**性能约束（文字描述）**：
- 分层规则校验延迟：毫秒级（不应成为读写热路径瓶颈）。
- 项目激活时的分层上下文加载：秒级以内（取决于 Project KB 条目总数，条目规模超过阈值后由 3-1 技术方案处理分片）。
- 7 天过期扫描：后台低优先级任务，不抢占热路径资源。

---

### 8.5 🚫 禁止行为（明确清单）

- 🚫 **禁止两个项目共享同一 Project KB 存储位**（跨项目只走 Global）
- 🚫 **禁止 Session 条目直接升 Global**（必须走 Project 中转）
- 🚫 **禁止运行时注册新 kind**（8 类封闭）
- 🚫 **禁止跳过 schema 校验写入**（硬约束 5）
- 🚫 **禁止物理删除过期 Session 条目**（保留审计）
- 🚫 **禁止对层边界破例**（例：让某条 Session 条目被其他项目读到）
- 🚫 **禁止将分层规则变更做成"配置项"而非"硬约束"**（PM-06 级不可绕）

---

### 8.6 ✅ 必须职责（明确清单）

- ✅ **必须**维护 Session / Project / Global 三层独立的存储位
- ✅ **必须**提供 scope 优先级规则（S > P > G）的统一查询入口
- ✅ **必须**对 Session 条目执行 7 天生命期扫描 + 过期标记
- ✅ **必须**对所有写请求执行 schema 校验（未通过即拒绝）
- ✅ **必须**在项目激活 / 切换时广播 KB 分层就绪事件
- ✅ **必须**对晋升请求提供跨层规则判定（允许 / 拒绝）
- ✅ **必须**对所有分层变更事件走 IC-09 落盘

---

### 8.7 🔧 可选功能职责（可做但非硬性）

- 🔧 **分层容量监控**：当 Session / Project KB 条目数接近阈值时发出 INFO 告警（用于 retro 分析）
- 🔧 **跨项目知识迁移工具**：由用户明示触发的"把某项目某条目推荐到 Global"辅助（仍走 L2-04 仪式）

## 9. L2-02 · KB 读（分层查询） 详细定义

### 9.2 输入 / 输出（文字级）

2. kind 过滤（支持单 kind / 多 kind / 无过滤）
3. `applicable_context` 匹配（route / task_type / 技术栈多条件）
4. 调用 L2-05 做 rerank + 截断
5. 读失败降级路径（返回空 + 告警，不 halt）
6. 读路径的审计事件落盘

**Out-of-scope**：
- 不做 rerank 打分逻辑 → L2-05
- 不做写入 / 晋升 → L2-03 / L2-04
- 不做分层规则维护 → L2-01
- 不做向量检索（scope §5.6.3 明禁）
- 不做自然语言问答（scope §5.6.3 + §5.6.5 🚫 明禁）

**边界规则**：本 L2 是"召回层"，具体相关性排序不归它管；只保证"能召回 + 过滤对"。

### 9.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-06** KB 三层 + 阶段注入（本 L2 实现"读"侧）
- **PM-10** 事件总线（所有读请求走 IC-09 审计）

**硬约束清单（文字描述）**：
1. **scope 优先级不可打破**：合并结果必须 S > P > G；同 id 冲突时高优先级覆盖低优先级。
2. **applicable_context 过滤必跑**：即使调用方不提供过滤条件，也要跑「当前项目 + 当前阶段」基础过滤。
3. **读失败不可 halt 整个系统**：底层不可达时返回空 + 降级信号，交给调用方决定是否无 KB 继续。
4. **rerank 不可跳过**：召回后必须经 L2-05 rerank，禁止直接返回原始顺序。
5. **读请求必须走 L2-01 校验**：禁止绕过分层管理器直接访问物理存储。

**性能约束（文字描述）**：
- 常规 kb_read 请求 P99 延迟：亚秒级（KB 是决策前置动作，不能卡主 loop）。
- 单次召回规模上限：实现上应有 top_k 和候选集上限，防止无界扫描。
- 高并发读：多个 L1（主 loop + Quality Loop + Supervisor）并发读时不应互相阻塞。

### 9.5 🚫 禁止行为（明确清单）

- 🚫 **禁止跳过 L2-05 rerank 直接返回原始顺序**（相关性不可省）

## 10. L2-03 · 观察累积器（Session 层） 详细定义

## 12. L2-05 · 检索 + Rerank 上下文相关性排序器 详细定义

### 12.1 职责 + 锚定

**一句话职责**：在 L2-02 召回候选条目集合后执行"多信号相关性打分 + 截断 top_k"的 rerank；并在 L1-02 阶段切换事件到达时按 PM-06 注入策略表主动触发反向召回 + rerank + 推送到 L1-01 决策上下文，作为 KB 3 层检索链路的"排序层"和"主动注入入口"。

**上游锚定**：
- scope §5.6.6 必须义务 1 "按 scope 优先级读（Session > Project > Global）" + 义务 2 "按 `applicable_context` 过滤"
- scope §5.6.6 必须义务 3 "必须在阶段切换时按注入策略表推条目给 L1-01"
- BF-L3-05 read 路径第 4 步 "排序（observed_count desc）"（本 L2 在此基础上多信号化）
- BF-X-05 KB 注入策略执行流 + PM-06 注入策略表
- 任务说明特定背景："rerank（上下文相关性排序）"

**下游服务**：
- L2-02（被其回调做 rerank + 截断）
- L1-01（阶段切换主动推送条目集）
- L1-02（订阅 stage_changed 事件）

---

### 12.2 输入 / 输出（文字级）

**输入**：
- 来自 L2-02 的"召回候选集 + 请求上下文"（候选条目、当前请求的 applicable_context、top_k）
- 来自 L1-02（经 L1-09 事件总线）的 stage_changed 事件（stage_from / stage_to / project_id）
- 来自 L2-01 的分层规则查询结果（用于主动注入时确认可读层）

**输出**：
- 返回 L2-02 的"排序后条目集 + 每条打分理由"
- 阶段切换主动注入：向 L1-01 推送"本阶段建议条目集"（含注入类别 + rerank 后 top_k）
- rerank 过程事件（含打分概要，走 IC-09）

---

### 12.3 边界

**In-scope**：
1. 多信号 rerank（相关性、observed_count、last_observed_at 近因、阶段匹配、applicable_context 匹配程度）
2. 截断 top_k
3. 阶段切换主动注入触发（订阅 stage_changed）
4. 注入策略表应用（PM-06：S1→trap+pattern / S2→recipe+tool_combo / S3→anti_pattern / S4→pattern / S5→trap / S7→反向收集）
5. 阶段主动推送给 L1-01（不等主 loop 主动 read）

**Out-of-scope**：
- 不做物理召回 → L2-02
- 不做向量 embedding（scope §5.6.3 明禁）
- 不做学习型排序（本 L1 V1-V2 不自学习）
- 不做分层规则维护 → L2-01
- 不做写 / 晋升 → L2-03 / L2-04

**边界规则**：本 L2 只管"排序 + 阶段注入推送"，不动存储；reank 权重是可调的"策略"而非硬约束（3-1 技术方案定参数）。

---

### 12.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-06** KB 三层 + 阶段注入（本 L2 实现"阶段注入"侧）

**硬约束清单（文字描述）**：
1. **注入策略表不可绕**：S1→trap+pattern / S2→recipe+tool_combo / S3→anti_pattern / S4→pattern / S5→trap / S7→反向收集。scope §5.6.5 禁止清单 7。
2. **阶段切换主动注入必跑**：state_transition 完成后必须至少尝试一次主动注入。
3. **不做向量检索**：rerank 纯结构化多信号；禁止引入 embedding 模型。
4. **top_k 截断必发生**：调用方未显式给 top_k 时使用策略级默认值，但必须截断（防注入膨胀）。
5. **rerank 不得改变条目内容**：只改排序与筛选，不得篡改任何字段。
6. **失败降级**：rerank 异常时 fallback 到"原始召回顺序 + 硬截断"（不 halt）。

**性能约束（文字描述）**：
- rerank 延迟：应显著低于召回延迟，不成为读热路径瓶颈（亚百毫秒级）。
- 主动注入延迟：阶段切换完成后到条目推送到 L1-01 上下文的总延迟控制在秒级。
- 注入策略表查表本身：毫秒级。

## 附录 A · 术语（L1-06 本地）

| 术语 | 含义 |
|---|---|
| **KB 三层** | Session / Project / Global 知识库 3 层物理存储 |
| **Session KB** | 本次会话临时层，默认生命期 7 天 |
| **Project KB** | 跨 session 本项目层，项目级共享 |
| **Global KB** | 跨项目永久层，所有项目可读 |
| **scope 优先级** | 读合并时 Session > Project > Global |
| **kind** | 条目类别：pattern / trap / recipe / tool_combo / anti_pattern / project_context / external_ref / effective_combo（8 类白名单） |
| **applicable_context** | 条目的适用条件（route / task_type / 技术栈） |
| **observed_count** | 条目被观察累积次数（自动累计，合并同类时 +1） |
| **first_observed_at** | 条目第一次被观察的时刻 |
| **last_observed_at** | 条目最近一次被观察的时刻（用于近因排序） |
| **注入策略表** | PM-06 按阶段映射 kind 的表（S1→trap+pattern / S2→recipe+tool_combo / S3→anti_pattern / S4→pattern / S5→trap / S7→反向收集） |
| **晋升仪式** | Session → Project（observed_count ≥ 2 或用户批准）/ Project → Global（≥ 3 或用户批准）的正式迁层动作 |
| **rerank** | 在召回候选上做"多信号相关性打分 + top_k 截断"的排序层 |
| **主动注入** | 阶段切换完成后 L2-05 不等主 loop 主动 read 即把条目推到 L1-01 决策上下文 |
| **Session 7 天过期** | Session 条目会话结束后 7 天未晋升自动过期标记（不物理删除，保留审计） |
| **跨层跳跃** | Session 直接升 Global（硬禁）；必须 Session → Project → Global |
| **用户批准路径** | 晋升的 "或" 分支 —— observed_count 未达阈值时由用户显式批准可旁路阈值 |
