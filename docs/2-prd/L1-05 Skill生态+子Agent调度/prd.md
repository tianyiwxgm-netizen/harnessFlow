# L1-05 · Skill 生态 + 子 Agent 调度能力 · PRD

> **版本**：v1.0（5 个 L2 产品级完备；L3 实现设计留给 `docs/3-1-Solution-Technical/L1-05/tech-design.md`）
> **定位**：HarnessFlow 的**调度层 + 适配层** —— 把"主 Agent 要某个能力"的抽象请求翻译成对真实 skill / 子 Agent / 原子工具的调用，并在失败时走降级链；它是 Goal §2 所说"调度整个 Skill 生态的高阶调度器"的落地载体。
> **严格遵循**：本 PRD **不得与** `docs/2-prd/L0/scope.md §5.5` 冲突。如冲突以 scope 为准。
> **严格边界**：本 PRD 只描述"要做什么"（职责 / 边界 / 约束 / 禁止 / 必须 / 交互 / 验收）；"怎么做"（算法 / schema / 配置 / 状态机实现 / 子 Agent 进程模型）迁到 `docs/3-1-Solution-Technical/L1-05/tech-design.md`。
> **PM-14 项目上下文声明**：**所有 Skill 调用 / 子 Agent 委托的 context 必须携带 `harnessFlowProjectId`**（PM-03 独立 session 不意味着可以丢 project_id —— 相反，委托时必须把 project_id 作为受限 context 的必填字段传给子 Agent）；异步结果回传时事件必带 project_id 以便路由回正确 project 的主 loop。详见 `docs/2-prd/L0/projectModel.md` §9.1（L1-05 使用 project_id 方式）。

## 2. L2 清单（5 个）

| L2 ID | 名称 | 一句话职责 | 聚合自 BF | 核心问题 |
|---|---|---|---|---|
| **L2-01** | Skill 注册表（能力抽象层与可用性账本）| 维护"能力点 → skill 候选链"映射 + skill 可用性 / 版本 / 成本 / 历史成功率的账本 + 子 Agent 注册表 + 原子工具柜元数据；作为 L2-02 / L2-03 / L2-04 的单一数据源 | BF-X-07（能力抽象层载体）+ BF-L3-02 / 03 / 04（被查询入口）| 候选池从哪来 |
| **L2-02** | Skill 意图选择器（能力点 → skill 候选链）| 接收"要某能力点 + 约束（成本上限 / 超时 / 质量偏好）"的请求，从 L2-01 取候选 → 按可用性 / 成本 / 历史成功率 / 失败记忆排序 → 产出"首选 + fallback 链"供 L2-03 / L2-04 消费 | BF-L3-02 优先级排序 + BF-X-07 | 怎么选 + 怎么排 |
| **L2-03** | Skill 调用执行器（调用 · 签名登记 · fallback 驱动）| 按 L2-02 产出的链执行 skill 调用：记入参 hash / skill 版本 / 结果 / 耗时签名 → 失败时沿 fallback 链前进（2 备选 → 内建逻辑 → 硬暂停）→ 通过 L2-05 做回传 schema 校验 → 返回结构化结果给调用方；原子工具（Read/Write/Bash/Grep/Glob/WebSearch/WebFetch/Playwright/MCP）也走本 L2 签名登记 | BF-L3-02 调用 + BF-L3-04 工具调用 + BF-E-05 降级链 | 怎么调 + 失败怎么办 |
| **L2-04** | 子 Agent 委托器（独立 session 生命周期管理）| 按 IC-05 / IC-12 / IC-20 发起子 Agent 委托：从 L2-01 查子 Agent 元数据 → 包只读 context 副本 + 明确 goal + 工具白名单 → 启动独立 session → 监控进度 / 心跳 / 超时 → 超时或 crash 时 kill + 回收 + 走 BF-E-09 降级；不承担子 Agent 业务逻辑本身 | BF-L3-03 + BF-E-09 + PM-03 + BF-S5-01 | 怎么开子 Agent + 怎么保证不污染主 session |
| **L2-05** | 异步结果回收器（schema 校验 · 超时 · 结果回传）| 所有异步返回（skill 结构化回传 / 子 Agent 结构化报告 / 长时工具调用结果）的统一回收点：按对应能力的 schema 校验返回值 → 校验失败算失败 + 触发 fallback → 校验通过转发给原始调用方 + 落审计事件；承担"结构化回传正确性"的守门责任 | BF-L3-02 接收结构化回传 + BF-L3-03 返回结果 + BF-E-05 / BF-E-09 失败判定 + Goal §4.1 可追溯 | 回来的东西对不对 |

**为什么切成 5 个 L2（brainstorming 结论）**：

1. **L2-01 注册表独立**：candidate pool、可用性探测、版本与成功率账本是"数据资产"，节奏是"低频更新、高频读"；与"选谁"（L2-02）和"怎么调"（L2-03）变化节奏不同，必须作为独立底座。
2. **L2-02 选择器与 L2-03 执行器分开**：选择是**策略**（可观察、可 A/B、可回放），执行是**动作**（副作用 + 签名），两者变化节奏和测试方式完全不同；合并会让"为什么选它"的可追溯变模糊，违反 Goal §4.1。
3. **L2-04 子 Agent 委托单独成 L2**：子 Agent 有"独立 session"边界 + "生命周期"（启动 / 心跳 / 超时 / kill / 回收）+ "context 副本"隔离三层复杂度，完全不同于同步 skill 调用；如果把它塞进 L2-03，L2-03 会同时承担同步/异步两种完全不同的生命周期模型，爆炸。
4. **L2-05 回收器独立**：所有异步回传都要做 "schema validate + 决定算成功还是失败"，这是"正确性的最后一道门"；放 L2-03 / L2-04 任何一方都会让另一方绕过校验，违反 scope §5.5.6 义务 6"回传必 schema validate"。独立成 L2 才能把"正确性守门"做成跨路径的统一关卡。
5. **4 vs 5 的抉择**：曾考虑把 L2-05 合到 L2-04（理由：子 Agent 回传更复杂，skill 回传简单）；但 IC-04 `invoke_skill` 也常是 LLM 驱动的结构化返回，错误率不比子 Agent 低；独立一个 L2 成本换来"不会漏校验"的硬保证，值得。

**关键规则**：
- **L2-01 是单一候选池来源**：L2-02 / L2-03 / L2-04 都只从这一处取候选，禁止任何 L2 内部硬编码 skill 名。
- **L2-02 / L2-03 / L2-04 同层并列**：选择、同步执行、异步委托是三种并列动作；L2-02 产出的"链"不限定给 L2-03 还是 L2-04，由调用请求的类型（invoke_skill 还是 delegate_subagent）决定。
- **L2-05 是所有回传的唯一回收口**：L2-03 的 skill 回传、L2-04 的子 Agent 报告都走它做 schema 校验；这是把 Goal §4.1 "可追溯 100%" 中"结构化正确性"一环落到单点的设计。
- **事件面（L1-09）横切所有**：每一次调用 / 心跳 / 超时 / 降级 / 校验结果都必须走 IC-09，L1-05 自身可审计的唯一手段。

## 5. L2 间业务流程（8 条）

### 流 A · IC-04 invoke_skill 同步调用（主 loop 最高频路径）

```
[L1-01 主 loop 在决策心跳选择"调 skill"]
     ↓ IC-04 invoke_skill(capability, params, constraints)
L1-05 入口分派
     ↓
L2-02 Skill 意图选择器
     · 读 L2-01 候选池（按 capability）
     · 按约束（max_cost / max_timeout / preferred_quality）过滤
     · 按可用性 / 成本 / 历史成功率 / 失败记忆排序
     · 产出 "首选 + fallback 链"
     ↓ 能力链
L2-03 Skill 调用执行器
     · 取链首项 → 调用
     · 签名登记（params_hash / version / attempt=1）
     · 等待返回
     ├── 成功 → L2-05 schema 校验
     │         ├── 校验通过 → 返回结果给 L1-01
     │         └── 校验失败 → 记 validate_failed → 链前进（attempt=2）
     └── 失败 → 记 skill_call_failed → 链前进（attempt=2）
     ↓
（上述 fallback 循环到链尽头仍失败）
     ↓
L2-03 抛 capability_exhausted → L1-01
     ↓
每步事件走 IC-09 → L1-09
```

### 流 B · IC-05 delegate_subagent 子 Agent 委托（S5 verifier 路径典型）

```
[L1-04 Quality Loop S5 阶段完成 S4 artifacts]
     ↓ IC-20 delegate_verifier(s3_blueprint, s4_artifacts, dod_exprs)
     ↓（等价于 IC-05，指向 "verifier" 子 Agent 能力点）
L1-05 入口分派
     ↓
L2-02 选择器
     · 读 L2-01 "verifier" 子 Agent 候选（通常只 1 个主选 + 简化版备选）
     · 产出能力链
     ↓
L2-04 子 Agent 委托器
     · 裁剪 context 副本（只含验证所需最小子集）
     · 附 goal + tools_whitelist + 返回 schema 指针
     · 启动独立 session（子进程 / 子对话 / 子上下文，由 3-1 技术方案定）
     · 记事件 "subagent_started(verifier, session_id)"
     · 进入监控循环：
       ├── 定期收心跳 → 记 "subagent_heartbeat"
       ├── 超时（5 分钟无进展）→ kill + 回收 → 走响应面 2
       └── 子 Agent 主动完成 → 获取报告
     ↓ 子 Agent 报告（JSON）
L2-05 schema 校验
     ├── 通过 → 返回 {verifier_report_json} 给 L1-04
     └── 失败 → 记 validate_failed → 走 BF-E-09 降级
          · 重试 1 次（新 session）
          · 仍失败 → 简化版 verifier
          · 再失败 → 抛 subagent_capability_exhausted
     ↓
每步事件 IC-09 → L1-09
```

### 流 C · BF-L3-04 原子工具调用（Read/Grep/Bash/...）

```
[任意 L1 需要原子 I/O]
     ↓ IC-04 invoke_skill(capability="tool:Read", params={path: ...})
L1-05 入口
     ↓
L2-02 识别为"工具能力点"（不走 LLM skill 路径，但仍统一抽象）
     · 从 L2-01 工具柜元数据直查
     · 工具通常无 fallback（或 fallback 为"提示用户手工查"）
     ↓
L2-03 执行器
     · 调用对应原子工具
     · 签名登记（tool_name / params_hash / duration）
     · 返回结果
     ↓
L2-05 轻量校验（工具返回结构固定，简单字段检查）
     ↓ 通过
返回给调用方
     ↓ 事件 IC-09 → L1-09（含 tool_name / path / duration）
```

### 流 D · Skill 失败降级链（BF-E-05）

```
[L2-03 首选 skill 调用失败]
     ↓
L2-03 记 "skill_call_failed(skill_id, reason, attempt=1)"
     ↓
L2-03 沿 L2-02 提供的链前进 → 备选 1（attempt=2）
     ├── 成功 → L2-05 校验 → 返回
     └── 失败 → 前进 → 备选 2（attempt=3）
         ├── 成功 → 返回
         └── 失败 → 前进 → 内建逻辑（attempt=4）
             ├── 成功 → 返回（标记 "builtin_fallback_used"）
             └── 失败 → 抛 "capability_exhausted"
                        → 调用方决定硬暂停（走 L1-07 硬红线）
     ↓
全程每步 IC-09 落事件
     ↓
L2-02 更新 L2-01 失败记忆（累计次数 + 最后失败时间）
     ↓ 供下次同能力请求降权排序
```

### 流 E · 子 Agent 失败降级（BF-E-09）

```
[L2-04 子 Agent crash / 超时 / 校验失败]
     ↓
L2-04 kill + 回收 + 记事件
     ↓
判定：是否属于同名子 Agent 本轮第几次失败？
     ├── 第 1 次 → 同名重试 1 次（新 session，新 context 副本）
     │   ├── 成功 → 返回
     │   └── 失败 → 进入第 2 次判定
     └── 第 2 次 → 标记 degraded → 走降级版
         · verifier → 简化版 DoD eval
         · retro-generator → 最简模板填充
         · architecture-reviewer → 跳过（只记事件 "review_skipped"）
         · failure-archive-writer → 降级为纯事件归档
     ↓
降级仍失败 → 抛 subagent_capability_exhausted → 调用方决定硬暂停
     ↓
L2-01 账本更新子 Agent 失败记忆
全程 IC-09 落事件
```

### 流 F · 能力抽象层查询（BF-X-07）

```
[L2-02 接到请求：能力点=X]
     ↓
查 L2-01：
     · 返回该能力点的候选列表
     · 每候选附：skill_id / version / 可用性 status / 估算 cost / 历史成功率 / 最后失败时间
     ↓
L2-02 按以下权重排序（文字描述，具体权重 3-1 定）：
     · 当前可用性（必要条件，不可用直接剔除）
     · 成本约束（调用方给的 max_cost）
     · 历史成功率（近 N 次调用的成功率，N 由 3-1 定）
     · 失败记忆（最近连续失败 → 降权）
     · 成本偏好（如果调用方给了 preferred_quality 优先高质量候选）
     ↓
L2-02 输出"首选 + fallback 链" 给 L2-03 / L2-04
     ↓
事件 IC-09（type=capability_resolved, candidates=[...]）
```

### 流 G · 子 Agent 启动 → 心跳 → 回收完整生命周期

```
[L2-04 委托请求进入]
     ↓
① 裁剪 context 副本（只读）
② 拼子 Agent 启动包：
   · goal 文字描述
   · 工具白名单（从 L2-01 取该子 Agent 默认 + 调用方附加）
   · 返回 schema 指针
   · 超时时限（默认 5 分钟，可被能力元数据覆盖）

     ↓
④ 监控循环（每 N 秒，N 由 3-1 定）：
   · 读子 Agent 心跳信号
   · 记 subagent_heartbeat 事件
   · 检查是否超过"无进展"阈值
     ↓ 超过
⑤ kill（先 SIGTERM 后 SIGKILL）→ 回收资源
   · 回收临时目录
   · 回收占用的文件锁
   · 记 subagent_killed 事件
     ↓
⑥ 无超时情况下 → 子 Agent 主动上报完成
   · 读取结构化报告
   · 交 L2-05 校验
     ↓
⑦ 无论成功失败都走 IC-09 终局事件（含 final_status / duration / 产出摘要）
```

```
       → 可追溯到具体 skill 的某次调用 + 入参 hash + 耗时 + 结果
     ↓
```

## 6. IC-L2 契约清单（10 条 · 一句话 + 方向）

> 本节严格遵守 v0.2 规范：只写「一句话意义 + 方向」，不含字段级 schema。字段定义留给技术方案阶段（docs/3-1-Solution-Technical/L1-05）。

| IC ID | 调用方 → 被调方 | 方向 | 一句话意义 |
|---|---|---|---|
| **IC-L2-01** | L2-02 → L2-01 | 内部 | 接到能力点请求后向注册表查 "该能力的候选列表 + 每候选的元数据（可用性 / 版本 / 成本 / 历史成功率 / 失败记忆）" |
| **IC-L2-02** | L2-04 → L2-01 | 内部 | 委托前向注册表查 "子 Agent 元数据（默认工具白名单 / 超时 / 返回 schema 指针 / 默认 goal 模板）" |
| **IC-L2-03** | L2-03 → L2-02 | 内部 | skill 首选失败后请求 "下一候选" (fallback 前进) ·L2-02 返回链上下一项 |
| **IC-L2-04** | L2-04 → L2-02 | 内部 | 子 Agent 首选失败后请求 "下一备选子 Agent 或降级简化版" |
| **IC-L2-05** | L2-03 / L2-04 → L2-05 | 内部 | 收到异步回传（skill / 子 Agent）后交给回收器做 schema 校验 |
| **IC-L2-06** | L2-05 → L2-01 | 内部 | 校验时按 capability 查注册表取 "回传 schema 指针"（实际 schema 定义在 3-1） |
| **IC-L2-07** | L2-02 → L2-01 | 内部 | fallback 结束后 (无论成功失败) 回写 "失败记忆 / 成功率" 账本更新 |
| **IC-L2-08** | 全 L2-* → L1-09 | 外部（经 IC-09） | 每次调用 / 心跳 / kill / 回收 / fallback / 校验结果必走一次审计事件落盘 |
| **IC-L2-09** | L2-04 → 后台监控 | 内部 | 子 Agent 心跳采集契约（由 L2-04 维持的 per-session 监控关系）|
| **IC-L2-10** | L2-04 → 调用方 | 外部回传 | 子 Agent 最终报告回传 (仅经 L2-05 校验后) · 禁止旁路直接回传 |

**契约设计原则**：
- L2-01 是所有读路径的"单一数据源"，三处 L2（02/04/05）都必须经它查；禁止任何 L2 内部硬编码 skill 名或子 Agent 名。
- L2-02 与 L2-03 / L2-04 之间的"链前进"是受控副作用，必须走 IC-L2-03 / IC-L2-04 这两条显式契约，禁止隐式重入。
- L2-05 作为"回传校验守门"，是 L2-03 和 L2-04 的共同下游；不允许绕过它直接回传调用方（凡绕过 = 违反 scope §5.5.6 必须 6）。

## 8. L2-01 · Skill 注册表（能力抽象层与可用性账本）详细定义

- BF-X-07 能力抽象层调度流（PM-09 的物理实现载体）
- PM-09 能力抽象层调度 "主 Agent 绑能力点不绑 skill 名"

**下游服务**：
- L2-02（选择器的候选池来源）

### 8.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-09** 能力抽象层调度（本 L2 的核心业务模式承载体）
- **PM-10** 事件总线单一事实源（账本变更必须走 IC-09）

**硬约束清单（文字描述）**：
1. **每能力点 ≥ 2 个候选 skill**（PM-09 必须义务 1；注册表不满足即启动告警）。
2. **skill 名不向外暴露为硬编码**：所有对外接口只暴露"能力点"概念；skill 名仅在注册表内部存在。
3. **账本更新原子化**：任何一条失败记忆 / 成功率的回写必须串行化（借 L1-09 锁），防并发导致账本不一致。
4. **生态变更必须记事件**：任何新增 / 弃用 / 版本升级都走 IC-09，供审计溯源。
5. **子 Agent 注册条目必须含 "返回 schema 指针"**：缺则禁止注册（给 L2-05 校验打基础）。
6. **候选排序不在本 L2 做**：本 L2 只返回列表，排序由 L2-02 按策略完成（边界自律）。

**性能约束（文字描述）**：
- 能力点查询 P99 延迟：毫秒级（是调度热路径前置动作，不能成瓶颈）。

## 9. L2-02 · Skill 意图选择器（能力点 → skill 候选链）详细定义

### 9.1 职责 + 锚定

**一句话职责**：接收调用方以"能力点 + 调用约束（成本上限 / 超时 / 质量偏好）"表达的抽象请求 —— 从 L2-01 取该能力点的候选池 → 按**可用性过滤、成本约束、历史成功率、失败记忆、调用方偏好**的多维策略排序 → 产出可供 L2-03 / L2-04 直接消费的"**首选 + fallback 链**"；调用结束后把成功 / 失败结果回写 L2-01 账本，形成"调用—学习—下次更聪明"的正反馈闭环。本 L2 是"**抽象→物理**"翻译层的核心策略点，也是 PM-09 "每能力 ≥ 2 备选 fallback" 的实际执行者。

**上游锚定**：
- Goal §2 调度器定位 + §7.4 多 skill 可替代
- Goal §4.1 决策可追溯 → 每次选择必须可解释（为什么选这个）
- scope §5.5.1 "Skill 匹配 + 优先级排序（可用性 / 成本 / 历史成功率）"
- scope §5.5.1 "fallback 链管理（每能力点 ≥ 2 备选）"
- scope §5.5.6 必须义务 2 "调用失败时执行 fallback 链"
- BF-L3-02 "查能力抽象层 → 匹配可用 skill → 优先级排序"
- BF-X-07 能力抽象层调度流
- PM-09 能力抽象层调度

**下游服务**：
- L2-03（消费 skill 同步调用链）
- L2-04（消费子 Agent 委托链）
- L2-01（回写账本）

---

### 9.2 输入 / 输出（文字级）

**输入**：
- 来自 L1-05 入口的"能力点请求"：含 capability（能力点名）+ 调用约束（max_cost / max_timeout / preferred_quality / exclude_set 等）+ 调用方 L1 标识
- 来自 L2-03 的"fallback 前进"请求：在 skill 失败后请求链上下一项
- 来自 L2-04 的"子 Agent fallback 前进"请求：同上但针对子 Agent
- 来自 L2-05 的"回传校验失败"反馈：该候选本次视为失败，链前进
- 来自 L2-01 的候选元数据查询返回（IC-L2-01）

**输出**：
- 输出给 L2-03 / L2-04 的"能力链"：含首选 + 有序 fallback 列表；每项标注来源（skill / 子 Agent / 内建逻辑）+ 预期 attempt 编号
- 输出给 L2-01 的账本回写请求（IC-L2-07）：含 capability / skill_id / 结果 / 耗时 / 失败原因（若失败）
- "capability_resolved" 事件（含选择理由摘要，供 L1-09 审计 + L1-07 观察）
- "capability_exhausted" 事件（链耗尽仍失败时）

---

### 9.3 边界

**In-scope**：
1. 接收能力点请求（不限调用方 L1）
2. 按约束条件（成本 / 超时 / 质量偏好）过滤候选
3. 按可用性 status 剔除"当前不可用"候选
4. 按多信号策略（成功率 / 失败记忆 / 最后使用时间 / 调用方偏好）排序
5. 产出"首选 + fallback 链"结构（链末端必有"内建逻辑"兜底）
6. 响应 L2-03 / L2-04 的 "fallback 前进" 请求
7. 接收"回传校验失败"作为降权信号
8. 调用结束后触发账本回写（IC-L2-07）

**Out-of-scope**：
- 不做实际调用（那是 L2-03 / L2-04 的职责）
- 不做候选池数据的维护（那是 L2-01 的职责）
- 不做回传 schema 校验（那是 L2-05 的职责）
- 不做子 Agent 的进程启停（那是 L2-04 的职责）
- 不做排序策略的 A/B 实验框架（如需可由 3-1 在策略层引入）

**边界规则**：本 L2 是"策略顾问"，只回答"先试谁 / 失败后试谁 / 最后兜底是什么"；具体"怎么调"完全留给 L2-03 / L2-04。

---

### 9.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-09** 能力抽象层调度（本 L2 是 PM-09 "每能力 ≥ 2 备选 fallback" 的执行点）
- **PM-10** 事件总线（capability_resolved / exhausted 事件必走 IC-09）
- **PM-11** 5 纪律（选择过程必有"为什么"的自然语言理由，支撑质量纪律）

**硬约束清单（文字描述）**：
1. **fallback 链末端必有"内建逻辑"兜底**：若纯 skill 候选耗尽，必须回退到本 L1 内建的极简实现（可能能力弱，但保证流程不硬断）；内建仍失败才允许抛 capability_exhausted。
2. **选择理由必须可解释**：每次"capability_resolved" 事件必须携带选择摘要（哪些候选被剔除 / 为什么选首选 / 链的具体顺序）。
3. **失败记忆必须用于降权**：最近连续失败 ≥ 2 次的候选必须明显降权（不能与正常候选同等对待）。
4. **不可用候选必须剔除**：L2-01 账本中 status=unavailable 的候选必须直接从链中移除（不仅是降权）。
5. **链长 ≥ 2**：任何输出的能力链至少含 2 项（首选 + 至少 1 个 fallback），不足则告警。
6. **账本回写不可缺席**：调用结束后必须在有限时间内回写（不允许静默丢失）。
7. **边界越界禁止**：本 L2 不得直接调用 skill / 启动子 Agent / 读 schema。

**性能约束（文字描述）**：
- 能力链产出 P99 延迟：毫秒级（决策热路径）。
- fallback 前进响应：毫秒级（不能成为降级路径的新瓶颈）。
- 策略排序不做重度计算：3-1 技术方案层若引入 ML 打分需显式标注非阻塞路径。

## 10. L2-03 · Skill 调用执行器（调用 · 签名登记 · fallback 驱动）详细定义

### 10.1 职责 + 锚定

**一句话职责**：把 L2-02 输出的"首选 + fallback 链"变成**真实的副作用动作** —— 按链首项发起 skill 或原子工具调用，每次调用产出完整的**签名**（capability / skill_id / version / params_hash / duration / attempt / caller_l1 / result_summary / validate_status）并走 IC-09 落盘；调用失败（返回错 / 超时 / 异常）或回传 schema 校验失败时沿链**前进**（IC-L2-03）到下一候选继续尝试，直到成功、链耗尽、或命中兜底内建逻辑；原子工具（Read/Write/Edit/Bash/Grep/Glob/WebSearch/WebFetch/Playwright/MCP 等）也统一走本 L2 的签名登记通道，保证"任何副作用均可审计"。

**上游锚定**：
- Goal §4.1 "决策可追溯率 100%" → 签名登记的根本理由
- scope §5.5.1 "原子工具柜（10 类）调度"
- scope §5.5.4 硬约束 1 "工具调用必须留签名登记"
- scope §5.5.4 硬约束 3 "skill 调用失败走 BF-E-05 降级链，不得裸报错退出"
- scope §5.5.6 必须义务 3 "每次调用落事件"
- BF-L3-02 调用段（调用 → 接收结构化回传 → 写回 task-board）
- BF-L3-04 工具调用流（工具柜 → 调用 → 签名 → 返回）
- BF-E-05 Skill 失败降级流
- PM-08 可审计全链追溯

**下游服务**：
- 调用方（L1-01 / L1-02 / L1-04 / L1-08 等）的最终结果接收
- L2-05（所有回传先过校验）

---

### 10.2 输入 / 输出（文字级）

**输入**：
- 来自 L2-02 的"能力链"（首选 + 有序 fallback 列表 + 内建兜底指针）
- 来自 L2-02 的"IC-L2-03 前进后的下一候选"（fallback 过程）
- 调用方的原始请求参数（从 L1-05 入口透传）
- 来自 L2-05 的"校验结果"（ok / format_invalid / not_validated）
- 原子工具的实际返回（Read/Grep/Bash 等 I/O 结果）

**输出**：
- 对 L2-05 发起"回传校验"请求（把 skill 返回值交给回收器）
- 对 L2-02 发起"fallback 前进"请求（失败时）
- 对调用方返回最终结构化结果（成功时 · 经 L2-05 校验后）
- "capability_exhausted" 异常（链耗尽仍失败时，交调用方硬暂停）
- 每次调用的"调用签名事件"走 IC-09
- 每次 fallback 的"skill_call_failed(skill_id, reason, attempt)" 事件

---

### 10.3 边界

**In-scope**：
1. 同步 / 短时 skill 调用（LLM 驱动的结构化返回 skill）
2. 原子工具调用（10 类工具柜全量）
3. 调用前的签名准备（params_hash / started_at / attempt）
4. 调用后的签名完结（ended_at / duration / result_summary）
5. 事件落盘（IC-09）
6. 失败沿链前进（IC-L2-03 → L2-02）
7. 链末端内建兜底逻辑的调用（兜底本身的"内建实现"由 3-1 定，本 L2 只管调用流）
8. 把所有返回交 L2-05 校验

**Out-of-scope**：
- 不做"子 Agent"的启动 / 监控 / kill（那是 L2-04）—— 哪怕子 Agent 是 "LLM 驱动"的同类思路，也因其"独立 session + 生命周期" 复杂度归 L2-04
- 不做"选择策略"（那是 L2-02）
- 不做"回传 schema 定义"（schema 字段级细节在 3-1）
- 不做"schema 校验"（那是 L2-05）
- 不做 KB 读写（那是 L1-06）
- 不做事件落盘物理层（那是 L1-09 / L2-05 of L1-09）
- 不做 task-board 的直接写入（只通过 IC-09 触发）

**边界规则**：本 L2 是"执行臂"，只关心"怎么把链变成真实调用 + 怎么记下签名 + 怎么在失败时前进"；所有"策略 / 校验 / 持久化"都委托上/下游。

---

### 10.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-08** 可审计全链追溯（签名登记的业务模式根据）
- **PM-09** 能力抽象层（执行链来自 L2-02，本 L2 不绕）
- **PM-10** 事件总线单一事实源

**硬约束清单（文字描述）**：
1. **每次调用必留签名**：调用发起前必生成"调用签名" + 调用完成后必补齐结果字段 + 必作为一条事件走 IC-09。
2. **失败禁止裸报错退出**：任何 skill 失败必触发 IC-L2-03 前进（除非链已耗尽且内建兜底也失败，此时才允许抛 "capability_exhausted"）。
3. **签名字段完整性不可妥协**：关键字段（capability / skill_id / version / params_hash / attempt）缺任一即违反 Goal §4.1；缺字段时事件 append 必须拒绝。
4. **所有返回必经 L2-05 校验**：不允许"我认为这 skill 返回简单就直接返回给调用方"旁路。
5. **原子工具调用也必须签名**：包括最小的 Read/Grep 等（scope §5.5.5 禁止 3 "工具调用不留签名"）。
6. **params_hash 不记原值**：只记 hash（保护可能的敏感参数），但必须同一入参在同 session 产同 hash（可用于去重 + 审计链）。
7. **attempt 顺序单调递增**：同一 capability 请求内，fallback 前进 attempt 必须 +1（供审计分析 fallback 深度）。

**性能约束（文字描述）**：
- 同步 skill 调用：端到端延迟受底层 skill 本身决定，本 L2 附加开销须 < 100ms（签名准备 + 落盘）。
- 原子工具调用附加开销：毫秒级（签名登记不应成为工具调用的瓶颈）。
- fallback 前进转场：亚秒级（前进到下一候选不能带来长停顿）。
- 高并发：多个调用方（主 loop + Quality Loop + Supervisor 观察触发）可能同时调用；应支持并发但每次调用的签名完整性独立。

### 10.5 🚫 禁止行为（明确清单）

- 🚫 **禁止 skill 失败后裸报错退出**（违反硬约束 2 + scope §5.5.5 禁止 4）
- 🚫 **禁止任何调用（含原子工具）不留签名**（scope §5.5.5 禁止 3）
- 🚫 **禁止把 skill 返回直接返给调用方**（必须经 L2-05 校验）
- 🚫 **禁止在本 L2 内部硬编码 skill 名**（必须来自 L2-02 产出的链）
- 🚫 **禁止 params 原值落盘**（只落 hash，保护敏感参数）
- 🚫 **禁止调用 subagent**（越界进 L2-04）
- 🚫 **禁止事件字段不完整时落盘**（宁可抛错也不允许"半签名"）
- 🚫 **禁止并发调用时共享签名临时状态**（每次调用独立签名生命周期）

## 11. L2-04 · 子 Agent 委托器（独立 session 生命周期管理）详细定义

### 11.1 职责 + 锚定

**一句话职责**：HarnessFlow "**让子 Agent 独立 session 跑**"的唯一落地载体 —— 接收 IC-05 / IC-12 / IC-20 三类子 Agent 委托请求，从 L2-01 查子 Agent 元数据（默认工具白名单 / 超时 / schema 指针 / 降级策略指针），**裁剪只读 context 副本** + 附**明确 goal** + 附**工具白名单** + 附**返回 schema 指针**，**启动独立 session**（具体进程模型由 3-1 技术方案定），在启动后维持**心跳监控 / 超时检测 / kill + 回收**的完整生命周期；子 Agent 失败走 **BF-E-09** 降级（新 session 重试 1 次 → 简化版 → 跳过 → 硬暂停）；子 Agent 回传**只经 L2-05 校验后**通过 IC-L2-10 回传调用方，**禁止反向写主 session 的 task-board**。本 L2 是 PM-03 "子 Agent 独立 session 委托"的实际执行点。

**上游锚定**：
- scope §5.5.1 "委托独立 session 的子 Agent（verifier / retro-generator / failure-archive-writer / architecture-reviewer / security-reviewer / codebase-onboarding 等）"
- scope §5.5.1 "子 Agent 生命周期管理（启动 / 监控 / 超时 / 回收）"
- scope §5.5.4 PM-03 "只读 context 副本 + 结构化回传，禁 session 间共享状态"
- scope §5.5.4 硬约束 2 "子 Agent 失败 2 次自动降级到 fallback"
- scope §5.5.4 硬约束 4 "子 Agent 超时（默认 5 分钟）必须 kill + 回收"
- scope §5.5.5 禁止 2 "禁止子 Agent 共享主 session 上下文"
- scope §5.5.5 禁止 5 "禁止子 Agent 超时不 kill"
- scope §5.5.5 禁止 6 "禁止子 Agent 反向写主 task-board"
- scope §5.5.6 必须义务 4 "必须给子 Agent 包 context 副本 + 明确 goal + 工具白名单"
- scope §5.5.6 必须义务 5 "必须在子 Agent 超时后 kill + 回收资源 + 走 BF-E-09"
- BF-L3-03 子 Agent 委托流
- BF-E-09 子 Agent 失败流
- BF-S5-01 Verifier 独立调用流（典型触发路径）
- PM-03 子 Agent 独立 session 委托

**下游服务**：
- 调用方（L1-04 for verifier；L1-08 for codebase-onboarding；L1-01 for 通用委托）
- L2-05（报告回传时的校验）

### 11.2 输入 / 输出（文字级）

**输入**：
- 来自 L1-05 入口分派的 IC-05 / IC-12 / IC-20 请求（含子 Agent 名 / 调用方 L1 / 原始 context 引用 / goal 描述 / 可选工具白名单覆盖 / 可选超时覆盖）
- 来自 L2-01 的子 Agent 元数据（默认工具白名单 / 默认超时 / schema 指针 / 降级策略指针）
- 来自 L2-02 的"子 Agent 候选链"（主选 + 若干备选或简化版）
- 来自运行中子 Agent session 的心跳信号
- 来自子 Agent 的最终结构化报告

**输出**：
- 启动独立 session（进程 / 子对话 / 隔离上下文；具体形式 3-1 定）
- 事件序列：subagent_started / subagent_heartbeat / subagent_completed / subagent_killed / subagent_degraded / subagent_capability_exhausted
- 对 L2-05 发起"回传校验"请求（IC-L2-05）
- 通过 IC-L2-10 把校验通过的报告转发调用方
- 降级路径决策（重试 / 简化版 / 跳过 / 硬暂停）
- 回收资源动作（临时目录 / 占用锁）

### 11.3 边界

**In-scope**：
1. 接收 IC-05 / IC-12 / IC-20 三类委托
2. 查询 L2-01 子 Agent 元数据
3. 从 L2-02 拿子 Agent 链
4. 裁剪只读 context 副本
5. 组装启动包（goal + 工具白名单 + schema 指针 + 超时）
6. 启动独立 session（与 3-1 定的进程模型对接）
7. 心跳监控 + 超时检测 + kill（先 TERM 后 KILL）
8. 资源回收（临时目录 / 锁 / 内存）
9. BF-E-09 降级链执行
10. 报告回传必经 L2-05 校验
11. 记录完整生命周期事件

**Out-of-scope**：
- 不做子 Agent 的内部业务逻辑（各子 Agent 自己的 SKILL.md 定义业务 —— scope §5.5.3 Out-of-scope 2）
- 不做同步 skill 调用（那是 L2-03）
- 不做回传 schema 的字段级定义（3-1 定）
- 不做回传校验本身（那是 L2-05）
- 不做跨子 Agent 的协同（子 Agent 之间通信由主 loop 和它们各自的独立 session 返回来回）
- 不做事件总线物理落盘（那是 L1-09）

**边界规则**：本 L2 是"带娃老师" —— 把孩子（子 Agent）送进独立教室（session）+ 定下作业（goal）+ 限定文具（工具白名单）+ 盯上课时间（心跳 / 超时）+ 放学回收（kill / 资源回收）；**不教课内容**（子 Agent 自己的业务逻辑）。

---

### 11.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-03** 子 Agent 独立 session 委托（本 L2 核心模式）
- **PM-09** 能力抽象层（子 Agent 也是能力点，绑能力点不绑名）
- **PM-10** 事件总线单一事实源

**硬约束清单（文字描述）**：
1. **只读 context 副本 + 结构化回传**：禁止子 Agent session 内的写操作传导回主 session（PM-03）。
2. **同名子 Agent 连续 2 次失败自动降级**：含"新 session 重试 1 次"作为第 2 次；超过即标 degraded 走简化版。
3. **超时必 kill + 回收**：默认 5 分钟，具体由 L2-01 元数据可覆盖；先 SIGTERM（宽限期由 3-1 定）再 SIGKILL；资源必回收。
4. **禁反向写主 task-board**：子 Agent 只能通过结构化报告回传，禁直接改 task-board（scope §5.5.5 禁止 6）。
5. **启动包必含完整 goal + 工具白名单 + schema 指针**：缺任一即启动失败（scope §5.5.6 必须 4）。
6. **工具白名单严格生效**：子 Agent 只能调被允许的工具；额外工具调用由 3-1 的隔离机制拦截（本 L2 不实现拦截但必须指定白名单）。
7. **降级不能无限链**：降级层级 ≤ 3（重试 1 次 → 简化版 → 跳过），超过即抛 capability_exhausted（不允许"简化版的简化版的……"）。
8. **多子 Agent 并发受限**：同时活跃子 Agent 数有上限（默认由 3-1 定，典型 ≤ 2），防资源爆炸；超限时排队或拒绝。

**性能约束（文字描述）**：
- 启动子 Agent 到首次心跳：秒级（具体 SLA 由 3-1 定）。
- 心跳采集周期：分钟级（防过于频繁浪费 token）。
- 超时 kill 动作：5 秒内完成（SIGTERM + 必要时 SIGKILL）。
- 报告回传通道：从子 Agent 完成到调用方收到应 < 1 秒（不含校验本身）。
- 资源回收：kill 后 30 秒内释放临时目录与锁（防残留）。

## 12. L2-05 · 异步结果回收器（schema 校验 · 超时 · 结果回传）详细定义

### 12.1 职责 + 锚定

**一句话职责**：L1-05 "正确性守门的最后一关" —— 所有异步 / 结构化回传（L2-03 的 skill 返回、L2-04 的子 Agent 报告、长时原子工具的结果）**必须**经过本 L2 按能力点对应的**回传 schema**做校验：字段缺失 / 类型错 / 必须字段为空 等任一情况都**算失败**，校验失败时通知原始路径（L2-03 / L2-04）沿 fallback 链前进；校验通过才把结构化结果**转发给原调用方**。本 L2 同时承担：把长时等待里的"回传超时"识别出来算作失败信号 + 把每次校验结果走 IC-09 供审计。它是 scope §5.5.6 义务 6 "对子 Agent 回传做 schema validate，格式错算失败" 的跨路径统一实现点。

**上游锚定**：
- Goal §4.1 "决策可追溯率 100%" → 回传校验失败也是决策数据
- scope §5.5.6 必须义务 6 "必须对子 Agent 回传做 schema validate（格式错算失败）"
- scope §5.5.1 "失败降级策略执行"（校验失败驱动的 fallback）
- scope §5.5.2 输出 "子 Agent 独立 session 的结构化报告"（结构化正确性要验证）
- BF-L3-02 接收结构化回传段
- BF-L3-03 子 Agent 回传段
- BF-E-05 / BF-E-09（本 L2 的失败判定触发降级）
- PM-08 可审计全链追溯（校验事件审计）
- PM-10 事件总线单一事实源

**下游服务**：
- 原调用方（L1-01 / L1-02 / L1-04 / L1-08 等）
- L2-02（间接：校验失败作为"降权信号"）
- L2-03 / L2-04（校验失败驱动它们的 fallback 前进）

---

### 12.2 输入 / 输出（文字级）

**输入**：
- 来自 L2-03 的"skill 回传"（IC-L2-05，含 capability / skill_id / 原始返回）
- 来自 L2-04 的"子 Agent 报告"（IC-L2-05，含子 Agent 名 / 原始结构化报告）
- 来自 L2-01 的"回传 schema 指针"（IC-L2-06）
- 来自长时调用的超时信号（自行维持的回传超时监控）

**输出**：
- 对 L2-03 / L2-04 返回校验结果（ok / format_invalid / timeout / schema_unavailable）
- 对原调用方转发校验通过的结构化结果（校验失败时**不转发**，由 L2-03 / L2-04 走 fallback）
- 每次校验结果走 IC-09（含 capability / validate_status / failed_field 摘要）
- 反馈给 L2-02 的"校验失败 → 作为降权信号"（经 L2-03 / L2-04 的 fallback 前进间接传导）

---

### 12.3 边界

**In-scope**：
1. 按 capability 查对应回传 schema 指针（IC-L2-06）
2. 按 schema 校验返回值（字段存在 / 类型 / 非空约束）
3. 根据校验结果决定 "转发 or 触发 fallback"
4. 回传超时识别（等待返回超过能力元数据的 timeout 即视为失败）
5. 校验事件落盘
6. 校验失败的结构化错误摘要（缺哪个字段 / 哪个类型错）

**Out-of-scope**：
- 不做 schema 字段级定义（3-1 定；本 L2 只用指针）
- 不做返回值的语义正确性判断（例 verifier 报告里 verdict=PASS 但三段证据链为空，那是 L1-04 的三段证据链校验；本 L2 只管格式）
- 不做调用本身（L2-03 / L2-04）
- 不做 fallback 链排序（L2-02）
- 不做子 Agent 的生命周期（L2-04）
- 不做 skill 原生接口层的重试（那是 L2-03）

**边界规则**：本 L2 是"格式警察"—— 只管"回传的形状对不对"，不管"回传的内容对不对"。

---

### 12.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-08** 可审计全链追溯（校验事件必审计）
- **PM-10** 事件总线单一事实源
- **PM-09** 能力抽象层（schema 通过 capability 间接关联，不直接耦合 skill 名）

**硬约束清单（文字描述）**：
1. **所有异步回传必经本 L2**：L2-03 / L2-04 产生的返回都要走 IC-L2-05，禁绕过。
2. **校验失败 = 视为调用失败**：触发 L2-03 / L2-04 各自的 fallback 前进（不能"校验失败但结果依然回传"）。
3. **校验事件必审计**：ok / format_invalid / timeout / schema_unavailable 任一情况都落 IC-09。
4. **schema 缺失处理明确**：若 capability 在 L2-01 找不到 schema 指针（数据异常），视为 schema_unavailable → 该次视为失败 → 走 fallback（不允许"没 schema 就放行"）。
5. **回传超时独立检测**：不依赖调用侧的 timeout，本 L2 维持自己的异步超时表（防 L2-03 / L2-04 漏检测）。
6. **结构化错误必具体**：校验失败事件必须附"缺哪个字段 / 类型错在哪 / 必须字段为空在哪"，不允许只报 "format_invalid" 不解释。
7. **校验过程不改返回内容**：只做读校验，不做"补字段 / 填默认值 / 自动转换类型"（防静默修复掩盖真 bug）。

**性能约束（文字描述）**：
- 校验开销：毫秒级（典型 schema 轻量）。
- 大结构化返回（如 codebase-onboarding 的全项目扫描结果）：亚秒级内完成校验，避免成为子 Agent 回传通路的新瓶颈。
- 高并发：多条回传同时进入时不应互相阻塞。
- 超时监控：分钟级精度即可（与心跳周期对齐）。

### 12.5 🚫 禁止行为（明确清单）

- 🚫 **禁止返回值旁路本 L2 直接回传调用方**（scope §5.5.6 义务 6 的硬性对立面）
- 🚫 **禁止校验失败仍转发结果给调用方**（违反硬约束 2）
- 🚫 **禁止校验过程修改返回内容**（静默修复 = 掩盖 bug）
- 🚫 **禁止 schema 找不到时"放行"**（必须视为失败）
- 🚫 **禁止校验事件不落盘**（违反 PM-08）
- 🚫 **禁止结构化错误只报"format_invalid"无细节**（违反硬约束 6）
- 🚫 **禁止在本 L2 做 schema 定义 / 字段语义**（越界进 3-1）
- 🚫 **禁止对 skill 结果做业务语义判断**（越界进 L1-04 等调用方）

| **子 Agent** | 独立 session 执行的 AI 代理（verifier / retro-generator / failure-archive-writer / architecture-reviewer / security-reviewer / codebase-onboarding 等）|
| **context 副本** | L2-04 裁剪给子 Agent 的**只读**上下文子集（PM-03 核心）|
| **独立 session** | 子 Agent 运行在与主 session 完全隔离的进程 / 对话 / 上下文环境中（具体形式由 3-1 定）|
| **心跳（heartbeat）** | 子 Agent session 定期上报的存活与进度信号 |
| **kill（先 TERM 后 KILL）** | 超时或需要终止时的两阶段信号：先优雅 SIGTERM，宽限后仍未退再 SIGKILL |
| **回传 schema** | 某能力点的结构化返回契约定义（字段 / 类型 / 必填性）；具体字段在 3-1 定，本 L1 只用 schema 指针 |
| **schema_unavailable** | capability 在 L2-01 找不到 schema 指针时的校验结果（视为失败）|
| **capability_exhausted** | 某能力点的链（含内建兜底）全部耗尽仍失败 · 调用方需决定硬暂停 |
| **subagent_capability_exhausted** | 子 Agent 的 BF-E-09 降级层级耗尽仍失败的终态 |
| **失败记忆** | L2-01 账本中记录某候选近期连续失败次数 + 最后失败时间 · 供 L2-02 降权 |
| **历史成功率** | 某候选近 N 次调用的成功率（滑动窗口）· 供 L2-02 排序 |
| **工具白名单** | 子 Agent session 被允许调用的工具集合（scope §5.5.6 义务 4）|
| **工具柜** | 原子工具的集合：Read / Write / Edit / Bash / Grep / Glob / WebSearch / WebFetch / Playwright / MCP |
| **降级链层级** | 子 Agent 的 3 级降级：重试 1 次 → 简化版 → 跳过（BF-E-09）|
| **生态变更** | 外部 skill 库的新增 / 弃用 / 升级事件 · L2-01 负责吸收 |
| **capability_resolved** | L2-02 成功产出链的事件类型 |
| **validate_status** | 回传校验的三态：ok / format_invalid / timeout / schema_unavailable |

**BF-L3-02 Skill 调度流在 L2 的落位**：
1. 查能力抽象层 → L2-02 发 IC-L2-01 给 L2-01
2. 匹配可用 skill → L2-01 返回候选 + L2-02 按 available 过滤
3. 优先级排序 → L2-02 多信号排序
4. 调用 → L2-03 按链首执行 + 签名
5. 接收结构化回传 → L2-05 校验
6. 写回 task-board → 调用方走 IC-09 自己写（本 L1 不直接改 task-board）
7. 失败分支 → L2-03 IC-L2-03 前进 / 走 BF-E-05

**BF-L3-03 子 Agent 委托流在 L2 的落位**：
1. 包 context 副本 → L2-04 裁剪只读副本
2. 新 session 启动子 Agent → L2-04 启动 + 事件 subagent_started
3. 子 Agent 独立决策心跳 → L2-04 监控心跳（本 L2 不管子 Agent 内部决策逻辑）
4. 返回结果 → L2-04 收报告 → L2-05 校验 → IC-L2-10 回传
5. 主 Agent 整合 → 调用方收到校验通过的报告（本 L2 不整合）
6. 失败分支（crash / 超时）→ L2-04 kill + BF-E-09 降级

**BF-L3-04 工具调用流在 L2 的落位**：
1. 从工具柜取 → L2-03 查 L2-01 工具柜元数据
2. 调用 → L2-03 实际调用
3. 记录签名 → L2-03 签名登记 → IC-09
4. 返回 → L2-05 轻量校验 → 转发调用方
5. 失败分支（重试 3 次）→ L2-03 内部重试（具体次数 3-1 定），仍失败走 IC-L2-03 / IC-L2-04 视为本能力点失败

**BF-X-07 能力抽象层调度流在 L2 的落位**：
- 定义层面：L2-01 数据底座（能力点 → 候选映射）
- 执行层面：L2-02 按请求实时查 + 排序 + 产出链
- 学习层面：L2-02 调用结束后通过 IC-L2-07 把结果回写 L2-01，形成"调用—学习—下次更聪明"闭环

**BF-E-05 Skill 失败降级流在 L2 的落位**：
1. 能力抽象层查 fallback 列表 → L2-02 输出链
2. 尝试第 2 优先 skill → L2-03 IC-L2-03 前进到备选 1
3. 仍失败 → 尝试第 3 → 备选 2
4. 全失败 → 降级到内建逻辑 + 告警 → 链末端内建兜底
5. 内建也失败 → 硬暂停 + 上报 → capability_exhausted → 调用方决定硬暂停

**BF-E-09 子 Agent 失败流在 L2 的落位**：
1. 重试 1 次（新 session）→ L2-04 第 1 次降级
2. 仍失败则降级（简化版 / 跳过 / 归档 TODO）→ L2-04 第 2 层降级（按子 Agent 元数据）
3. 降级失败则硬暂停 → subagent_capability_exhausted

**BF-S4-02 / BF-S5-01 的调用痕迹**：
- BF-S4-02 WP 内 mini-PMP 流：S4 阶段高频调 skill（tdd / prp-implement 等），全部走 IC-04 → L2-02 → L2-03
- BF-S5-01 Verifier 独立调用流：S5 阶段 L1-04 用 IC-20 → L2-02 → L2-04 的典型路径

**BF-X-04 审计追溯流的证据依赖**：
- L1-09 的 IC-18 query_audit_trail 依赖本 L1 各 L2 产生的签名事件作为反查基础；缺签名 = 审计链断 = 违反 Goal §4.1
