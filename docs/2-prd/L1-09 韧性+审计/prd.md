**关键规则**：

- **L2-01 是唯一写入点**：全 L1 只通过它改事件总线；L2-01 只接受已取锁（L2-02）+ 已校验（L2-05）上下文的调用
- **L2-02 是"串行化"的唯一手段**：不走 L2-02 直接并发写 = 破坏单一事实源
- **L2-04 是"复活"的唯一路径**：重启一律走它恢复 task-board
- **L2-03 是"为什么"的唯一答案**：任何"这行代码为什么在？"的问题终点
- **L2-05 是横切下垫层**：L2-01 / L2-04 所有写都经过它，它不直接面向外部 L1

```
 ║   → L1-01 调 L2-03 查历史压缩过的事件列表                          ║

 ║   → L1-01 决定压缩范围 → 写 `context_compressed` 事件              ║
 ║   → L2-01 落盘 → 压缩事件指向原事件 id（不删原事件）                ║
 ║   → L1-01 从 context 移除细节（仍可通过事件总线按需取回）          ║
 ║                                                                  ║
 ║ 核心保证：无论如何降级、压缩，事件总线原始事件永不删除              ║
 ╚══════════════════════════════════════════════════════════════════╝
```

## 5. L2 间业务流程（9 条）

### 流 A · 正常事件落盘一轮（最高频路径）

```
某 L1 产生 decision / event
    ↓
调 IC-09 append_event(event) → 进 L1-09
    ↓
L2-02 acquire_lock（按 event 关联的资源，默认"全局事件总线锁"）
    ├── 取锁成功 → 继续
    └── 取锁超时（≥3s）→ 返 timeout 到调用方，调用方走软红线
    ↓
L2-01 接棒：
   1. 分配下一序列号（单调递增）
   2. 计算 hash 链（基于上一事件 hash + 本事件内容）
   3. 拼 jsonl 行
    ↓
L2-05 atomic_append：
   · 写到临时位置 → fsync → rename 到 events/YYYY-MM-DD.jsonl
   · 写不成功 → 上抛 critical
    ↓
L2-01 释放 L2-02 锁
    ↓
L2-01 异步广播事件给订阅者：
   · L1-07 Supervisor（8 维度消费）
   · L1-10 UI（时间轴 push）
   · L2-03 审计镜像（建索引）
   · L2-04 恢复器（记录"最后落盘序号"用于 snapshot 触发判断）
    ↓
ack {event_id, sequence, hash} 回给调用方
```

### 流 B · 周期 checkpoint（snapshot 触发）

```
L2-04 周期定时器触发（默认每 30 秒一次；关键事件后也触发）
    ↓
L2-04 IC-L2-03 snapshot_trigger → L2-02 acquire_lock（task-board 锁）
    ↓
L2-02 授予 task-board 锁（所有 task-board 写必须先锁，保证 snapshot 一致性）
    ↓
L2-04 读当前内存态 task-board → 拼 snapshot
    ↓
L2-05 atomic_write → task-boards/<project_id>.json + checkpoints/<seq>.json
    ↓
L2-04 记录 checkpoint 元信息（序号 + 覆盖事件范围 + hash）
    ↓
L2-04 写 "checkpoint_written" 事件到总线（经 L2-01）
    ↓
L2-02 释放 task-board 锁

硬约束：snapshot 期间允许事件总线继续接写（因为总线锁是更细的"事件序号锁"，和 task-board 锁是分开的资源）
```

### 流 C · 会话退出干净 flush（BF-E-01）

```
主 loop 收 SIGINT / SIGTERM
    ↓
L1-01 调 L2-04 begin_shutdown()
    ↓
L2-04 设置 "shutting_down" 标志（L2-01 新事件 reject 或排队）
    ↓
L2-04 等在飞事件全部 ack 或达 drain 超时（默认 2s）
    ↓
L2-02 主动 release 所有未决锁（记录到 shutdown_audit）
    ↓
L2-04 触发最后一次 checkpoint（流 B）
    ↓
L2-04 写 "session_exit" 事件（含退出 state + 退出原因 + checkpoint 序号）
    ↓
L2-01 强制 flush 当日 jsonl + fsync
    ↓
L2-05 完整性自检（当日 jsonl hash 链完整 / checkpoint 完整）
    ↓
L2-04 ack 给 L1-01 "shutdown_clean"
    ↓
L1-01 返回控制给 OS → 进程退出

硬约束：整个流程 ≤ 5 秒；超时 → 强制落地（可能丢 < 1 秒数据，但绝不丢 checkpoint）
```

### 流 D · 跨 session 恢复（BF-E-02）

```
harnessFlow skill 启动
    ↓
L2-04 boot 模式：
   1. 扫 task-boards/*.json 找 status ≠ CLOSED 的项目
   2. 若无 → 进"全新项目流"（不归本 L1 管）
   3. 若有 → 进恢复路径
    ↓
L2-04 读对应项目最新 checkpoint
    ↓
L2-05 完整性校验：
   · checkpoint hash / 链尾 seq 与事件总线一致？
   · 若损坏 → 回退上一 checkpoint → 再失败 → 从事件总线完整重放
   · 事件也损坏 → 跳损坏块（L2-05 降级）+ 大告警 + 降级到最后有效 state
    ↓
L2-04 从 checkpoint 序号之后按序读事件 → 在内存重放到 task-board
    ↓
L2-04 重建内存态 task-board（命中退出前 state）
    ↓
L2-04 广播 `system_resumed` 事件（含恢复 state + 项目 id + 恢复耗时）
    ↓
全 L1 订阅者：
   · L1-02 恢复未决 Gate 卡片
   · L1-04 恢复 Quality Loop 中间态
   · L1-03 恢复 WP 执行指针
   · L1-07 恢复 8 维度 baseline
   · L1-10 UI 拉出恢复卡片 "已恢复 project X, state=Y, 继续？"
    ↓
用户确认 → L1-01 接管 → tick 恢复

硬约束：整个恢复 ≤ 30 秒；超时 → 明确失败告警 + 不做假恢复
```

### 流 E · 审计追溯查询（BF-X-04）

```
用户在 UI 发 "为什么 docs/planning/requirements.md 有这一行？"
    ↓
L1-10 → IC-18 query_audit_trail(anchor={type='file_line', path='docs/planning/requirements.md#L42'})
    ↓
L2-03 按锚点类型分派查询：
   · file_line 锚点 → 先查产出物提交事件（L1-02 L2-03 产出 REQ-007 写入）
   · artifact_id 锚点 → 直接命中事件流
   · decision_id 锚点 → 一跳到决策事件
    ↓
L2-03 组装 trail（按时间倒序）：
   · decision 层：哪次决策下的笔 → 决策理由 + 输入证据 + 5 纪律回答
   · event 层：相关事件序列（从决策前 5 条 → 决策本身 → 决策后 5 条）
   · supervisor_comment 层：该决策期间 Supervisor 的 INFO/SUGG/WARN（若有）
   · user_authz 层：是否触发过用户授权 / 变更请求（若有）
    ↓
L2-03 完整性校验：
   · 4 层至少 decision + event 必非空；其他两层可空
   · 若 decision 空 → 告警 "可追溯率破产"（Goal §4.1 红线）
    ↓
返回 trail 到 L1-10 UI → 渲染审计面板
```

### 流 F · 并发写入串行化（响应面 5）

```
三方同时触发 task-board 写：
   · 主 loop L1-01 tick → append_event type=decision
   · 子 Agent 回调 via L1-05 → append_event type=subagent_result
   · Supervisor 30s tick via L1-07 → append_event type=supervisor_event
    ↓
三方都调 IC-09 append_event → 进 L1-09
    ↓
L2-02 排队 acquire_lock（FIFO，公平）
    ↓
Caller A 取锁 → 流 A 走完 → release
Caller B 取锁 → 流 A 走完 → release
Caller C 取锁 → 流 A 走完 → release
    ↓
事件总线里三条事件按 B→A→C 或 A→C→B 等顺序严格排列（由到达先后决定）

关键保证：不允许"同一序号两条事件"（L2-02 + L2-01 联防）
降级：单请求等锁 ≥ 3 秒 → timeout → 调用方获得具名错误 → 走软红线
```

### 流 G · 事件落盘失败硬 halt（响应面 4）

```
某调用方发 append_event
    ↓
L2-02 取锁 OK → L2-01 分序号 → L2-05 尝试原子写
    ↓
L2-05 写失败（磁盘 IO error / ENOSPC / 权限）
    ↓
L2-05 本地重试 2 次（指数 backoff 100ms / 300ms）
    ↓
仍失败 → 抛 critical 回 L2-01
    ↓
L2-01 ack(status=FAIL, reason=disk_error) 给调用方
    ↓
L2-04 写 "bus_write_failed" 事件到降级备份面（内存 + 额外备份目录，尽力而为）
    ↓
L2-02 释放锁
    ↓
L2-01 通过 L1-07 常规红线路径广播 hard_halt
    ↓
L1-01 停 tick → 全系统静默
    ↓
L1-10 UI 红屏告警 "事件总线写失败，系统已停机，请诊断"
    ↓
等待用户手动介入（本 L1 不自动恢复）
```

### 流 H · Snapshot 损坏 → 事件重放恢复（流 D 分支）

```
L2-04 启动恢复 → 读 checkpoint
    ↓
L2-05 完整性校验 FAIL
    ↓
L2-04 降级尝试：
   Tier 1: 回退上一版 checkpoint（如果有）
     → L2-05 再校验 → OK → 从那点起回放事件
     → FAIL → 进 Tier 2
    ↓
   Tier 2: 完全丢弃 checkpoint，从事件总线开头完整重放
     → L2-05 扫事件 hash 链
     → 全链完整 → 完整重放成功 → ✅
     → 某处 hash 链断裂 → 进 Tier 3
    ↓
   Tier 3: 跳过损坏事件块 + 重放剩余事件
     → L2-04 记录 "reconstruction_partial" 到最终 task-board
     → 告警用户 "已部分恢复，可能缺失 X 条事件"
     → L1-10 UI 高亮警告
    ↓
   Tier 4: 事件总线也完全损坏
     → L2-04 拒绝重建空白 task-board（scope §5.9.5 禁止）
     → 抛 CRITICAL 给 L1-01 + UI 告警 "无法恢复"
     → 让用户手动选择：放弃项目 / 指向备份
```

### 流 I · 裁剪与压缩对事件总线的影响（响应面 6）

```
L1-01 识别 context > 80% 触发 BF-E-06 压缩
    ↓
L1-01 决定压缩范围 → 发 `context_compressed` 元事件（经 IC-09）
   · 该事件内容 = 被压缩事件的 id 列表 + 压缩摘要
    ↓
L2-01 正常落盘该元事件
    ↓
L2-03 审计镜像建立"压缩索引"：
   · 原事件仍在 jsonl 里（不删）
   · 压缩索引指向原事件 id
    ↓
L1-01 从 in-context 移除原事件详情
    ↓
后续如需回查：L2-03 按原事件 id 仍可返还完整内容

保证：压缩是"遗忘 in-context 细节"，不是"删盘"。审计追溯率永远 100%。
```

## 8. L2-01 · 事件总线核心（jsonl 落盘 + 订阅） 详细定义

### 8.1 职责 + 锚定

**一句话职责**：接收全 L1 的 `append_event` 请求，在锁保护下分配序列号 + 计算 hash 链，通过崩溃安全层落盘到当日 jsonl，并异步广播给订阅者 —— 是 HarnessFlow 所有状态变更的**唯一写入口** + **唯一事实源**。

**上游锚定**：
- Goal §4.1 "决策可追溯率 100%"
- PM-10 事件总线单一事实源
- scope §5.9.1 职责 "事件总线单一事实源"
- scope §5.9.4 硬约束 1 "事件只追加"、硬约束 5 "hash 链防篡改"
- BF-X-03 事件总线落盘流
- BF-X-08 持久化与跨 session 恢复流（落盘段）

**下游服务**：
- 全 L1（IC-09 的承接方；scope §8.2 IC-09）
- L2-02 锁管理器（取 / 释放锁）
- L2-05 崩溃安全（落盘原子性）
- L2-03 审计记录器（订阅消费全流）
- L2-04 恢复器（订阅消费 + 回放源）
- L1-07 Supervisor（订阅消费 8 维度观察）
- L1-10 UI（订阅消费时间轴）

---

### 8.2 输入 / 输出

**输入（文字描述，非 schema）**：

- **`append_event` 调用**：来自全 L1-01~08/10 任何一方；携带事件时间戳、类型（按 L1 前缀分配，如 `L1-01:decision` / `L1-04:verdict` / `L1-07:supervisor_event`）、actor（调用方身份）、当前 state、事件内容（业务字段，文字）、关联引用（指向其他事件 / 产出物 / 决策 id 的链接）
- **订阅注册**：来自 L2-03 / L2-04 / L1-07 / L1-10；携带订阅者身份 + 过滤规则（按 type 前缀 / actor / state）
- **退出 drain 信号**：来自 L2-04 恢复器；进入 shutdown 模式后拒绝新事件或转入排队

**输出（文字描述）**：

- **落盘 jsonl 行**：追加到 `events/YYYY-MM-DD.jsonl`；每行一条事件；含分配的序列号 + hash
- **ack 回调**：返回给调用方 `{event_id, sequence, hash}`；失败时返回错误类型（disk_error / lock_timeout / shutdown_rejected）
- **订阅者广播**：异步推送事件副本给所有已注册订阅者；不保证订阅者消费成功（fire-and-forget）
- **审计镜像 feed**：给 L2-03 提供实时流

---

### 8.3 边界

**In-scope（本 L2 做什么）**：

1. 接收 `append_event` 调用（经 IC-09）
2. 取锁（via L2-02）+ 分配单调递增序列号
3. 计算 hash 链（基于上一事件 hash + 本事件内容）—— 只描述"需要 hash 链"，具体算法在 3-1
4. 调 L2-05 原子写入 jsonl 行
5. 落盘成功 → ack 调用方 + 广播订阅者
6. 落盘失败 → 抛 critical 给恢复路径 + 触发硬 halt（响应面 4）
7. 订阅者注册 / 注销管理
8. 事件流订阅的过滤分发
9. Shutdown 模式下的 drain + reject

**Out-of-scope（本 L2 不做，谁做）**：

- ❌ 不做锁 → L2-02
- ❌ 不做原子写 → L2-05
- ❌ 不做 snapshot / checkpoint → L2-04
- ❌ 不做审计查询（只广播给 L2-03，查询在 L2-03）→ L2-03
- ❌ 不做事件内容合法性校验（type 命名空间是否属于 L1-XX 等）→ 调用方自己
- ❌ 不做订阅者消费失败的重试 → 订阅者自负
- ❌ 不做跨项目事件总线（scope §5.9.5 禁止）→ 每项目独立

**边界规则**：

- 本 L2 只做"**写入 + 广播**"，不判内容对错
- 事件**只追加**，绝不允许原地修改
- 同一序号绝不允许两条事件（强一致）

---

### 8.4 约束

**业务模式引用**：

- **PM-10 事件总线单一事实源**（核心）
- **PM-08 可审计全链追溯**（每事件都要可追）

**硬约束清单**：

1. **只追加 append-only**（任何改已落盘事件 = 违反，必 halt）
2. **序列号单调递增**（保证回放顺序性，永不重复、永不倒序）
3. **hash 链闭合**（每事件含前事件 hash，任一节断裂 → 必告警）
4. **落盘失败 = 硬 halt**（scope §5.9.4 + 响应面 4；不降级静默）
5. **每次追加必经 L2-02 取锁**（不允许无锁写入）
6. **每次追加必经 L2-05 原子写**（不允许绕过崩溃安全层）
7. **订阅者广播不阻塞主写路径**（异步，订阅者慢不影响事件落盘）
8. **Shutdown 模式下拒绝新事件**（由 L2-04 控制，保证 drain 时无新事件乱入）

**性能约束（文字描述）**：

- 单次 `append_event` 端到端延迟（从调用入口到 ack）：P95 ≤ 50ms，P99 ≤ 200ms
- 高峰吞吐：≥ 100 事件 / 秒（保证主 loop 高频 tick 不堵）
- 订阅者广播延迟：≤ 500ms（UI 感知实时）
- 当日 jsonl 文件大小超过 阈值时（由 3-1 定具体值）自动切换到下一个分片（但本 PRD 只说"必须支持切分"）

---

### 8.5 🚫 禁止行为（明确清单）

- 🚫 **禁止原地修改已落盘事件**（append-only 硬约束 1；违反 = 审计链断裂，系统必须 halt）
- 🚫 **禁止绕过 L2-02 锁直接写 jsonl**（破坏串行化，序号会乱）
- 🚫 **禁止绕过 L2-05 崩溃安全层**（破坏原子性，崩溃时数据损坏）
- 🚫 **禁止对落盘失败做"假装成功"降级**（必须 halt；不允许"先在内存留着等恢复"，那会让其他 L1 误以为事件已入账）
- 🚫 **禁止为了性能而批量写（打包多事件一次落盘）打破序号语义**（必须一条一落盘 fsync；如需批量写优化，由 3-1 给出"保持语义"的方案）
- 🚫 **禁止跨项目共享事件总线**（每项目独立 jsonl 文件；scope §5.9.5）
- 🚫 **禁止暴露"改事件"的 API**（接口面必须只有 append + read）
- 🚫 **禁止在 shutdown 模式下静默丢弃新事件**（必须明确 reject + ack 给调用方）

---

### 8.6 ✅ 必须职责（明确清单）

- ✅ **必须**对每一次 `append_event` 调用分配唯一单调递增序列号
- ✅ **必须**为每一事件计算并写入 hash（含前事件 hash）
- ✅ **必须**每次追加前取 L2-02 锁，追加后释放
- ✅ **必须**每次追加走 L2-05 原子写（临时文件 + rename + fsync 语义，具体在 3-1）
- ✅ **必须**落盘失败时抛 critical + 触发硬 halt（响应面 4）
- ✅ **必须**支持订阅者注册 / 注销 / 过滤分发
- ✅ **必须**在 shutdown 模式下干净 drain（不丢在飞事件）
- ✅ **必须**为每个 L1 的事件分配独立 type 前缀（scope §5.9.6 义务 4；前缀清单在附录 A）

---

### 8.7 🔧 可选功能职责（可做但不硬性要求）

- 🔧 **事件批量读取 API**（给订阅者的初始化 replay 用；非主路径）
- 🔧 **订阅者健康度观测**（每订阅者消费延迟统计，用于 retro）
- 🔧 **事件类型统计报表**（按 L1 前缀 / type 统计事件频次，给监控面用）
- 🔧 **jsonl 当日分片轮转**（超过大小阈值自动切新文件；默认策略由 3-1 决定）
- 🔧 **事件反向索引建立协助**（配合 L2-03 建立高频查询索引）

### 8.9 🎯 交付验证大纲（Given-When-Then）

**正向场景 1 · 单条事件正常落盘**
- Given：系统已启动，L2-01 / L2-02 / L2-05 就绪
- When：L1-01 发 `append_event(type='L1-01:decision', actor='main_loop', state='S4', content='...')`
- Then：
  - L2-02 授予锁（延迟 ≤ 10ms）
  - L2-01 分配序号 N（比上一序号大 1）
  - 落盘到当日 jsonl（新增一行）
  - hash 链闭合（hash_N 含 hash_{N-1}）
  - ack 返回 `{event_id, sequence=N, hash}`
  - L1-07 / L1-10 / L2-03 收到广播事件

**正向场景 2 · 三方并发写严格串行**
- Given：三方同时调 append_event
- When：L2-02 FIFO 发锁
- Then：
  - 三事件按取锁顺序排列
  - 序号连续无重（N, N+1, N+2）
  - hash 链连续闭合
  - 无"同一序号两条事件"

**负向场景 3 · 落盘失败触发硬 halt**
- Given：磁盘被填满（ENOSPC）
- When：L1-01 发 append_event
- Then：
  - L2-05 写失败 → 本地重试 2 次 → 仍失败
  - L2-01 ack(FAIL, reason=disk_error)
  - 系统广播 hard_halt → 主 loop 停 tick
  - UI 展示红屏告警
  - 无"假装成功"事件进账

**负向场景 4 · 订阅者慢不拖累主路径**
- Given：某订阅者故意 sleep 5s
- When：L1-01 发 10 条事件
- Then：
  - 主 append_event 路径 P95 延迟仍 ≤ 50ms
  - 10 条事件全部落盘成功
  - 慢订阅者事件积压（由订阅者自己处理；非本 L2 问题）

**集成场景 5 · 与 L1-07 Supervisor 联动**
- Given：Supervisor 30s tick 周期扫事件总线
- When：主 loop 产生 100 条事件
- Then：
  - Supervisor 每 30s 读一批事件
  - 读取不阻塞 L2-01 写入（读写隔离）
  - Supervisor 命中 8 维度异常 → 产 supervisor_event 回写（闭环）

**性能场景 6 · 高频吞吐 baseline**
- Given：空总线（无慢订阅）
- When：压测连续发 1000 条 append_event
- Then：
  - 总耗时 ≤ 10s（即 ≥ 100 事件/秒）
  - P99 延迟 ≤ 200ms
  - 序号 1000 连续无缺

## 9. L2-02 · 锁管理器 详细定义

### 9.1 职责 + 锚定

**一句话职责**：给 L1-09 内部和跨 L1 对敏感资源（事件总线写、task-board 状态切换、snapshot 窗口）的并发访问提供互斥原语（acquire/release），保证单一事实源在多方写入下仍严格串行 —— 是"写入秩序"的最后一道防线。

**上游锚定**：
- PM-10 事件总线单一事实源（防竞态破坏）
- scope §5.9 硬约束 2 "任何 L1 的状态变更必须经事件总线（禁绕过）"
- BF-X-08（并发段）
- scope §8.2 IC-07 `acquire_lock`（对外契约，由 L2-02 承担）

**下游服务**：
- L2-01（每次事件追加前取总线锁）
- L2-04（每次 snapshot 前取 task-board 锁）
- 全 L1（通过 IC-07 可对 task-board 的特定范围申请锁，用于复合操作）

---

### 9.2 输入 / 输出

**输入**：
- **`acquire_lock` 调用**：携带资源名（如 `event_bus` / `task_board` / `task_board:<project_id>` / `checkpoint_window`）+ 调用方身份 + 等锁超时
- **`release_lock` 调用**：携带原 acquire 返回的 lock token
- **shutdown 广播**（来自 L2-04）：进入 shutdown 模式时强制释放所有未决锁

**输出**：
- **lock token**（acquire 成功）或 **timeout 错误**（等锁超时）或 **dead-lock 拒绝**（识别到死锁路径时主动拒绝申请）
- **锁持有者 / 等待队列观测数据**（给 retro + 监控用）
- **释放 ack**（release 成功回执）

---

### 9.3 边界

**In-scope**：

1. 支持多种资源名的独立锁（resource-level locking）
2. FIFO 公平排队
3. 等锁超时（默认 3 秒，具体值 3-1 定）
4. 超时后返回明确错误给调用方
5. 死锁静态识别（如 A 持 X 求 Y + B 持 Y 求 X 的环形等待；具体算法 3-1）
6. shutdown 模式下主动释放所有未决锁 + 拒绝新申请
7. 锁持有观测（每锁的持有时长 / 等待队列长度）

**Out-of-scope**：

- ❌ 不做分布式锁（本地单进程）→ 部署运维
- ❌ 不做锁内容序列化（锁只是互斥原语，不管被保护的内容）→ L2-01 / L2-04
- ❌ 不做事务（跨多锁的原子提交） → 上层调用方（复合场景组合多锁）
- ❌ 不做性能级无锁结构（MVCC / 无锁队列等）→ 3-1 若性能不足再考虑
- ❌ 不做锁优先级（FIFO 唯一；Supervisor 和 user 没有 VIP 通道）→ 一视同仁

**边界规则**：

- 锁是"资源名 + 互斥"的语义原语；调用方自己决定"保护谁"
- 锁不嵌套（本 PRD 不支持重入锁；如需嵌套场景由调用方组合新资源名）
- 每个锁的持有方必须自己 release；shutdown 是兜底

---

### 9.4 约束

**业务模式引用**：

- **PM-10 事件总线单一事实源**（通过锁保证串行）
- **PM-04 WP 拓扑并行推进**（WP 并行可能触发同 task-board 写，需锁挡）

**硬约束清单**：

1. **FIFO 公平**（不允许 LIFO / 优先级插队）
2. **等锁超时 ≤ 3 秒**（超时返错，不无限等）
3. **shutdown 模式强制释放**（防止 drain 被挂死）
4. **单资源单持有者**（不支持共享读锁；本 PRD 简化，3-1 可扩读写锁）
5. **acquire 和 release 必须成对**（未成对 release = 资源永久泄漏，必须被自动识别并告警）
6. **死锁识别 → 主动拒绝**（不允许系统陷入死锁静默）

**性能约束**：

- 无竞争 acquire 延迟：P95 ≤ 5ms
- 高竞争（10 方等锁）acquire 延迟：P95 ≤ 100ms
- release 延迟：P95 ≤ 2ms
- 支持同时管理 ≥ 20 个不同资源名的锁

---

### 9.5 🚫 禁止行为（明确清单）

- 🚫 **禁止违反 FIFO**（任何形式的"关键调用方插队"都不允许；公平是正确性基石）
- 🚫 **禁止锁永不释放 + 静默**（超时未释放 = 必须告警 + 自动强制释放并记"锁泄漏"事件）
- 🚫 **禁止支持跨进程 / 跨主机锁**（本地单进程边界；避免分布式复杂度）
- 🚫 **禁止同一资源名并行授予两把锁**（写-写并发是破坏单一事实源的源头）
- 🚫 **禁止锁申请阻塞整个 L2-02 服务**（单调用方失败不能堵塞其他等待者）
- 🚫 **禁止在事件总线写路径外暴露"绕过锁"的后门**（任何"快速路径"都不允许）
- 🚫 **禁止 shutdown 期间接受新锁申请**（应直接返回 shutdown_rejected）

---

### 9.6 ✅ 必须职责（明确清单）

- ✅ **必须**为每次 acquire_lock 返回唯一 lock token + 记审计事件
- ✅ **必须**FIFO 公平排队
- ✅ **必须**强制等锁超时 ≤ 3 秒 + 超时返错
- ✅ **必须**识别环形等待（死锁）+ 主动拒绝新申请打破环
- ✅ **必须**支持 shutdown 模式 force_release_all
- ✅ **必须**提供锁持有观测数据（供 retro 分析并发瓶颈）
- ✅ **必须**对 acquire / release 不配对的调用告警（锁泄漏红线）
- ✅ **必须**在 release_lock 调用无效 token 时明确拒绝（防错误释放他人锁）

---

### 9.7 🔧 可选功能职责

- 🔧 **锁统计仪表盘数据**（每分钟竞争度、平均等锁延迟）
- 🔧 **锁调用链追踪**（记录哪个 L1 的哪次调用持有过锁，便于审计）
- 🔧 **持有超时告警**（单次持有 > N 秒 → 发 WARN 给 Supervisor）
- 🔧 **锁粒度动态建议**（检测到"粗粒度锁争用高"时建议拆细，但由 3-1 或 retro 实施）

---

### 9.8 与其他 L2 / L1 交互（IC 契约清单）

**作为调用方**：

| IC | 被调方 | 何时调 | 一句话 |
|---|---|---|---|
| —— | L2-01 审计事件 | 每次锁申请 / 释放 / 泄漏 | 走常规 append_event 写锁操作审计 |

**作为被调方**：

| IC | 调用方 | 何时被调 | 一句话 |
|---|---|---|---|
| **IC-07**（scope §8.2） | 全 L1（用于跨资源写复合场景）| L1 需要"原子写多个相关 task-board 字段" | 授予复合锁 |
| **IC-L2-01** | L2-01 事件总线 | 每次 append_event | 取 event_bus 锁 |
| **IC-L2-03**（锁部分） | L2-04 恢复器 | 每次 snapshot | 取 task_board 锁 |

### 9.9 🎯 交付验证大纲（Given-When-Then）

**正向场景 1 · 无竞争快速通过**
- Given：锁空闲
- When：单方 acquire(event_bus)
- Then：≤ 5ms 返回 lock token，记审计事件

**正向场景 2 · 10 方竞争 FIFO 顺序**
- Given：10 方几乎同时 acquire 同一资源
- When：第 1 方先到达（微秒差）
- Then：
  - 按到达顺序授予
  - 每个 release 后下一个立即拿到
  - 所有 10 方最终都成功（无饥饿）

**负向场景 3 · 等锁超时返错**
- Given：锁被某方持有 5 秒不释放
- When：另一方 acquire 带超时 3s
- Then：
  - 在 3s ± 100ms 时返回 timeout 错误
  - 锁泄漏事件记到审计（持有方被标 N 秒）
  - 调用方走软红线降级路径

**负向场景 4 · 死锁识别主动拒绝**
- Given：A 持 X 求 Y，B 持 Y 求 X（环形）
- When：B 发起 acquire Y
- Then：
  - L2-02 检测到环
  - 主动拒绝 B 的申请（返回 deadlock_rejected）
  - 记 CRITICAL 审计事件 → 上报 Supervisor

**集成场景 5 · shutdown 强制释放**
- Given：3 方持有不同锁，系统 SIGINT
- When：L2-04 广播 shutdown
- Then：
  - 3 锁全部强制释放
  - 记 force_release 审计事件
  - 后续 acquire 一律返回 shutdown_rejected

**性能场景 6 · 高并发无饥饿**
- Given：20 方持续申请同一资源
- When：跑 10 秒
- Then：
  - 每方至少获锁 1 次
  - P99 等锁延迟 ≤ 200ms

## 10. L2-03 · 审计记录器 + 追溯查询 详细定义

### 10.1 职责 + 锚定

**一句话职责**：订阅事件总线的全量流，在内存 / 索引层建立"决策 ↔ 事件 ↔ 监督点评 ↔ 用户授权"的审计镜像；接收外部锚点（代码行 / 产出物路径 / 决策 id）的审计查询请求 → 按四层反查拼装完整链返回 —— 是 Goal §4.1"决策可追溯率 100%"的**最终落脚点**。

**上游锚定**：
- Goal §4.1 "决策可追溯率 100%"
- PM-08 可审计全链追溯
- scope §5.9.1 职责 "审计追溯链"
- scope §5.9.6 义务 6 "审计查询时返回完整链（决策 + 理由 + 事件 + 监督点评 + 用户批复）"
- BF-X-04 审计追溯流
- scope §8.2 IC-18 `query_audit_trail`

**下游服务**：
- L1-10 UI（IC-18 的承接方；审计追溯查询面板 P1）
- Supervisor retro 分析面（可选读取）

---

### 10.2 输入 / 输出

**输入**：

- **事件流订阅**：来自 L2-01 的全量事件 push（通过 IC-L2-07）
- **审计查询请求**：通过 IC-18，来自 L1-10；携带锚点（文件行 / 产出物 id / 决策 id）+ 查询深度（默认完整链）
- **恢复期停写通知**：来自 L2-04（IC-L2-10）；恢复中禁接查询

**输出**：

- **审计查询结果 trail**（文字描述）：按时间序拼装的 4 层链：
  - 层 1 决策：决策 id + 时间 + 决策文本 + 理由 + 5 纪律回答 + 输入证据
  - 层 2 事件序列：决策前后各 N 条相关事件（配置范围由 3-1 定）
  - 层 3 Supervisor 点评：该决策期间的 INFO/SUGG/WARN/BLOCK（若有）
  - 层 4 用户授权：涉及的 user_intervene 事件（若有）
- **可追溯率告警**（当 trail 的决策层为空时）：上报 Goal §4.1 红线破产
- **审计镜像健康度**（观测面数据）

---

### 10.3 边界

**In-scope**：

1. 订阅事件总线全流 + 建立内存审计镜像（按事件 id / 产出物 id / 决策 id / 文件路径的反向索引）
2. 接收 IC-18 query_audit_trail 请求
3. 按锚点类型分派查询（文件行 / 产出物 / 决策 id 三种路径）
4. 拼装 4 层 trail（决策 / 事件 / 监督 / 授权）
5. 完整性校验（决策层必非空；否则告警）
6. 不完整 trail 的明确标记（层 2/3/4 可为空但必标"本层无数据"，而非"未查"）
7. 响应面 6 压缩元事件的透明跳转（压缩索引下仍能拿原事件详情）
8. 恢复期查询 gate（恢复中拒查，恢复后解禁）

**Out-of-scope**：

- ❌ 不做事件落盘（L2-01）
- ❌ 不做全文检索（grep-level 即可；scope §5.9.3 明确不做 elastic）
- ❌ 不做决策生成（L1-01）
- ❌ 不做 Supervisor 判定（L1-07）
- ❌ 不做用户授权决策（L1-10）
- ❌ 不做跨项目 trail（每项目独立）

**边界规则**：

- 本 L2 是**只读查询 + 只订阅**；除了自己的审计镜像数据结构，不写任何全局状态
- trail 的任一层"缺"都必须显式返回"缺"，不允许静默跳过
- 查询性能不能拖累事件落盘（L2-01 性能与 L2-03 完全解耦）

---

### 10.4 约束

**业务模式引用**：

## 11. L2-04 · 检查点与恢复器（system_resumed 广播） 详细定义

### 11.1 职责 + 锚定

**一句话职责**：周期性 + 关键事件后触发 task-board 的 snapshot 落盘（checkpoint）；在 SIGINT/SIGTERM 时协调干净 flush；在 Claude Code 重启后扫未 CLOSED 项目 → 读 checkpoint → 回放事件 → 重建 task-board → 广播 `system_resumed` 让全 L1 接续 —— 是"跨 session 无损恢复"的**执行者**。

**上游锚定**：
- Goal §4.1 "可跨 session 无损恢复"
- scope §5.9.4 硬约束 3 "checkpoint 频率 ≤ 1 分钟" + 硬约束 4 "重启后 30 秒内完成恢复或告警"
- scope §5.9.6 义务 3 "必须对 SIGINT/SIGTERM 做干净 flush + checkpoint + 优雅退出"
- scope §5.9.6 义务 5 "重启自动触发恢复 + 向用户明示"
- BF-X-08 持久化与跨 session 恢复流
- BF-E-01 会话退出流
- BF-E-02 跨 session 恢复流

**下游服务**：
- L2-01（订阅 + 回放）
- L2-02（snapshot 取 task-board 锁）
- L2-05（落盘 checkpoint 用原子写 + 校验完整性）
- L2-03（恢复期停查 / 解禁通知）
- 全 L1（`system_resumed` 广播）

---

### 11.2 输入 / 输出

**输入**：

- **周期触发信号**：内部定时器（默认 30 秒一次；具体值 3-1 定）
- **关键事件触发**：某类高价值事件后立即 snapshot（如 Stage Gate 通过 / state 切换 / WP 完成）
- **shutdown 信号**：SIGINT/SIGTERM 捕获，或 L1-01 主动调 `begin_shutdown()`
- **boot 启动信号**：skill 启动时自动执行
- **事件流订阅**：用于回放（读路径）

**输出**：

- **checkpoint 文件**：落盘到 `checkpoints/<seq>.json`（具体命名 3-1 定）
- **task-board 当前快照**：`task-boards/<project_id>.json`
- **`system_resumed` 事件**：恢复完成后通过 L2-01 广播给全 L1
- **恢复失败告警**：超过 30 秒硬约束或完全无法恢复时告警
- **shutdown_clean ack**：退出流程完成后给 L1-01

---

### 11.3 边界

**In-scope**：

1. 周期 checkpoint 触发
2. 关键事件 checkpoint 触发
3. SIGINT/SIGTERM 的干净 flush + 最终 checkpoint
4. 重启时扫未 CLOSED 项目（查 task-boards/ 目录）
5. 读最新 checkpoint + 校验
6. 从 checkpoint 序号开始回放事件
7. 重建内存态 task-board
8. 广播 `system_resumed` 事件
9. 恢复失败的多级降级（Tier 1~4 见流 H）
10. 恢复期限管控（30 秒硬约束）
11. 恢复期 audit gate 通知 L2-03
12. Shutdown drain 阶段（L2-01 暂停接事件 → 等在飞完成 → 最终落盘）

**Out-of-scope**：

- ❌ 不做事件落盘（L2-01）
- ❌ 不做锁（L2-02）
- ❌ 不做原子写（L2-05）
- ❌ 不做 trail 查询（L2-03）
- ❌ 不做 task-board 业务字段定义（那是各 L1 自己管）
- ❌ 不做 cloud 备份（本地文件系统；scope §5.9.3）
- ❌ 不做恢复后用户确认 UI（由 L1-10 UI 负责；本 L2 只发 resumed 事件）

**边界规则**：

- checkpoint 是 task-board 某时刻的完整快照（文件大小较大；3-1 定）
- 回放起点 = checkpoint 序号 + 1；回放终点 = 当前最大序号
- 回放期间不接新事件（L2-01 的订阅暂停）
- 恢复失败时**绝不重建空白 task-board**（scope §5.9.5 硬禁）

---

### 11.4 约束

**业务模式引用**：

- **PM-10 事件总线单一事实源**（checkpoint 不是主事实源，总线才是）
- **PM-01 methodology-paced**（恢复后无缝接 methodology 节奏）

**硬约束清单**：

1. **checkpoint 频率 ≤ 1 分钟**（scope §5.9.4 硬约束 3）
2. **重启恢复 ≤ 30 秒**（scope §5.9.4 硬约束 4；超时 → 告警不假恢复）
3. **shutdown flush ≤ 5 秒**（SIGINT 后窗口；超时强制落地）
4. **checkpoint 必须经 L2-05 原子写 + 完整性校验**（不允许损坏 checkpoint 进盘）
5. **恢复失败绝不重建空白 task-board**（scope §5.9.5 硬禁）
6. **恢复期间 L2-03 必须收到停查通知**（防脏读）
7. **system_resumed 广播前 task-board 必须已完全重建**（防全 L1 接力时状态不全）
8. **同一时刻只有一个活跃 snapshot 窗口**（防并发导致 checkpoint 不一致）

**性能约束**：

- 单次 checkpoint 落盘耗时 ≤ 2 秒（task-board 大小适中时；3-1 压测）
- 回放 1 万事件耗时 ≤ 10 秒（保证重启 30 秒硬约束）
- shutdown flush 整体耗时 ≤ 5 秒

### 11.5 🚫 禁止行为（明确清单）

- 🚫 **禁止在恢复失败时重建空白 task-board**（scope §5.9.5 硬禁 6；必须告警让用户决定）

## 12. L2-05 · 崩溃安全层（原子写 + 完整性校验） 详细定义

### 12.1 职责 + 锚定

**一句话职责**：为所有落盘操作（事件 jsonl 追加 / checkpoint 写 / task-board snapshot 写）提供**文件级原子性**（临时文件 + 原子 rename + fsync 语义）与**完整性校验**（hash 链检验 / 文件 header 校验）；在中断、断电、IO 错误下保证**永不产生坏数据**，并提供损坏检测 + 分级降级路径。

**上游锚定**：
- scope §5.9.4 硬约束 1 "append-only" + 硬约束 5 "hash 链防篡改"
- scope §5.9.6 义务 1 "每次事件追加后 fsync"
- BF-E-01 会话退出流（退出保完整）
- BF-E-02 跨 session 恢复流（损坏检测 + 降级）

**下游服务**：
- L2-01（事件追加的原子写）
- L2-04（checkpoint / task-board snapshot 的原子写 + 恢复时的完整性校验）
- L2-03 审计记录器（重建审计镜像时可能回查损坏范围）

---

### 12.2 输入 / 输出

**输入**：

- **`atomic_append` 调用**（来自 L2-01）：携带目标 jsonl 路径 + 一行内容 + hash
- **`atomic_write_snapshot` 调用**（来自 L2-04）：携带目标路径 + snapshot 内容 + 元信息
- **`verify_integrity` 调用**（来自 L2-04）：携带目标文件路径 + 校验方式（hash 链 / 头尾一致性）
- **损坏范围汇报请求**：恢复路径在 Tier 1/2/3 降级时调

**输出**：

- **成功 ack**：携带文件 offset + 校验和
- **失败告警**：携带错误类型（disk_full / permission / io_error / hash_mismatch / rename_failed）
- **完整性报告**：文件健康 / 损坏范围（哪段事件 hash 链断）
- **降级建议**：给 L2-04 恢复用（如"跳损坏块 N~M"）

---

### 12.3 边界

**In-scope**：

1. 文件级原子写（临时文件 → fsync → rename → 父目录 fsync 语义；具体顺序在 3-1）
2. jsonl 追加的原子性（每行一个原子单位；不允许写半行）
3. hash 链检验（扫描事件流验证每条事件 hash 正确）
4. 文件 header / footer 一致性检验（针对 checkpoint 有 header 元信息）
5. 本地重试 2 次（瞬时 IO 错误容忍）
6. 损坏范围识别（指出哪段事件损坏 / 哪个 checkpoint 损坏）
7. 分级降级建议生成（Tier 1/2/3，见流 H）
8. 硬 halt 触发（落盘彻底失败时经 L2-01 触发响应面 4）

**Out-of-scope**：

- ❌ 不做文件系统级的分布式一致性（单机 POSIX 语义）
- ❌ 不做加密（审计可见，不加密；加密是运维层）
- ❌ 不做压缩（事件 jsonl 原文落；压缩是优化）
- ❌ 不做锁（L2-02）
- ❌ 不做事件分配序号（L2-01）
- ❌ 不做 checkpoint 触发时机（L2-04）
- ❌ 不做数据库层抽象（只是文件原子操作层）

**边界规则**：

- 本 L2 是**横切下垫层**，不直接面向外部 L1
- 所有"落盘"动作的最底层，必走本 L2
- 原子性保证的粒度：文件级 + jsonl 行级（不保证跨文件的原子性；跨文件的原子性由上层用 2PC 或锁组合）

---

### 12.4 约束

**业务模式引用**：

- **PM-08 可审计全链追溯**（完整性是可追溯的前提）
- **PM-10 事件总线单一事实源**（事实源必须可信）

**硬约束清单**：

1. **任何写必须 fsync**（返回成功 = 数据已在盘；不允许延迟 fsync）
2. **任何写必须经过临时文件 + rename**（不允许直接覆盖；中断时不产生坏文件）
3. **jsonl 行原子性**（要么整行在盘，要么整行不在盘，不允许半行）
4. **本地重试最多 2 次**（防长时间卡死；重试间指数 backoff 100ms/300ms，具体 3-1）
5. **hash 链不匹配 → 必标损坏**（不静默接受不一致）
6. **完整性校验绝不返回"大概对"**（是 / 否 / 部分损坏，三态清晰）
7. **原子性失败必上抛**（不吞错，让上层决定硬 halt 或降级）

**性能约束**：

- 单次 `atomic_append` 延迟：P95 ≤ 30ms（含 fsync）
- 单次 `atomic_write_snapshot`（任务级 snapshot）：P95 ≤ 500ms
- 完整性校验一万事件：≤ 5 秒
- 重试附加延迟：最多 2 次 × 300ms = 600ms

---

### 12.5 🚫 禁止行为（明确清单）

- 🚫 **禁止延迟 fsync**（"返回先不 fsync"类优化一律禁，数据丢失风险太大）
- 🚫 **禁止直接覆盖写**（必须临时文件 + rename；否则中断即坏）
- 🚫 **禁止写入半行 jsonl**（jsonl 行原子性硬约束；部分写会让 hash 链断）
- 🚫 **禁止吞 IO 错误**（吃掉错误 = 上层以为成功，事实源崩塌）
- 🚫 **禁止静默接受 hash 不匹配**（必须告警）
- 🚫 **禁止无限重试卡死**（最多 2 次）
- 🚫 **禁止完整性校验的"模糊结果"**（必须 true / false / partial 三态）
- 🚫 **禁止绕过本层直接调用 OS write 接口**（其他 L2 的落盘都必须经本层）

---

### 12.6 ✅ 必须职责（明确清单）

- ✅ **必须**每次写走临时文件 + rename + fsync 完整语义
- ✅ **必须**每次 jsonl 追加保证行级原子性
- ✅ **必须**hash 链检验（事件流）+ header 校验（checkpoint）
- ✅ **必须**本地重试 2 次后还失败即上抛
- ✅ **必须**识别损坏范围并给 L2-04 明确降级建议（Tier 1/2/3）
- ✅ **必须**IO 错误时触发 L2-01 的硬 halt 路径（响应面 4）
- ✅ **必须**完整性校验返回明确三态（OK / CORRUPT / PARTIAL）
- ✅ **必须**为每次写操作产生审计事件（落盘结果 + 耗时）

---

### 12.7 🔧 可选功能职责

- 🔧 **写盘健康度统计**（成功 / 失败 / 重试率）给 retro
- 🔧 **文件切分管理**（jsonl 单文件超大小阈值自动切新）—— 由 3-1 定策略
- 🔧 **损坏恢复的断点续传**（Tier 3 里跳损坏块后继续，非重来）
- 🔧 **写盘队列深度感知**（如高负载时主动降速）
- 🔧 **元数据备份**（关键 checkpoint 写双份以提高抗损）

### 12.9 🎯 交付验证大纲（Given-When-Then）

**正向场景 1 · 原子追加正常**
- Given：jsonl 文件大小 1MB
- When：atomic_append(一行 200 字节)
- Then：
  - 临时文件写成功
  - fsync 完成
  - rename 完成
  - 父目录 fsync 完成
  - 返回 offset + checksum
  - P95 ≤ 30ms

**正向场景 2 · 中断不坏数据（断电测试）**
- Given：atomic_append 执行到 fsync 一半
- When：模拟断电（kill -9）
- Then：重启后：
  - 已落盘的 N-1 条事件完整（rename 已完成的）
  - 正在写的第 N 条事件**不在盘**（rename 未发生）
  - hash 链闭合到第 N-1
  - 无半行 jsonl

**负向场景 3 · 磁盘满触发硬 halt**
- Given：磁盘 ENOSPC
- When：atomic_append
- Then：
  - 写失败
  - 本地重试 2 次仍失败
  - 上抛 critical
  - L2-01 触发 bus_write_failed → 响应面 4 硬 halt

**负向场景 4 · hash 链断裂检测**
- Given：恢复时读到事件 e_050 的 hash 不匹配前一条 hash
- When：verify_integrity
- Then：返回 CORRUPT + 损坏范围 "e_050"，L2-04 按 Tier 3 跳过

**负向场景 5 · checkpoint header 损坏**
- Given：checkpoint 文件 header 被截断
- When：verify_integrity
- Then：返回 CORRUPT，L2-04 回退到上一版 checkpoint

**集成场景 6 · shutdown 末自检**
- Given：干净 shutdown 流程
- When：最终 checkpoint + jsonl flush 完成
- Then：L2-05 自检返回 OK，L2-04 ack shutdown_clean

**性能场景 7 · 一万事件校验延迟**
- Given：当日 jsonl 1 万事件
- When：启动时完整性校验
- Then：≤ 5 秒完成（保证恢复 30 秒硬约束）
