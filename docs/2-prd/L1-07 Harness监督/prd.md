## 2. L2 清单（6 个）

| L2 ID | 名称 | 一句话职责 | 聚合自 BF | 核心问题 |
|---|---|---|---|---|
| **L2-01** | 8 维度监督状态采集器 | 实时读 task-board / 事件流 / verifier_report / WP 完成率 / skill 调用密度 / 子 Agent 失败率 / 上下文占用 / 预算消耗等原始信号，按 30s tick + PostToolUse + state 转换三重触发，产出 8 维度指标快照（目标保真度 / 计划对齐 / 真完成质量 / 红线安全 / 进度节奏 / 成本预算 / 重试 Loop / 用户协作） | BF-X-02 采集段 | **看到了什么** |
| **L2-02** | 4 级偏差判定器 | 对 L2-01 产出的 8 维度指标快照做阈值对比 + 规则判定 → 产出 4 档 verdict（INFO 记录 / SUGGESTION 可选采纳 / WARN 必须书面回应 / BLOCK 硬拦截），每档 verdict 附自然语言消息与动作建议 | BF-X-02 判定段 + BF-S5-03 | **定性多严重** |
| **L2-03** | 硬红线拦截器（3 核心 + 5 类全集） | 对 5 类硬红线（DRIFT_CRITICAL goal_anchor 漂移 / IRREVERSIBLE_HALT 不可逆操作 / 预算超 200% / 死循环升级到极重 / 极重度 FAIL 回 S1）执行**即时硬暂停 L1-01 + 告警 L1-10** + 必须用户文字授权才解除；其中最核心的 3 条（goal_anchor 漂移 / S2 Stage Gate 4 件套跳过 / S5 未 PASS 跳过进 S7）对应 Goal §4.1 "3 红线准确率 100%" | BF-E-12 硬红线上报流 | **黑名单硬拦截** |
| **L2-04** | Supervisor 副 Agent 事件发送器 | 作为独立 session 的副 Agent 载体 —— 持续订阅 L1-09 事件流 → 调用 L2-01 产采集 → 调用 L2-02 产判定 / L2-03 产拦截 / L2-05 产软红线 / L2-06 产路由 → 通过 IC-12 推建议队列给 L1-01 / IC-13 推回退路由给 L1-04 / IC-14 推硬暂停 / 每建议落 `supervisor_events` 到 L1-09；同时承担周期状态报告 + 风险登记册更新 + 变更请求评估三个 BF-S6 产出 | BF-S6-01/02/03 + BF-X-02 发送段 | **谁把话传出去** |
| **L2-05** | Soft-drift 模式识别器（软红线 8 类自治） | 对 8 类软红线（DoD 证据链缺 / 进度偏差 <30% / Skill 失败 / Context >80% / 轻度 TDDExe FAIL / WP 超时 <2x / 单 KB 条目读不到 / 网络瞬时失败）识别 + 触发自治修复动作指令（不打扰用户，只记事件）；同时做 soft-drift 趋势识别（BF-E-07：state 序列偏 >30% / goal_anchor 命中率 <60% / "暂时跳过" ≥5 次 → WARN + replan 建议） | BF-E-07 + BF-E-11 | **灰色地带自己修** |
| **L2-06** | 死循环升级器 + Quality Loop 回退路由控制器 | 双重职责：① 死循环保护 —— 监测任意位点同级 FAIL 连续 ≥3 次 → 自动升到上一级（轻→中→重→极重→硬拦截）；② Quality Loop 4 级回退路由 —— 基于 verifier_report 结构化 verdict 精确跳转（轻→S4 / 中→S3 / 重→S2 / 极重→S1），禁止主观越权回退 | BF-E-10 + BF-S5-04 | **卡住了怎么升级 / 怎么回去** |

**为什么切成 6 个 L2（brainstorming 结论）**：
- **L2-01 / L2-02 / L2-03 / L2-05 / L2-06 是五个独立"判定能力"**：各管一类输入 → 输出。
- **L2-04 是"传话 + 副 Agent 身份"的独立物**：scope §5.7 明确要求 Supervisor 是独立 session 副 Agent，且每个建议必须经过统一的发送管道（保证 log / 审计 / UI 推送一致），因此单列为 L2。
- **L2-01 与 L2-02 分开**（"采集" vs "判定"）：两者变更节奏不同 —— 采集跟指标源变动走（新增维度 = 改 L2-01），判定跟阈值 / 规则走（调阈值 / 重定义"严重" = 改 L2-02）。
- **L2-03 与 L2-05 分开**（硬红线 vs 软红线）：两者心智模型完全不同 —— 硬红线必须立即硬暂停 + 用户授权；软红线必须不打扰用户 + 自治修复。混一块会破坏 PM-12 分级自治。
- **L2-06 为什么合并"死循环升级 + Quality Loop 回退路由"**：两者都基于"verdict 等级判定"做"路由 / 升级"，且都要求"基于 verifier 结构化 verdict 不得主观"，本质是**一条升降级控制器**的两个出口。

```
  │                                                      │
  │  L1-01 主 loop       → 接建议 / 硬暂停                │
  │  L1-04 Quality Loop  → 接 4 级回退路由                │
  │  L1-10 UI            → 🛡️ 监督 tab + 红线告警角      │
  │  L1-09 事件总线      → 落 supervisor_events           │
  │  docs/status-reports → 周期状态报告 md                │
  │  docs/risk-register  → 风险登记册更新                 │
  │                                                      │
  └──────────────────────────────────────────────────────┘
```

**关键规则**：

- **L2-01 → L2-02 是所有判定的唯一源头**：6 个 L2 任何路径都必须先经过"采集 → 判定"；禁止绕过 L2-01 直接产 verdict。
- **L2-03 硬红线路径是"一票否决"**：一旦命中 5 类硬红线之一，必须立即走 L2-04 发 IC-14 hard_halt 给 L1-01；不得被 L2-05 / L2-06 抢占或合并。
- **L2-05 软红线路径是"静默自治"**：命中 8 类软红线时只在事件总线 + UI 告警角记录，不推建议队列打扰用户；只有升级为 WARN+ 才通知用户。
- **L2-06 Quality Loop 回退路由必须基于 verifier_report 结构化 verdict**：禁止主观越权；如 verifier 输出 INSUFFICIENT_EVIDENCE 则不得推回退路由，而是回 L2-02 作为 WARN 走"需补证据"建议。
- **L2-04 是唯一出声口**：全部 6 个 L2 的对外行为（建议 / 路由 / 硬暂停 / 报告 / 登记 / 评估）都经过 L2-04 统一发送，保证 schema 一致、审计一致、UI 推送一致。
- **所有读写事件都必须走 L1-09**（IC-09）：这是 Supervisor 自身可审计的唯一手段；Supervisor 只写自己的 `supervisor_events` 子命名空间，不碰其他 L1 的命名空间。
- **Supervisor 是只读 + 独立 session**：本 L1 不持有主 loop 的任何可变句柄；所有影响主 loop 的改动都通过 IC-12/IC-13/IC-14 经过 L1-01 的采纳决策。

## 5. L2 间业务流程（8 条 · 产品级 · 不含代码）

> 本节给出 6 个 L2 之间的主要协作流，严格产品级叙述（不含伪码 / schema）。

### BFL-07-01 · 30s 观察 tick 常规流（响应面 1 落地）

1. L2-04 副 Agent 定时器触发
2. L2-04 调 L2-01 做"全 8 维度采集"（只读事件总线 + task-board + verifier_report + WP 状态）
3. L2-01 产出 8 维度指标快照（每维度含当前值 + 对比基线）
4. L2-04 将快照交给 L2-02 判定
5. L2-02 按阈值规则产 verdict（INFO / SUGG / WARN / BLOCK + 消息 + 动作建议）
6. L2-04 根据 verdict 分发：
   - INFO → 仅落 supervisor_events
   - SUGG / WARN → 推 IC-12 建议队列 + 落事件
   - BLOCK → 转响应面 3 硬红线路径（流 07-03）
7. L2-04 统一走 IC-09 落盘 supervisor_events

**硬约束**：全流程 ≤ 10 秒；超时记 INFO "tick 超时"并不拖延下一轮。

---

### BFL-07-02 · PostToolUse 极速判定流（响应面 2 落地）

1. 主 loop 完成工具调用 → L1-09 广播 PostToolUse 事件
2. L2-04 订阅者收事件（副 Agent 独立 session 异步消费）
3. L2-04 调 L2-01 做"受影响维度增量采集"（只算"真完成质量" + "红线安全" 两维优先 · 其他维度按需）
4. L2-02 增量判定 verdict
5. 若判定 verdict=BLOCK → 立即转响应面 3（不等 30s 批次）
6. 若判定 SUGG / WARN → 汇入下一次 30s tick 的建议池（去重）
7. L2-04 落 supervisor_events

**硬约束**：PostToolUse 判定必须 ≤ 3 秒；超时则降级记 INFO "极速通道超时"并排入下一 tick。

---

### BFL-07-03 · 硬红线硬拦截流（响应面 3 落地 · BF-E-12）

1. L2-02 判定 verdict=BLOCK
2. L2-03 检查是否命中 5 类硬红线之一
   - 若未命中 5 类硬红线（但 verdict=BLOCK）→ 走 WARN 级强制书面回应路径（非硬暂停）
3. L2-03 命中确认 → L2-04 立即执行：
   - IC-14 hard_halt → L1-01 主 loop 立刻停 tick（停在安全点）
   - 推 UI 红屏告警给 L1-10（红线告警角）
   - 写 supervisor_events（type=hard_halt · 含红线类型 + 证据 + 建议操作）
4. 系统进入 HALTED 状态
5. 用户在 UI 看到告警 → 输入文字授权 / 拒绝
   - 授权继续 → L2-04 广播 halt_released · L1-01 恢复 tick
   - 拒绝 / 终止 → L1-01 走 ABORTED 流（BF-E-01 变体）
6. 授权记录落 `user_authz` 事件（L1-09）+ supervisor_events 关联

**硬约束**：
- 硬拦截必附文字告警（"为什么拦 + 证据 + 建议操作"三要素）
- 用户授权必须是文字（不接受默认 / 自动 / 隐式）
- 拦截到解除全程可审计（Goal §4.1 可追溯率 100%）

---

### BFL-07-04 · 软红线自治修复流（响应面 4 落地 · BF-E-11）

1. L2-02 判定命中 8 类软红线之一
2. L2-05 模式识别器确认类别并选择自治动作
3. L2-04 通过 IC-12 把"动作指令"发给 L1-01（但 UI 不弹告警角，仅 🛡️ tab 事件流显示）
4. L1-01 按 PM-02 自决采纳（大多数场景无脑采纳，因为是软红线自治）
5. 相应 L1（L1-04 / L1-06 / L1-09 等）执行动作
6. L2-04 记 supervisor_events（type=soft_auto_fix · 含动作 + 执行结果）
7. 若同一软红线 3 次自治仍未消除 → 触发响应面 6 死循环升级（流 07-06）

**硬约束**：
- 软红线不弹用户告警（仅 tab 内显示）
- 每条软红线自治必留"动作 + 结果" 审计对

---

### BFL-07-05 · Quality Loop 4 级回退路由流（响应面 5 落地 · BF-S5-04）

1. L1-04 产出 verifier_report · verdict ≠ PASS
2. L2-01 采集该 verifier_report
3. L2-02 判定（基于 verifier 结构化 verdict · 不主观）
   - verdict=INSUFFICIENT_EVIDENCE → 推 WARN"补证据" 给 L1-04（不走回退路由）
   - verdict=FAIL_L1~L4 → 走回退路由
4. L2-06 Quality Loop 路由器精确映射：
   - FAIL_L1 轻度 → S4 自修
   - FAIL_L2 中度 → S3 重审 TDD 蓝图
   - FAIL_L3 重度 → S2 重审 4 件套
   - FAIL_L4 极重度 → S1 重锚 goal（同时触发响应面 3 硬红线）
5. L2-04 通过 IC-13 quality_loop_route 推路由指令给 L1-04
6. L1-04 按 PM-05 机器可校验 DoD 执行回退
7. L2-04 记 supervisor_events（type=qloop_route · 含 from/to/evidence）

**硬约束**：
- 路由必须基于 verifier_report 结构化 verdict，禁主观越权
- 同一 WP 同级 FAIL 3 次 → 自动升级（流 07-06）
- 极重度回 S1 双保险（回退 + 硬红线）

---

### BFL-07-06 · 死循环升级流（响应面 6 落地 · BF-E-10）

1. L2-01 持续观察"同级 FAIL 计数"（按 WP / Gate / Stage 等粒度分组）
2. 某粒度同级 FAIL 达 3 次阈值 → L2-06 死循环升级器触发
3. L2-06 等级映射：
   - 轻 × 3 → 升中
   - 中 × 3 → 升重
   - 重 × 3 → 升极重 → 必触响应面 3 硬红线（流 07-03）
   - 极重 × 3 → 直接 ABORTED（保护兜底）
4. L2-04 通过 IC-13（新 level）或 IC-14（硬拦截 / ABORTED）推到下游
5. L2-04 记 supervisor_events（type=escalation · 含 from_level/to_level + 计数证据）
6. L1-10 UI "🛡️ 监督 tab" 升级提示（带 retro 观察指示）

**硬约束**：
- 死循环保护禁关闭（PM-12 硬约束）
- 升级记录必含计数证据（几次 FAIL / 在哪个粒度 / 时间窗口）

### BFL-07-07 · 周期状态报告生成流（BF-S6-01）

1. 定时器触发（周 / 里程碑 / 用户请求）

**硬约束**：
- 报告必含"4 级 verdict 分布 + 硬红线命中记录 + 死循环升级记录 + 软红线自治动作数"（本周期零也要明写"本周零"）

### BFL-07-08 · 风险识别与变更评估流（BF-S6-02 / BF-S6-03）

   - 输出变更评估报告 → 走 UI 展示 + 请示用户 Go / No-Go
4. L2-04 记 supervisor_events（type=risk_update / change_eval）

**硬约束**：
- 风险登记册只加不删（已消除风险归档而非删除）

## 8. L2-01 · 8 维度监督状态采集器 详细定义

### 8.1 职责 + 锚定

**一句话职责**：作为 L1-07 的"眼睛" —— 定时（30s）+ 事件驱动（PostToolUse）+ 状态转换（state_changed）三重触发下，以**只读**方式读取 L1-09 事件流、L1-02 task-board 快照、L1-04 verifier_report、L1-03 WP 完成率、L1-05 skill 调用密度 / 子 Agent 失败率、L1-01 上下文占用 / 预算消耗，产出**8 维度指标快照**（目标保真度 / 计划对齐 / 真完成质量 / 红线安全 / 进度节奏 / 成本预算 / 重试 Loop / 用户协作），并交付给 L2-02 判定器消费。本 L2 **只算值不做判定**。

**上游锚定**：
- Goal §4.1 量化指标："监督 Agent 3 红线准确率 100%" + "决策可追溯率 100%"（本 L2 是 100% 可观察的根源）
- Goal §4.3 methodology-paced autonomy："监督 Agent 密集观察"
- scope §5.7.1 职责第 1 条 "8 维度指标实时计算"
- scope §5.7.2 输入全集（事件总线 + task-board + verifier_report + 主 loop 决策链 + KB 已知 trap）
- scope §5.7.6 必须义务 "只读权限访问 task-board / 事件总线"
- BF-X-02 监督观察流（30s + PostToolUse + state 转换三重触发）
- BF-X-02 明确 8 维度："目标保真度 / 计划对齐 / 真完成质量 / 红线安全 / 进度节奏 / 成本预算 / 重试 Loop / 用户协作"

**下游服务**：
- L2-02 · 4 级偏差判定器（消费指标快照做 verdict）
- L2-04 · Supervisor 事件发送器（触发采集 + 中转快照）
- L2-06 · 死循环升级器（消费"同级 FAIL 计数"子信号）

---

### 8.2 输入 / 输出（文字级）

**输入**：
- 来自 L1-09 的事件流订阅（通过 IC-11 supervisor_observe）：decision / tool_invoked / skill_called / verifier_output / stage_gate_result / user_authz 等全量事件
- 来自 L1-02 的 task-board 快照（只读）：当前 project state / 已完成 state 序列 / goal_anchor 引用
- 来自 L1-03 的 WP 状态（只读）：WP 完成率 / WP 依赖拓扑 / 未完 WP 列表 / 预估工时 vs 实耗工时
- 来自 L1-04 的 verifier_report（只读）：verdict / three_segments 证据链完整度 / 失败位点
- 来自 L1-05 的 skill 调用密度 + 子 Agent 失败率（只读 · 从事件流统计）
- 来自 L1-01 的上下文占用 + 预算消耗（只读 · 从事件流统计）
- 来自 L1-06 的 KB 已知 trap（只读 · 通过 IC-06 kb_read，用于"红线安全"维度的软匹配）
- 来自 L2-04 的三重触发信号（30s tick / PostToolUse 事件 / state_changed 事件）

**输出**：
- 交付给 L2-02 的"8 维度指标快照"（含每维度当前值 + 历史基线对比 + 计算依据事件引用 id）
- 交付给 L2-06 的"同级 FAIL 计数"子信号（按 WP / Gate / Stage 分组）
- 每次采集落 supervisor_events（type=observe_snapshot · 走 IC-09）的审计事件

---

### 8.3 边界

**In-scope**：
1. **8 维度指标计算公式的业务定义**（非公式数值，而是"哪些事件贡献哪个维度"的规则表）
2. **三重触发机制的触发条件与节奏**（30s 周期 / PostToolUse / state_changed）
3. **增量 vs 全量采集策略**（30s tick 跑全量；PostToolUse 只跑受影响维度）
4. **历史基线维护**（用于"趋势类"维度如 goal_anchor 命中率）
5. **指标快照的事件引用追溯**（每值必有来源事件 id，保障 Goal §4.1 可追溯 100%）
6. **同级 FAIL 计数的分组规则**（按 WP id / Gate id / Stage id 做计数桶）

**Out-of-scope**：
- 不做阈值对比与 verdict 判定 → L2-02
- 不做硬红线识别 → L2-03
- 不做软红线动作选择 → L2-05
- 不做死循环升级决策 → L2-06
- 不产出对外建议 / 路由 / 硬暂停 → L2-04
- 不修改任何事件 / task-board / KB 数据（只读硬约束）
- 不做指标值的"好坏"语义解读（交给 L2-02）

**边界规则**：本 L2 是"**数据产出者**" —— 保证每次采集都**快（≤10 秒全量 / ≤3 秒增量）+ 全（8 维度无遗漏）+ 可追溯（值 ↔ 事件 id 1:1）**，但不参与"这是好是坏"的判断。

---

### 8.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-02** 主-副 Agent 协作（Supervisor 只读观察）
- **PM-08** 可审计全链追溯（每指标值必有事件引用）
- **PM-10** 事件总线单一事实源（所有观察基于事件流）

**硬约束清单（文字描述）**：
1. **只读硬约束**：禁止对 task-board / 事件总线 / KB 做任何写操作（除 supervisor_events 子命名空间的审计事件）。
2. **8 维度无遗漏**：每次全量采集必产 8 个维度的值；若某维度输入缺失（如 verifier_report 尚未就绪），须标注"N/A + 缺失原因"而不是跳过。
3. **指标值必追溯事件**：每维度值必含"证据事件 id 列表"（至少 1 条），保障审计可达。
4. **三重触发互不抑制**：30s tick 与 PostToolUse 与 state_changed 允许并发（由 L2-04 的副 Agent session 串行化）。
5. **历史基线窗口固定**：趋势类维度使用最近 N 条决策 / 最近 M 小时窗口（具体 N/M 由 3-1 技术方案定），但窗口不得随运行时动态伸缩绕过观察。
6. **同级 FAIL 计数不可跳过**：任一 FAIL 事件必纳入计数，禁止"认为这条不算"的主观过滤。

**性能约束（文字描述）**：
- 30s tick 全量采集：P99 ≤ 10 秒（含 8 维度全算）
- PostToolUse 增量采集：P99 ≤ 3 秒（只算受影响维度）
- 单次采集不得阻塞主 loop（采集走 Supervisor 副 Agent 独立 session）
- 采集失败降级：不得因单维度失败拖累其他维度（分维度独立容错）

---

### 8.5 🚫 禁止行为（明确清单）

- 🚫 **禁止写 task-board / 事件总线 / KB**（只能写 supervisor_events 子命名空间）
- 🚫 **禁止对某维度做判定**（只产值，不产"好 / 坏"）
- 🚫 **禁止跳过某维度**（必须 8 维度齐全，缺失也要明标 N/A + 原因）
- 🚫 **禁止指标值不带事件引用**（破坏 Goal §4.1 可追溯）
- 🚫 **禁止在采集路径里调 skill / 子 Agent**（Supervisor 不调用能力层）
- 🚫 **禁止运行时动态调整历史基线窗口**（窗口由硬约束定）
- 🚫 **禁止阻塞主 loop**（必须走副 Agent 独立 session 异步）

---

### 8.6 ✅ 必须职责（明确清单）

- ✅ **必须**在 30s tick + PostToolUse + state_changed 三重触发下工作
- ✅ **必须**全量采集产出 8 维度指标快照

- ✅ **必须**以 supervisor_events 形式落盘每次采集结果（审计）

## 9. L2-02 · 4 级偏差判定器 详细定义

### 9.1 职责 + 锚定

**一句话职责**：作为 L1-07 的"大脑判定层" —— 接收 L2-01 产出的 8 维度指标快照 → 按预定义规则 / 阈值做对比 → 输出 **4 档 verdict**（`INFO` 仅记录 / `SUGGESTION` 可选采纳 / `WARN` 必须书面回应 / `BLOCK` 硬拦截），每个 verdict 附"自然语言消息 + 动作建议 + 证据引用"三要素，并按类型分流到 L2-03（硬红线）/ L2-05（软红线）/ L2-06（回退路由 / 死循环），**本 L2 只定性不执行**。

**上游锚定**：
- Goal §4.1 "监督 Agent 3 红线准确率 100%"（判定准确率是红线准确率的前置能力）
- Goal §7.3 "Supervisor 分级干预权威"
- scope §5.7.1 职责第 2 条 "4 级分级干预（INFO / SUGGESTION / WARN / BLOCK）"
- scope §5.7.4 硬约束 3 "建议被主 loop 驳回必须书面理由留痕（可审计）"
- scope §5.7.4 硬约束 4 "Quality Loop 回退路由必须基于 verifier_report 的结构化判定，不得主观越权回退"
- scope §5.7.6 必须义务 7 "每次建议明确 level + 消息内容 + 动作建议"
- BF-X-02 判定段
- BF-S5-03 偏差等级判定流
- PM-12 红线分级自治

**下游服务**：
- L2-03 · 硬红线拦截器（verdict=BLOCK 且命中硬红线 → 走硬拦截）
- L2-05 · Soft-drift 模式识别器（verdict 命中软红线候选 → 走自治）
- L2-06 · 死循环升级器 + 回退路由（verdict=FAIL_L1~L4 基于 verifier 结构化 verdict → 走路由）
- L2-04 · 统一发送器（INFO / SUGG / WARN 直接分发）

---

### 9.2 输入 / 输出（文字级）

**输入**：
- 来自 L2-01 的"8 维度指标快照"（含每维度值 + 事件引用 + 历史基线对比）
- 来自 L2-01 的"同级 FAIL 计数"子信号（做 verdict 时参考计数决定是否已到升级阈值）
- 判定规则集（静态配置 · 产品级描述见下 §9.4 约束）
- 来自 L1-04 的 verifier_report 结构化 verdict（`PASS` / `FAIL_L1~L4` / `INSUFFICIENT_EVIDENCE` · 通过 L2-01 代采或直接订阅）

**输出**：
- 4 档 verdict 事件（每条含 `level` + `dimension` + `message` 自然语言 + `action_suggestion` 建议 + `evidence_refs` 事件引用列表）
- 分流信号：
  - verdict=BLOCK + 命中硬红线候选 → 送 L2-03 做最终判定
  - verdict 命中软红线候选 → 送 L2-05 做模式识别 + 动作选择
  - verdict=FAIL_L1~L4（来自 verifier_report）→ 送 L2-06 做回退路由
  - verdict ∈ {INFO, SUGG, WARN}（非硬 / 非软 / 非路由）→ 送 L2-04 直接分发
- 每次判定落 supervisor_events（type=verdict · 走 IC-09）的审计事件

---

### 9.3 边界

**In-scope**：
1. **4 档 verdict 的语义定义**（INFO = 仅记录；SUGG = 可选采纳；WARN = 必须书面回应；BLOCK = 硬拦截）
2. **8 维度 × 4 档的规则映射**（产品级：每维度在什么状态下升到什么级）
3. **自然语言消息生成**（每 verdict 附"发生了什么 + 为什么是这个级 + 建议做什么"三要素）
4. **分流决策**（根据 verdict 类型 + 维度命中做路由到 L2-03/05/06 或 L2-04）
5. **驳回理由追溯**（主 loop 驳回某条 SUGG/WARN 建议时记录驳回理由，保持可审计）

**Out-of-scope**：
- 不做指标值采集 → L2-01
- 不做硬红线识别与硬暂停动作 → L2-03（L2-02 只标"候选"）
- 不做软红线动作选择 → L2-05
- 不做回退路由精确映射（L2-02 只识别"需回退"，不决定 from/to）→ L2-06
- 不产对外建议（L2-02 只产 verdict + 候选分流信号，最终出声由 L2-04）
- 不做阈值数值的运行时动态调整（阈值由硬约束定）

**边界规则**：本 L2 是"**定性器**" —— 回答"这一刻系统处于什么健康档位"；具体"什么动作"和"推给谁"交给下游 L2 做。

---

### 9.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-02** 主-副协作（判定后由主 loop 采纳决定）
- **PM-05** Stage Contract 机器可校验（Quality Loop 分档必须基于 verifier 结构化）
- **PM-08** 可审计全链追溯（每 verdict 必带证据引用）
- **PM-12** 红线分级自治（4 档对应"静默 / 建议 / 必答 / 硬拦截"的分级自治策略）

**硬约束清单（文字描述）**：
1. **4 档 verdict 语义不可压缩**：禁止把 WARN 合成 SUGG 或 SUGG 合成 INFO 以"减少打扰"（违反 PM-12 分级自治）。
2. **verdict 必带三要素**：`level + message + action_suggestion + evidence_refs` 缺一不可；缺任一即视为判定未完成。
3. **Quality Loop 分档必须基于 verifier 结构化 verdict**：L1-04 未给出 `FAIL_L1~L4` 就不得推回退路由（禁主观越权）。
4. **主观越权禁令**：禁止 L2-02 在"verifier 没说 FAIL"的情况下自己升级到 BLOCK（红线安全 / IRREVERSIBLE_HALT 维度除外，这些有独立识别规则）。
5. **驳回必留痕**：L1-01 主 loop 若驳回某条 SUGG/WARN 建议，必须提交自然语言驳回理由，L2-02 记入审计（scope §5.7.4 硬约束 3）。
6. **同级 FAIL 计数参考制**：L2-02 判定时读 L2-01 的 FAIL 计数，若达 3 次阈值则自动升档（但最终升档决策委托 L2-06 做死循环升级；L2-02 只标"候选升档"）。
7. **INSUFFICIENT_EVIDENCE 不得路由回退**：若 verifier 产出此 verdict，L2-02 判 WARN"需补证据"而不是推回退路由（scope §5.7.5 禁"偏差判定主观越权"）。

**性能约束（文字描述）**：
- 单次 verdict 判定：P99 ≤ 1 秒（基于单次快照）
- 分流决策：P99 ≤ 500ms
- 驳回理由记录：同步持久化（不允许异步丢失）
- 批量 verdict 无打扰抑制（4 档 verdict 不做频率压制；若某类 INFO 过高由 L1-10 UI 展示层做折叠，不在 L2-02 压抑）

### 9.5 🚫 禁止行为（明确清单）

- 🚫 **禁止把 WARN 压为 SUGG**（或 SUGG 压为 INFO）以减少告警频率
- 🚫 **禁止在 verifier_report verdict=INSUFFICIENT_EVIDENCE 时推回退路由**（必须升 WARN 补证据）
- 🚫 **禁止无 evidence_refs 的 verdict 出口**（破坏可审计）
- 🚫 **禁止 L2-02 自己做硬拦截动作**（只产候选 · 最终由 L2-03）
- 🚫 **禁止驳回建议无书面理由**（硬约束 5）
- 🚫 **禁止运行时动态调整阈值**（阈值固化 · 变更走 ADR）
- 🚫 **禁止越过 verifier 主观升档**（只能按 verifier 结构化 verdict）
- 🚫 **禁止静默丢弃某条判定**（所有判定必落 supervisor_events）

## 10. L2-03 · 硬红线拦截器（3 条核心 + 5 类全集）详细定义

### 10.1 职责 + 锚定

**一句话职责**：L1-07 的**最后一道硬墙** —— 当 L2-02 标"硬红线候选"时，做**二次确认** + **立即硬暂停主 loop** + **红屏告警 UI** + **等用户文字授权**；职责覆盖 Goal §4.1 **"监督 Agent 3 红线准确率 100%"**的 3 条核心红线（`goal_anchor 漂移` / `Stage Gate 4 件套跳过` / `S5 未 PASS 进 S7`）与 BF-E-12 **5 类硬红线全集**（`DRIFT_CRITICAL` / `IRREVERSIBLE_HALT` / `预算超 200%` / `死循环升极重` / `极重度 FAIL`），并保证**硬拦截必附文字告警 + 授权必须是用户文字**两条不可妥协硬约束。

**3 条核心红线 vs 5 类全集的关系**：
- **5 类全集** = BF-E-12 的硬红线清单（用于运行时拦截）
- **3 条核心红线** = Goal §4.1 量化指标选定的"不可变之地"（是 5 类的语义超集 · 其中 `goal_anchor 漂移` ↔ `DRIFT_CRITICAL`；`Stage Gate 4 件套跳过` 与 `S5 未 PASS 进 S7` 都表现为 `极重度 FAIL` 的特殊形式并触发 `IRREVERSIBLE_HALT` 级拦截）
- 本 L2 以 5 类为执行面 · 以 3 条为**准确率考核指标**（Goal §4.1 要求 100% 准确率）

**上游锚定**：
- Goal §4.1 "监督 Agent 3 红线准确率 100%"（核心考核指标）
- Goal §3.5 硬约束 4 "S5 未 PASS 不得进 S7"（3 条核心之一）
- Goal §7.3 "Supervisor 分级干预权威"
- scope §5.7.1 职责 "硬红线 5 类硬拦截（必须用户文字授权）"
- scope §5.7.4 硬约束 2 "硬拦截必须附文字告警，用户授权才能解除"
- scope §5.7.5 禁"硬拦截不附文字告警（黑盒拦截 = 破坏可审计）"
- scope §5.7.6 必须 "对硬红线 5 类立即 BLOCK L1-01 + 告警 L1-10"
- BF-E-12 硬红线上报流（5 类清单）
- BF-S6-05 硬红线上报流
- PM-12 红线分级自治
- PM-08 可审计全链追溯（硬拦截全程必须可审计）

**下游服务**：
- L2-04 · Supervisor 事件发送器（通过 IC-14 hard_halt 推 L1-01 + UI 推 L1-10）

---

### 10.2 输入 / 输出（文字级）

**输入**：
- 来自 L2-02 的"硬红线候选 verdict"（level=BLOCK + 标注候选类型）
- 来自 L2-01 的"证据事件引用列表"（支撑硬红线成立的原始事件）
- 来自 L2-06 的"死循环升级到极重"信号（第 ④ 类硬红线的触发源）
- 来自 L1-04 verifier_report 的极重度 FAIL 信号（第 ⑤ 类硬红线的触发源）
- 来自 L1-09 的"goal_anchor sha256 快照"（用于 DRIFT_CRITICAL 识别）

**输出**：
- 命中确认信号（交给 L2-04 通过 IC-14 hard_halt 推 L1-01 立即停 tick）
- UI 红屏告警载荷（交给 L2-04 推 L1-10 "红线告警角"）
- 文字告警三要素：**原因（哪条红线 + 证据） + 影响（继续会发生什么） + 建议操作**
- 硬暂停事件（走 IC-09 落 supervisor_events · type=hard_halt）
- 授权解除事件（用户授权后由 L2-04 广播 halt_released）
- 被驳回候选（若 L2-03 二次确认未命中 5 类之一，返回 L2-02 降级为 WARN 级强制书面回应）

---

### 10.3 边界

**In-scope**：
1. **5 类硬红线的识别规则**（产品级描述"哪些状态属于哪一类"）：
   - ① `DRIFT_CRITICAL`：goal_anchor sha256 变化 / CLAUDE.md 被篡改 / `goal_anchor` 关键句在最近 N 条决策命中率 = 0
   - ② `IRREVERSIBLE_HALT`：不可逆破坏性操作候选（`rm -rf` 根系目录 / `git push --force` 到 main / `DROP DATABASE` / 生产环境 `deploy`）
   - ③ 预算超 200%：token / 时长 / 花费任一档次超出 S1 声明基线 200%
   - ④ 死循环升级到极重：L2-06 输出"极重度"升级信号
   - ⑤ 极重度 FAIL：verifier_report verdict=FAIL_L4（回 S1 级别失败）
2. **3 条核心红线的准确率责任**（与 5 类映射清晰 · 保证 Goal §4.1 100% 准确率考核可落地）
3. **二次确认逻辑**（L2-02 给"候选"、L2-03 判"成立"，避免误伤）
4. **文字告警三要素生成**（原因 + 影响 + 建议操作）
5. **授权解除路径**（用户文字 → halt_released 广播 → L1-01 恢复 tick）
6. **拦截器自身的健康探测**（若拦截器不工作，整个系统失去硬墙 → 必须自检）

**Out-of-scope**：
- 不做 4 档 verdict 判定（L2-02）
- 不做软红线（L2-05）
- 不做 Quality Loop 4 级回退（L2-06）
- 不直接调 L1-01 API（通过 L2-04 统一发送）
- 不定义 UI 渲染样式（只提供告警载荷 · UI 由 L1-10）
- 不自动给用户预设授权答案（必须用户文字）

**边界规则**：本 L2 是"**硬墙**" —— 仅对 5 类硬红线执行硬暂停；对不属于 5 类的 BLOCK 候选返回 L2-02 降级处理。硬墙的价值在于"**敢停 + 停得住 + 可审计**"，精准度优先于速度但速度不得低于极速通道硬约束。

### 10.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-12** 红线分级自治（硬红线必须文字授权）
- **PM-02** 主-副协作（Supervisor 通过 IC-14 行使硬拦截权威 · 不直接改数据）
- **PM-08** 可审计全链追溯（拦截 → 告警 → 授权 → 解除全程事件完整链）

**硬约束清单（文字描述）**：
1. **硬拦截必附文字告警**：禁止黑盒拦截；告警必含"原因 + 影响 + 建议操作"三要素（scope §5.7.5）。
2. **授权必须是用户文字**：拒绝"默认 / 自动 / 定时 / 超时视为同意"任何隐式授权（Goal §3.5 / scope §5.7.4 硬约束 2）。
3. **硬拦截响应时延**：从候选到达到 IC-14 发出 ≤ 2 秒（极速硬墙通道）。
4. **3 条核心红线准确率 100%**：不得漏报（Goal §4.1 考核指标）；误报允许但需在 retro 中归因。
5. **全程事件完整链**：`硬红线候选 → 命中确认 → hard_halt → 告警推送 → 用户授权 → halt_released` 六个事件必齐全（PM-08）。
6. **拦截器自检**：Supervisor 副 Agent 启动时必须做一次拦截器探测（人造测试红线 → 验证拦截器路径通）。
7. **不可变禁令**：scope §5.7.5 "禁止关闭死循环保护" + "禁止硬拦截不附文字告警"；所有解除权在用户文字，任何配置项不得绕过。

**性能约束（文字描述）**：
- 二次确认判定：P99 ≤ 1 秒（候选已到达 → 确认成立与否）
- IC-14 hard_halt 发出：P99 ≤ 500ms（确认成立后立即推）
- UI 红屏推送：P99 ≤ 1 秒（L2-04 → L1-10）

## 12. L2-05 · Soft-drift 模式识别器（软红线 8 类自治）详细定义

### 12.1 职责 + 锚定

**一句话职责**：L1-07 的"**隐形自治员**" —— 接 L2-02 推来的"软红线候选"，识别是 BF-E-11 定义的 **8 类软红线** 中的哪一类 + 选择对应**自治修复动作指令**（不打扰用户）；同时做 BF-E-07 定义的 **soft-drift 趋势识别**（state 序列偏差 >30% / goal_anchor 命中率 <60% / 决策理由重复"暂时跳过" ≥5 次 → 触发"WARN + replan 建议"）。本 L2 的核心心智是"**灰色地带自己修**"：8 类已知问题各有标准解决动作，Supervisor 直接推动作指令给 L1-01；但 L1-01 仍按 PM-02 拥有最终采纳权。

**8 类软红线（BF-E-11 清单 + 对应自治动作）**：

| 软红线 | 自治动作 |
|---|---|
| ① DoD 证据链缺一段 | Supervisor 自动 trigger verifier 重跑 / 补证据 |
| ② 进度偏差 <30% | 自动 replan WP 拓扑 |
| ③ Skill 调用失败 | 能力抽象层 fallback（BF-E-05） |
| ④ Context 占用 >80% | 自动压缩（BF-E-06） |
| ⑤ 轻度 TDDExe FAIL | 回 S4 自修（BF-S5-04） |
| ⑥ WP 超时 <2× 估算 | 自动延长 + 记入风险登记 |
| ⑦ 单 KB 条目读不到 | 降级到无 KB 决策 + 告警 |
| ⑧ 网络瞬时失败 | 指数 backoff 重试（BF-E-03） |

**上游锚定**：
- scope §5.7.1 职责 "软红线 8 类自治修复（不打扰用户）"
- scope §5.7.5 禁 "软红线打扰用户（仅 WARN+ 才通知用户）"
- scope §5.7.6 必须 "对软红线 8 类执行自治修复（不打扰用户）并落事件"
- BF-E-11 软红线自治修复流（8 类清单）
- BF-E-07 任务走偏 soft-drift 检测流（趋势触发条件）
- BF-S6-04 软红线自治修复流
- PM-12 红线分级自治
- L1-06 KB（读已知 trap / anti_pattern 辅助软红线识别 · scope §5.7.7）

**下游服务**：
- L2-04 · Supervisor 事件发送器（推动作指令给 L1-01 · UI 仅 🛡️ tab 事件流不弹告警角）
- L2-06 · 死循环升级器（同一软红线 3 次自治仍未消除 → 触发升级）

**输入**：
- 来自 L2-02 的"软红线候选 verdict"（含维度 / 初步消息 / 证据引用）
- 来自 L2-01 的指标快照（做趋势类 soft-drift 识别需历史基线）

**输出**：

### 12.3 边界

**In-scope**：
1. **8 类软红线识别规则**（产品级描述每类的触发条件）
2. **8 类标准自治动作的选择**（1:1 映射）
3. **soft-drift 趋势识别**（三条 BF-E-07 触发条件）
4. **已知 trap 匹配辅助**（通过 IC-06 读 KB，提升识别准确率）
5. **自治动作结果的反馈闭环**（动作执行后下一 tick 观察是否消除）
6. **3 次自治未消除的升级反馈**（交给 L2-06 做死循环升级）

**Out-of-scope**：
- 不做硬红线识别 → L2-03
- 不亲自执行动作（本 L2 只产出"动作指令"；执行由 L1-01/04/06/09 按 PM-02 采纳决定）
- 不做 4 档 verdict 判定 → L2-02
- 不做回退路由精确映射 → L2-06
- 不对用户直接告警 → 硬约束：软红线不打扰用户
- 不做动作模板的运行时扩展（8 类封闭 · 新增走 ADR）

**边界规则**：本 L2 是"**灰度自治员**" —— 把 L1 框架下已知的灰色问题（非硬红线也非 OK）自动转成标准动作指令。8 类的"封闭 + 确定 + 静默"是核心语义。

### 12.4 约束（文字级 + 性能文字）

**业务模式引用**：
- **PM-12** 红线分级自治（软红线必不打扰用户）
- **PM-02** 主-副协作（动作指令仍由 L1-01 采纳 · 本 L2 不直接执行）
- **PM-08** 可审计全链追溯
- **PM-09** 能力抽象层调度（③ 类软红线指向 fallback 路径）

**硬约束清单（文字描述）**：
1. **软红线必不打扰用户**：UI 仅 🛡️ tab 事件流记录 · 不弹红线告警角 · 不走 WARN+ 级建议通道。
2. **8 类封闭**：软红线清单严格 8 类 · 禁运行时扩展 · 新增走 ADR（PRD 冻结后）。
3. **每类必有动作模板**：无模板即无法识别为该类 · 退回 L2-02 升 WARN。
4. **动作仍归 L1-01 采纳**：本 L2 只产指令 · 最终执行由 L1-01 按 PM-02 采纳决定（虽然惯例是无脑采纳因为是软红线）。
5. **3 次自治未消除必升级**：同一位点同一软红线连续 3 次自治动作未消除 → 必须触发 L2-06 升级到 WARN+ 级（防止软红线成为"永久噪音"）。
6. **soft-drift 趋势严格按 BF-E-07 三条件**：state 序列偏 >30% / goal_anchor 命中率 <60% / "暂时跳过" ≥5 次 · 命中任一即升 WARN replan 建议（不再是软红线，是 WARN · 走 L2-04 常规通道）。
7. **动作执行结果必回写**：L2-05 需订阅"动作后事件"判断是否消除；未消除必参与 3 次计数。

**性能约束（文字描述）**：
- 软红线识别：P99 ≤ 500ms（L2-02 候选到达 → 类别确认）
- 动作指令产出：P99 ≤ 500ms
- soft-drift 趋势识别：跟随 30s tick 节奏（不增加额外时延）
- KB 查询辅助：P99 ≤ 1 秒（若超时则跳过 KB 辅助 · 走纯规则识别）
