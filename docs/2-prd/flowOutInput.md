# HarnessFlow 输入-输出规范（I/O Specification）

> **v0.2 定位修正**：本文档**只定义方法论级**的输入输出——**内容是什么 + 作用是什么 + 来源 + 去向**。**不下钻字段级 schema**。
>
> **字段级 schema 由下游文档承担**：
> - 跨 L1 契约字段 → `scope.md §8.2 IC-XX`
> - L1 内部契约字段 → `L1-XX/prd.md §6 IC-L2-XX`
> - 运行时数据结构 → `L1-XX/prd.md §X.10 L3 实现设计`
> - 最终实现级字段 → 技术方案文档（下一阶段）
>
> 本文档的价值：**让"做什么"明白**，让"怎么实现"留给专门文档。风格参考 PMI PMBOK 的 ITTO（Input / Tools & Techniques / Output）。

## 1. 文档定位

### 1.1 回答 3 个问题

1. **HarnessFlow 研发时每类文档（goal / scope / PRD 等）**：输入什么 + 产出什么 + 每项输入/产出的作用？
2. **HarnessFlow 运行时每阶段 / L1 / L2**：输入什么 + 产出什么 + 作用？
3. **下游 TDD 怎么用本文档**：本文档告诉 TDD **要测什么类型的东西**（不是具体字段）。

### 1.2 本文档**不做**的事

- ❌ 不定义 YAML / JSON 字段清单
- ❌ 不定义枚举值
- ❌ 不定义类型约束（int 范围、字符串正则等）
- ❌ 不定义具体 schema 版本

上述全部在具体 **PRD 内部**或**技术方案**定义。

### 1.3 本文档**做**的事

- ✅ 定义每类产出物**是什么**（中文概念描述，不是字段）
- ✅ 定义每项输入/产出的**作用**（为什么需要它）
- ✅ 定义**来源 / 去向**（谁产它、谁用它）
- ✅ 定义**高层约束**（必经事件总线 / 必须有消费者 / 必须可追溯 等）
- ✅ 画清楚**流转链路**（研发链路 + 运行时链路）

## 2. 通用规则（所有产出物必须遵守）

### 2.1 可追溯规则

每个产出物必可追溯至 Goal。文档链必须连续：
**Goal → businessFlow → scope → L1-PRD → L2 → L3 → TDD → 技术方案 → 代码**。

字段级 frontmatter（doc_id / doc_type / parent_doc / traceability / consumer）见 scope.md 参考实例。本文档只强调**"必须可追溯"**的高层约束。

### 2.2 消费者规则（防 paperwork theater）

每产出物**必有**至少一个消费者（下游文档 / 运行时 L1 / 用户）。**无消费者的产出物禁止生成**（引自 PM-07 产出物模板驱动）。

### 2.3 事件总线规则

所有**状态变更类产出**（decision / state_transition / audit / verdict）必经 L1-09 事件总线。**禁止绕过**（PM-10 单一事实源）。

### 2.4 不可变规则

已落盘的审计事件、历史状态、归档条目**不可修改**。修订语义使用新版本指针。

### 2.5 输入完整性规则

每个过程在启动时必须**输入完整**。缺输入 → 不能启动该过程（通过 Stage Gate 或前置检查拦截）。

## 8. 一致性校验原则（方法论级 · 不是字段级）

本文档的校验**不下钻到字段层**，只校验**概念层一致性**：

### 8.1 可追溯校验

- 每产出物必有 frontmatter `parent_doc` → 可反溯到 Goal
- 每产出物必有 `consumer` → 避免无消费者产出

### 8.2 完备性校验

- 每阶段 / 每 L1 / 每 L2 的"输入"列不为空
- 每阶段 / 每 L1 / 每 L2 的"输出"列不为空
- 每输入有"来源" · 每输出有"去向"

### 8.3 作用可解释校验

- 每输入有"作用"说明（为什么需要）
- 每输出有"作用"说明（给谁用、用来做什么）

### 8.4 非越权校验

- 本文档不定义具体字段（如发现任何 YAML/JSON 字段级约束，应挪到 PRD 或技术方案）
- L1 PRD 不越权定义跨 L1 IC（这是 scope.md 职责）
- 技术方案不越权改 PRD（技术方案是 PRD 的下游）

**字段级一致性**由具体 PRD 和技术方案各自保证。
